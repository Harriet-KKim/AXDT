# SoT Authoring

`docs/sot/`의 requirements·specification·test-design은 `_TEMPLATE.md`라는 빈 골격만 있다. 이 골격을 맨손으로 채우면 세 가지가 반복해서 어긋난다 — 형식 누락(미치환 플레이스홀더, 항목 ID 누락, 끊어진 참조 → ① 형식 검사 탈락), 추적 공백(요구를 구체화한 사양이 없거나 대응 요구 없는 고아 사양·테스트 조건 → ② 정합성·공백 검토 blocking), 수용 기준의 filler화("정상 동작하면 통과" 같은 판정 불가능한 문장). 이 스킬은 에이전트(**Author** 역할)가 사용자와 대화하며 이 골격을 채우고, PR로 올리기 직전 ① 형식을 스스로 맞춰 `sot/<slug>` PR로 제안한다.

## 경계

**한다**
- requirements·specification·test-design 문서를 대화형으로 신규 작성·수정한다.
- 세션 시작 시 읽은 템플릿의 필수 섹션·frontmatter를 빠짐없이 채운다.
- 요구·사양·테스트 설계를 한 세션에서 함께 다뤄, 요구·사양 ↔ 테스트 설계 추적(`covers`)을 만든 자리에서 잇는다.
- PR 제안 직전 **① 형식 자가검사**(결정적·객관적 항목)를 한다.
- `sot/<slug>` 브랜치에 변경을 얹고 **PR 제안(초안 제시)까지** 한다. PR 발의(생성)는 Maintainer가 한다.

**하지 않는다**
- **② 정합성·공백 검토(verdict 생성)**. 이건 `sot-readiness-review` 스킬이 호스트 CI에서 작성 세션과 분리해 실행한다(자기검토 편향 방지).
- **완료 판정**. 머지 게이트가 ①∧②∧③을 같은 판정 키로 계산해 확정한다.
- **③ 사용자 승인**. `rule-sot-change-user-gate`의 사용자 게이트 소관이다.
- `main` 직접 커밋.
- **`rule/` 문서 작성**. 이 스킬은 requirements·specification·test-design만 다룬다.

**자기판정 표류 금지**: "②를 안 한다"를 산출물에서도 지킨다.
- PR 본문·문서에 "정합성 확인 완료·준비도 충족" 류 주장을 쓰지 않는다.
- 감사 로그 `docs/interim/sot-readiness-review.md`(Maintainer 소유)에 쓰지 않는다.
- `review_clear` 등 verdict 유사 산출을 내지 않는다.

**경계 근거**: 형식(①)은 결정적이다 — 같은 내용이면 같은 결과라 작성 세션이 스스로 맞춰도 편향이 없다. 정합성(②)은 판단이라 작성한 사람이 하면 편향된다. 그래서 ①은 이 스킬 안에서 미리 맞추고 ②는 분리한다. 이 선이 이 스킬과 `sot-readiness-review`를 가른다.

## 언제·누가 쓰나

- **그린필드**: 대상 프로젝트의 SoT를 처음 세울 때. Author가 사용자와 직접 대화하며 요구·사양·테스트 설계를 작성한다.
- **증분**: 기존 SoT에 요구·사양·테스트 설계를 추가·수정할 때. 대화 상대는 주로 Leader의 사양변경요청(대화 없는 입력)이며(사람이 있으면 대화 가능 — 아래 부재 경로 참고), 가장 흔한 트리거는 **② blocking finding 해소**다. 여기서 ②는 별도 CI(`sot-readiness-review`)가 낸 finding이고, 이 스킬은 그 finding을 **입력으로만** 받아 SoT를 고친다 — 스킬이 ②를 실행하거나 verdict를 내는 게 아니다(위 경계와 일치).
- 두 경우 모두 **Author가 `sot/<slug>` push 주체**다(task 브랜치의 `docs/sot/**` 직접 수정은 `rule-protected-paths`가 거부한다). **PR 발의(생성)는 Maintainer만** 한다. 산출은 동일하게 `sot/<slug>` PR이다.

## 대화 흐름

**기본 자세**: 백지 질문을 던지지 않는다. 각 섹션의 초안을 먼저 제안하고 사용자가 수정·승인한다. 진짜 결정 지점만 질문하며, 모호하면 선택지 2~3개로 좁혀 단일 추천으로 제시한다.

**사용자 부재 경로**: 사람이 없으면 제안→승인 지점은 PR 리뷰(③ 사용자 게이트)로 이월한다. 입력(Leader 요청문)에서 도출할 수 없는 결정은 가정으로 표기해 PR 본문에 싣는다.

**원칙**: 요구(왜/무엇) → 사양(무엇/어떻게) → 테스트 설계(어떻게 검증) 순서로, 주제(topic) 하나를 이 순서로 끝까지 완성하고 다음 주제로 넘어간다 — 세 문서를 주제별로 함께 다뤄야 `covers` 추적을 만든 자리에서 바로 잇는다. 세 문서는 **항상 한 `sot/<slug>` PR에 함께** 싣는다(부분 SoT PR 금지).

1. **주제·파일·브랜치 확정** — `topic`(소문자 kebab-case), `docs/sot/{requirements,specification,test-design}/<topic>.md`, frontmatter `id`, PR 브랜치 `sot/<slug>`를 정한다. 증분이면 대상 문서와 변경 범위를 정한다(브랜치 재사용 금지, 같은 주제 재변경은 `sot/<slug>-2` 식 접미).
2. **문서 분해 결정** — 요구↔사양↔테스트 설계는 셋 다 다대다다. 신규로 쓸지 기존 문서 확장으로 흡수할지, 세 문서의 분해 축을 먼저 정한다.
3. **요구사항 작성** — 목표 → 범위 외 → 기능 요구(`FR-n`) → 비기능 요구(`NFR-n`) → 제약 → `related` 힌트. 각 `FR`/`NFR`을 확정한 **직후 그 자리에서 수용 기준을 함께** 확정한다(끝에 몰아 받지 않는다 — filler 방지). 수용 기준 대본: 무엇을 관찰하면 통과인가 / 측정치·조건은 / 실패는 어떻게 드러나는가.
4. **사양 작성** — 범위 → 구성요소(`SP-n`) → 인터페이스 → 데이터 모델 → 동작(정상 경로 + 에지 케이스). 각 `SP` 직후 수용 기준 → `covers`(구체화하는 `FR-n`/`NFR-n`).
5. **테스트 설계 작성** — 범위 → 테스트 조건(`TD-n`) → 커버리지 목표(블랙박스 기법: 동등분할·경계값·결정표·상태전이) → 스위트 구조 → 추적성 → 절차 원칙 → 수용 기준. 각 `TD` 직후 `covers`(검증하는 `FR-n`/`NFR-n`/`SP-n`)를 잇는다. **절단선 안내**: 여기 담는 것은 테스트 조건·기법·경계의 식별까지다. 거기서 뽑는 구체 입력값·기대 출력, 데이터·픽스처·실행 스크립트·환경, 화이트박스 커버리지는 `test/` 코드(Tester 소관) 몫이다 — 이 선을 대화에서 사용자에게 알리되, 그 아래 항목은 test-design 문서에 적지 않는다.
6. **추적·규칙 링크** — `covers`로 짝을 잇는다: 모든 `FR`/`NFR`이 최소 하나의 사양 `covers`에, 모든 요구·사양 항목이 최소 하나의 테스트 설계 `covers`에 등장하도록 한다(작성 완결성 목표이지 **②통과를 보장하진 않는다**). `rules`는 `docs/sot/rule/`을 스캔해 후보를 제시하고, 이유를 확인해 선택한다. 실제로 따르는 rule을 빠짐없이 선언한다 — 특히 `scope: local` rule을 `rules`에 선언하지 않으면 그 rule이 바뀌어도 재검토 대상에서 빠진다(낡은 완료로 이어짐).
7. **역행 루프**(횡단 규칙 — 순차 단계 아님) — 2~6 어느 지점에서든, 특히 사양·테스트 설계를 쓰다 요구·사양의 결함·누락을 발견하면 해당 항목으로 복귀·수정 → 사용자 재확인 → 영향받는 `covers` 재점검. `topic`/`slug` 재확정도 PR 제안 전까지 허용한다.
8. **① 형식 자가검사** — 아래 절 참고. 증분이어도 변경분만이 아니라 **결과 트리 전체(또는 영향받는 req/spec/test-design 전체)**에 대해 수행한다(참조 무결성이 국소 변경으로 원거리 문서를 깨뜨릴 수 있어서다).
9. **PR 제안** — 아래 절 참고.

증분 수정이면 2~7을 변경분 중심으로 진행하되, 8의 자가검사만은 결과 트리 전체(또는 영향받는 req/spec/test-design 전체)에 건다.

## 템플릿과의 관계

스킬은 구조를 새로 정의하지 않는다. **세션 시작 시 현재 `_TEMPLATE.md`(requirements·specification·test-design)를 읽어** 필수 섹션·frontmatter를 소스로 삼는다. 정본은 템플릿이고 스킬은 그것을 채우는 대화 절차만 규정한다 — 위 대화 흐름의 섹션 나열은 현재 템플릿 기준 예시일 뿐이니 하드코딩하지 않는다. 템플릿이 바뀌면 스킬은 자동으로 새 구조를 따라야 한다.

대화 순서(채워 넣는 순서)와 최종 문서의 섹션 배치(템플릿 순서)는 다르다 — 대화 순서로 문서를 재배열하지 않는다.

## ① 형식 자가검사

정본은 `docs/sot/rule/sot-readiness.md`의 "① 형식" 절이다. **세션 시작 시 그 절을 읽어 현재 항목을 확인한다** — 아래는 요약일 뿐, 판정 기준은 항상 그 절이다.

- requirements·specification·test-design 각각, frontmatter `items`에 항목을 선언한 문서가 최소 1개(README·`_TEMPLATE` 제외).
- 선언 정본은 frontmatter `items`다(본문에서 파싱하지 않는다). `items`에 선언한 항목 ID는 문서 종류 규약을 따른다(요구 `FR-n`·`NFR-n`, 사양 `SP-n`, 테스트 설계 `TD-n`).
- 같은 `(topic, ID)` 조합이 둘 이상 선언되지 않는다(유일성).
- 본문에 등장하는 그 문서 고유 종류의 항목 ID는 `items` 선언과 **양방향으로 일치**한다(본문에만 있거나 선언에만 있으면 위반).
- 문서 `id`는 `<종류접두>-<topic>`이고 `topic`은 파일 stem이다(`id`↔`topic` 일치).
- **참조 무결성**: `covers`가 가리키는 요구·사양 항목이 대상 문서 `items`에 선언돼 있고, `rules`가 가리키는 rule id가 실재한다(`related`는 힌트라 검사 제외).
- 미치환 플레이스홀더(`<...>`) 없음 — 코드 블록·인라인 코드는 제외.
- 금지어 없음 — 닫힌 목록 `TBD`·`TODO`·`FIXME`·`미정`·`추후`·`나중에`(ASCII는 대소문자 무시·단어 경계, 한글은 정확 매칭).
- 각 항목의 수용 기준 자리에 플레이스홀더·금지어가 아닌 문장이 있음.

**커버리지 완결성**(모든 요구가 사양으로 구체화되고, 모든 요구·사양이 테스트 조건으로 검증됨)은 ①이 아니라 **②**다. 자가검사 게이트에 넣지 말고 위 대화 흐름 6단계에서 챙긴다.

`sot-lint`(`WIP/axdt/sot_lint/`)가 위 ①의 기계 구현(C1~C6)이다. **① 자가검사 = sot-lint를 clean(종료 0)까지 돌리는 것**이다.

### 호출 방법

Phase 6 이전엔 패키지가 `WIP/` 아래 있어(import 루트=`WIP/`) 저장소 루트에서 모듈이 돌지 않는다. `WIP/`에서 실행하되 검사 경로 인자는 실제 SoT 트리(저장소 루트의 `docs/sot`)를 가리켜야 한다 — cwd가 `WIP/`이므로 `docs/sot`는 존재하지 않는 `WIP/docs/sot`로 잘못 해석된다.

```
cd WIP
py -3 -m axdt.sot_lint ../docs/sot --json
```

종료코드: `0`=clean, `1`=위반, `2`=파싱 오류. **clean(0)이 될 때까지 위반을 고친다.**

(Phase 6에서 `axdt` 패키지로 승격되면 어디서든 `axdt sot-lint <트리경로>`로 바뀐다.)

## PR 제안

`sot/<slug>`에 얹어 PR 초안을 제시한다(발의=Maintainer). 사용자 게이트 대기. **PR 본문에 핵심 결정과 기각한 대안 요약을 반드시 담는다**(대화에서 내려진 비자명 결정이 증발하지 않게).

ADR 촉발 조건(`rule-adr-recording`)에 걸리는 비자명 설계 결정은 **PR 본문에 ADR 링크 또는 생략 사유를 남기는 것이 강제층**이다(ADR 파일 생성 자체는 권고). 이 스킬이 저술하는 SoT는 **대상 프로젝트**의 것이므로, 촉발 시 ADR은 대상 프로젝트의 **`docs/interim/ADR/`**(형식 `docs/interim/ADR/_TEMPLATE.md`)에 쓴다 — AXDT 자체 결정용 `WIP/adr/`와 혼동하지 않는다.

## 항목 선언(`items`)

각 문서 frontmatter `items`에 그 문서가 선언하는 항목 ID를 **본문과 일치하게** 채운다(정본). 존재·유일성·참조 무결성은 frontmatter 기준으로 대조된다.

## test-design — SoT 4번째 타입

test-design은 요구·사양과 동급인 SoT 4번째 타입이다.

- **담는 것**: 테스트 조건(`TD-n`)·블랙박스 커버리지 목표(동등분할·경계값·결정표·상태전이)·스위트 구조·요구·사양 ↔ 테스트 추적(`covers`)·절차 원칙·수용 기준.
- **담지 않는 것**: 테스트 조건·기법·경계에서 뽑는 구체 입력값·기대 출력, 데이터·픽스처·실행 스크립트·환경, 화이트박스 커버리지(결정/분기/MC-DC) 측정. 이들은 `test/` 코드(Tester 소관) 몫이다.
