# B-1 설계 초안 — SoT 문서 작성 스킬 (대화형)

> 상태: 검토용 초안 (Codex·Fable 3회 리뷰 반영 + D16 정합화 + 브레인스토밍 6개 결정 반영). 스킬 구현(SKILL.md)이 아니라 "이 스킬이 무엇을 하는가"의 설계 합의를 만들기 위한 문서. 브레인스토밍은 종결됐고(§8), 다음은 사용자 스펙 리뷰다.
> 대상 TODO: Phase 1 "요구사항/사양/테스트 설계 작성 Skill 제작 (Agent와 대화형 작성)". 루트 `README.md`도 "요구사항, 사양, 테스트 설계는 Agent와 함께 정해진 Skill을 통해 작성"이라 명시 — 이 스킬이 그 Skill이다.
> 전제: "테스트 문서"는 SoT 4번째 타입 `docs/sot/test-design/`(요구·사양과 동급)으로 확정됐다(D16·`ADR-0008`). 이 스킬은 요구·사양·테스트 설계 셋을 함께 작성한다.

## 0. 왜 이 스킬이 필요한가

SoT의 `requirements/`·`specification/`·`test-design/`에는 지금 `_TEMPLATE.md`라는 빈 골격만 있다. 이 골격을 사람이 맨손으로 채우면 세 가지가 반복적으로 어긋난다.

1. **형식 누락** — 미치환 플레이스홀더(`<...>`), 항목 ID(`FR-n`/`NFR-n`/`SP-n`/`TD-n`) 누락, `covers`/`rules`의 dangling 참조(실재하지 않는 대상). → `rule-sot-readiness`의 **① 형식 검사** 탈락. (`covers`가 비어 커버가 없는 것은 ①이 아니라 아래 2번 — ②의 몫.)
2. **추적 공백** — 요구는 있는데 이를 구체화한 사양이 없거나(미커버), 대응 요구 없는 고아 사양, 검증하는 테스트 설계가 없는 요구·사양. → **② 정합성·공백 검토** blocking.
3. **수용 기준의 filler화** — "정상 동작하면 통과" 같은 판정 불가능한 기준.

스킬은 에이전트가 사용자와 대화하며 이 골격을 채우고, PR 게이트에 올리기 전에 형식을 스스로 맞추게 한다. 결과적으로 ①에서 막히지 않고, ②에서 지적받을 여지를 줄인 문서를 `sot/<slug>` PR로 제안하는 것이 스킬의 목표다.

## 1. 경계 — 무엇을 하고 무엇을 하지 않나

**한다**
- `requirements`·`specification`·`test-design` 문서를 대화형으로 신규 작성/수정.
- 세션 시작 시 읽은 `_TEMPLATE.md`(세 타입)의 필수 섹션·frontmatter를 빠짐없이 채움.
- 요구·사양·테스트 설계를 한 세션에서 다뤄, 요구·사양 ↔ 테스트 설계 추적(`covers`)을 만든 자리에서 잇는다.
- PR 제안 직전 **① 형식 자가검사**(결정적·객관적 항목).
- `sot/<slug>` 브랜치에 변경을 얹고 **PR 제안(초안 제시)까지**. PR 발의(생성)는 Maintainer, 실행 주체는 Author다(§2).

**하지 않는다**
- **② 정합성·공백 검토(verdict 생성).** 그건 `sot-readiness-review` 스킬이 **호스트 CI에서, 작성 세션과 분리 실행**한다(자기검토 편향 방지). 그리고 **완료 판정 자체는 어느 스킬도 하지 않는다** — 머지 게이트가 ①∧②∧③을 **같은 판정 키**에서 계산해 완료를 확정한다.
- **③ 사용자 승인.** 사용자 게이트(`rule-sot-change-user-gate`) 소관.
- `main` 직접 커밋.
- **`rule/` 문서 작성.** SoT엔 rule도 있으나 B-1은 req·spec·test-design만 다룬다(rule 변경은 별도 경로).

**자기판정 표류 금지 (산출물 차원)** — "②를 안 한다"를 산출물에서도 지킨다:
- PR 본문·문서에 "정합성 확인 완료·준비도 충족" 류 주장을 쓰지 않는다.
- 감사 로그 `docs/interim/sot-readiness-review.md`(Maintainer 소유)에 쓰지 않는다.
- `review_clear` 등 verdict 유사 산출을 내지 않는다.

**경계 근거**: 형식(①)은 결정적이다 — 같은 내용이면 같은 결과라 작성 세션이 스스로 맞춰도 편향이 없다. 정합성(②)은 판단이라 작성한 사람이 하면 편향된다. 그래서 ①은 스킬 안에서 미리 맞추고, ②는 분리한다. 이 선이 B-1과 `sot-readiness-review`를 가른다.

## 2. 언제·누가 쓰나

- **초기(그린필드)**: 대상 프로젝트의 SoT를 처음 세울 때. **Author**가 사용자와 직접 대화하며 요구·사양·테스트 설계를 작성한다. 진입점은 사용자 또는 사용자를 대리하는 초기 세션.
- **증분**: 기존 SoT에 요구/사양/테스트 설계를 추가·수정할 때. `rule-sot-change-user-gate`상 주로 Leader의 사양변경요청 → Author가 이를 입력으로 실행 → Maintainer가 PR화하는 경로. 증분 진입의 가장 흔한 트리거는 **② blocking finding 해소를 위한 SoT 수정**이다.
  - **실행 주체·대화 상대·push 권한(확정 §8-1)**: 그린필드·증분 모두 **Author**가 스킬을 돌린다(= `sot/<slug>` push 주체 — task 브랜치의 `docs/sot/**` 직접 수정은 `rule-protected-paths`가 거부한다; Phase 3 강제, 그전엔 규약). 대화 상대는 그린필드=사용자, 증분=Leader의 사양변경요청(대화 없는 입력)이다. **PR 발의(생성)는 Maintainer만** 한다.

두 경우 산출은 동일하게 `sot/<slug>` PR이다.

## 3. 대화 흐름 (현재 템플릿 기준 예시)

**중요**: 아래 섹션 목록은 고정이 아니다. 스킬은 세션 시작 시 현재 `_TEMPLATE.md`를 읽어 필수 섹션·frontmatter를 추출하고 그에 맞춰 흐름을 구성한다(§4). 아래는 현재 템플릿 기준의 예시 흐름이다.

**기본 자세**: 백지 질문을 던지지 않는다. 에이전트가 각 섹션의 **초안을 먼저 제안**하고 사용자가 수정/승인한다. 진짜 결정 지점만 질문하며, 모호하면 선택지 2~3개로 좁혀 제시한다(단일 추천으로 수렴하는 프로젝트 원칙과 일치).

**사용자 부재 경로(§8-5)**: 아래 흐름은 사용자가 있는 경우 기준이다. 사람이 없으면 제안→승인·재확인 지점은 PR 리뷰(③ 사용자 게이트)로 이월되고, 입력(Leader 요청문)에서 도출할 수 없는 결정은 가정으로 표기해 PR 본문에 싣는다. 사람 유무가 대화 여부를 가른다.

원칙: **요구사항 먼저(왜/무엇) → 사양(무엇/어떻게) → 테스트 설계(어떻게 검증)**. 주제(topic) 하나를 이 순서로 이어 완성하고 다음 주제로 넘어간다 — 요구·사양·테스트 설계를 주제별로 함께 다뤄야 `covers` 추적을 만든 자리에서 바로 잇는다. 섹션 단위로 좁혀 다룬다. 세 문서는 **항상 한 `sot/<slug>` PR에 함께 실린다**(부분 SoT PR 금지 — 확정 §8-3).

1. **주제·파일 확정** — `topic`(소문자 kebab-case), 파일명 `docs/sot/requirements/<topic>.md`·`docs/sot/specification/<topic>.md`·`docs/sot/test-design/<topic>.md`, frontmatter `id`(`req-<topic>`/`spec-<topic>`/`td-<topic>`), PR 브랜치 `sot/<slug>`. 증분이면 대상 문서 선택 + 변경 범위 확정(브랜치 재사용 금지, 같은 주제 재변경은 `sot/<slug>-2` 식 접미).
2. **문서 분해 결정** — 요구↔사양↔테스트 설계는 셋 다 다대다다. 하나의 요구가 복수 사양으로 갈리거나 기존 사양 확장으로 흡수될 수 있고, 테스트 설계도 요구·사양과 분해 축이 다를 수 있다(한 테스트 설계가 여러 요구·사양을 검증하거나 기존 test-design 확장으로 흡수). **신규 vs 기존 확장, 세 문서의 분해 축**을 먼저 정한다.
3. **요구사항 작성** — 목표 → 범위 외 → 기능 요구(`FR-n`) → 비기능 요구(`NFR-n`) → 제약 → `related` 힌트. 각 `FR`/`NFR`을 확정한 **직후 그 자리에서 수용 기준을 함께** 확정한다(끝에 몰아 받지 않는다 — filler 방지). 수용 기준 대본: "무엇을 관찰하면 통과인가 / 측정치·조건은 / 실패는 어떻게 드러나는가".
4. **사양 작성** — 범위 → 구성요소(`SP-n`) → 인터페이스 → 데이터 모델 → 동작(정상 경로 + 에지 케이스). 각 `SP` 직후 수용 기준 → `covers`(구체화하는 `FR-n`/`NFR-n` 링크).
5. **테스트 설계 작성** — 범위 → 테스트 조건(`TD-n`) → 커버리지 목표(블랙박스 기법: 동등분할·경계값·결정표·상태전이) → 스위트 구조 → 추적성(요구·사양↔`TD` 매핑 표 — `covers`와 일치, 공백·고아를 눈으로 보는 뷰) → 절차 원칙 → 수용 기준. 각 `TD` 직후 `covers`(검증하는 `FR-n`/`NFR-n`/`SP-n` 링크)를 잇는다. **절단선 안내**: 여기 담는 것은 테스트 조건·기법·경계의 식별까지 — 거기서 뽑는 구체 입력값·기대 출력, 데이터·픽스처·실행 스크립트·환경, 화이트박스 커버리지는 `test/` 코드(D12·Tester 소관) 몫이다. 이 절단선은 대화에서 사용자에게 알리되, 그 아래 항목(구체 입력값·기대 출력·데이터·픽스처·실행 스크립트·환경·화이트박스 커버리지)은 test-design에 적지 않는다(절단선 자체는 템플릿 배너로 문서에 남는다).
6. **추적·규칙 링크** — `covers`로 짝을 잇는다: 사양은 구체화하는 `FR`/`NFR`을, 테스트 설계는 검증하는 요구·사양 항목(`FR`/`NFR`/`SP`)을 가리킨다. 모든 `FR`/`NFR`이 최소 하나의 사양 `covers`에, 모든 요구·사양 항목이 최소 하나의 테스트 설계 `covers`에 등장하도록 작성한다(작성 완결성). `rules`는 이 문서가 따르는 rule id를 채운다: `docs/sot/rule/`을 스캔 → 후보 제시 → 이유 확인·선택. 특히 `scope: local` rule 선언 누락은 **낡은 완료**를 부른다(재검토 타겟팅에서 빠짐). `related`는 힌트라 무결성 검사 대상이 아니고, `covers`/`rules`는 ① 무결성 + 재검토 타겟팅의 입력이다. **이 링크들이 ② 통과를 보장하지는 않는다** — 커버가 충분한지, 짝이 정합한지는 ②의 판단이다.
7. **역행 루프 (횡단 규칙 — 순차 단계 아님)** — 2~6 어느 지점에서든, 특히 사양·테스트 설계를 쓰다 요구·사양의 결함·누락을 발견하면 해당 항목으로 복귀·수정 → 사용자 재확인 → 영향받는 `covers` 재점검. `topic`/`slug` 재확정도 PR 제안 전까지 허용.
8. **① 형식 자가검사** — §5. 증분이어도 변경분만이 아니라 **결과 트리 전체(또는 영향받는 req/spec/test-design 전체)**에 대해 수행한다(참조 무결성이 국소 변경으로 원거리 문서를 깨뜨릴 수 있어서다. 커버리지 점검은 ①이 아니라 §3-6 작성 흐름 소관).
9. **PR 제안** — `sot/<slug>`에 얹어 PR 초안 제시(발의=Maintainer), 사용자 게이트 대기. PR 본문에 **핵심 결정·기각 대안 요약을 필수**로 담고(대화에서 내려진 비자명 결정이 증발하지 않게), 촉발 조건에 걸리는 비자명 설계 결정은 interim ADR 생성을 권고한다. ADR 파일 생성 자체는 강제가 아니되, 촉발 시 **PR 본문에 ADR 링크 또는 생략 사유를 남기는 것**은 강제층이다(확정 §8-4 — `rule-adr-recording`이 촉발 조건과 2층 구조를 규정).

증분 수정이면 2~7을 변경분 중심으로 진행하되(1의 대상·범위 확정은 위와 동일), 8의 자가검사만은 §3-8대로 결과 트리 전체(또는 영향받는 req/spec/test-design 전체)에 건다.

## 4. 기존 템플릿과의 관계

스킬은 구조를 새로 정의하지 않는다. **세션 시작 시 현재 `_TEMPLATE.md`(requirements·specification·test-design)를 읽어 필수 섹션·frontmatter를 소스로 삼는다.** 정본은 템플릿이고 스킬은 그것을 채우는 대화 절차만 규정한다. §3의 섹션 나열은 현재 템플릿 기준 예시일 뿐이며, 구현자가 이를 하드코딩하면 안 된다 — 템플릿이 바뀌면 스킬은 자동으로 새 구조를 따라야 한다.

또한 §3의 대화 순서(목표→범위 외→FR→…)는 **채워 넣는 순서일 뿐**이고, **최종 문서의 섹션 배치는 템플릿 순서를 따른다**(대화 순서로 문서를 재배열하지 않는다).

## 5. ① 형식 자가검사 항목 (rule ①과 일치)

`rule-sot-readiness`가 Phase 6 CI에 배정한 ① 형식 검사와 **같은 항목**을 스킬이 미리 맞춘다(게이트 재현이 아니라 사전 정렬 — 최종 강제는 CI). 규칙 ①(`sot-readiness.md`)의 항목을 그대로 따른다.

- requirements·specification·test-design **각각**, 항목이 담긴 문서가 최소 1개(`README`·`_TEMPLATE` 제외).
- 각 문서에 항목 ID: `FR-n`·`NFR-n`(요구), `SP-n`(사양), `TD-n`(테스트 설계).
- **참조 무결성**: `covers`·`rules`가 가리키는 항목·rule id가 실재(dangling 없음). 테스트 설계 `covers`가 가리키는 요구·사양 항목도 포함. `related`는 힌트라 검사 제외.
- 미치환 플레이스홀더(`<...>`) 없음 — **코드 블록·인라인 코드 제외**.
- 금지어 없음 — `TBD`·`TODO`·`FIXME`·`미정`·`추후`·`나중에` 등(대소문자·한글 변형 포함). 정본 목록은 **sot-lint 스크립트**(확정 §8-2)가 관리하며, Author가 PR 전 이 스크립트를 호출한다. 단일 구현이라 CI 승격 후에도 목록 drift가 없다.
- 각 요구·사양·테스트 설계 항목의 수용 기준 자리에 **플레이스홀더·금지어가 아닌 문장이 존재**. 검증 가능한지·filler인지는 판정하지 않는다 — ②의 몫이다.

**커버리지 완결성**(모든 요구가 사양으로 구체화되고, 모든 요구·사양 항목이 테스트 조건으로 검증됨)은 ①이 아니라 **② 축1**이다. 작성 흐름(§3-6)에서 챙기되, 자가검사 게이트에는 넣지 않는다(경계 유지).

## 6. 테스트 설계 — SoT 4번째 타입 (확정)

TODO·루트 `README.md`가 이 스킬의 산출로 드는 **테스트 설계**(이전 초안에서 "테스트 문서"로 부르던 것)는 **`docs/sot/test-design/`의 4번째 SoT 타입**으로 확정됐다(요구·사양과 동급, D16·`ADR-0008`). 이 스킬은 요구·사양·테스트 설계 셋을 함께 작성하며, 세 문서는 한 `sot/<slug>` PR에 함께 실려 완료 판정(①②③)을 받는다.

**test-design이 담는 것 (SoT)**: 테스트 조건(`TD-n`)·블랙박스 커버리지 목표(동등분할·경계값·결정표·상태전이)·스위트 구조·요구·사양 ↔ 테스트 추적(`covers`)·절차 원칙·수용 기준. 스킬은 이 항목을 §3-5 흐름으로 채운다.

**담지 않는 것 (`test/` 코드, D12·Tester 소관)**: 테스트 조건·기법·경계에서 뽑는 구체 입력값·기대 출력, 데이터·픽스처·실행 스크립트·환경, 화이트박스 커버리지(결정/분기/MC-DC) 측정. 이들을 SoT 완료 요건으로 요구하면 코드 없이 충족 불가라 완료가 영구 미충족이 되므로 SoT에서 뺀다(`test-design/README`·`ADR-0008`). 스킬은 대화에서 이 선을 사용자에게 알리고 test-design에 구체 케이스·픽스처를 적지 않는다.

> 이 절은 이전 초안에서 "테스트 문서 = 최상위 열린 결정"으로 (A) 별도 타입 불필요 / (C) interim 테스트 계획 / (B) 별도 SoT 타입을 저울질하던 논의를 대체한다 — 그 논의는 (B)에 해당하는 결정으로 **종결**됐다(superseded). 루트 `README.md`도 이에 맞춰 "요구사항·사양·테스트 설계"로 갱신됐다.

## 7. 구현 형태

- **단일 스킬**, req·spec·test-design을 함께 다룬다(신규는 주제별로 요구→사양→테스트 설계 순차, 증분은 대상 지정). 별개 스킬로 쪼개지 않는다 — 요구·사양·테스트 설계를 한 세션에서 다뤄야 `covers` 추적을 만든 자리에서 잇는다. 세 문서는 항상 한 PR에 동반한다(부분 SoT PR 금지 — §8-3).
- **실행 주체**: Author가 스킬을 돌리고 PR을 제안하며, PR 발의(생성)는 Maintainer가 한다(§8-1).
- 산출: `.claude/skills/<name>/SKILL.md` + 대화 대본/체크리스트 + **sot-lint 검사기**(Python, §8-2).
- **PR 이후 라이프사이클**: ② 실행·강제는 규칙상 **호스트 CI(Phase 6)** 소관이다. Phase 6 이전엔 ②가 자동 실행되지 않는 공백이 있으나, B-1은 이 공백을 스스로 메우지 않으며 **별도 장치도 두지 않는다**(확정 §8-6). 작성 세션이 ②를 돌리면 자기검토 편향이고, 그 공백은 AXDT 자체 개발·테스트에만 걸리는 구간이라(실사용 SoT는 완성 후 배포) 메울 실익이 없다. Phase 6 이후엔 호스트 CI가 ②를 자동 실행한다.
- ① 자가검사는 **sot-lint 스크립트**로 수행한다(확정 §8-2). B-1 구현 산출물로 지금 만드는 결정적 Python 검사기이며, Author가 PR 전 호출해 clean까지 고친다. Phase 6에 axdt 패키지로 승격해 CI 필수 검사로 배선하되(검사 코드는 신뢰 ref/base에서 읽음) 단일 구현이라 CI-스킬 간 drift가 없다.

## 8. 확정된 결정 (브레인스토밍 종결)

B-1 브레인스토밍(2026-07-08)에서 아래 6개를 확정했다. 원 안건 번호를 유지한다. 파급(루트 README 역할표·`rule-terminology`에 Author 반영, `rule-adr-recording` 정식 승격, 도그푸딩 ADR)은 사용자 스펙 리뷰 승인 후 별도 사용자 게이트 PR로 처리한다.

1. **실행 주체 × 대화 상대 × push 권한 — Author 역할 신설.** SoT(요구·사양·테스트 설계) 저술 전용 역할 **Author**를 신설한다. 그린필드는 Author가 사용자와 대화하며, 증분은 Leader의 사양변경요청이 대화 없는 입력이 된다. Author가 `sot/<slug>`에 변경을 얹지만 **PR 발의(생성)는 Maintainer만** 한다. task Leader는 `rule-protected-paths`상 `docs/sot/**`를 직접 push할 수 없고 요청만 발의한다.
2. **① 형식 자가검사·금지어 — sot-lint 스크립트.** ①은 결정적이라 스크립트로 낸다. B-1 구현 산출물로 **sot-lint**(Python, D9) 검사기를 지금 만들고, Author가 PR 전 호출해 clean까지 고친다. Phase 6에 axdt 패키지로 승격해 CI 필수 검사에 배선한다(검사 코드는 신뢰 ref/base에서 읽는다). 검사기가 단일 구현이라 CI와 스킬 사이 금지어 목록 drift가 없다.
3. **req·spec·test-design PR 단위 — 항상 세 문서 동반.** 세 문서를 항상 한 `sot/<slug>` PR에 함께 싣는다. 부분 SoT PR은 금지하며, 증분에서도 일부만 분리하지 않는다(Author authoring 정책 — 분리를 제안하지 않는다).
4. **작성 근거 보존 — PR 요약 필수 + ADR 권고.** PR 본문에 핵심 결정·기각 대안 요약을 필수로 담는다. ADR 파일 생성 자체는 권고(강제 아님)이되, 촉발 조건에 걸리면 **PR 본문에 ADR 링크 또는 생략 사유를 남기는 것**이 강제층이다. 이 판단을 규칙화해 `rule-adr-recording` 초안을 작성했다(advisory·`scope: local`·2층: PR 요약·인정이 바닥선, ADR 파일이 촉발 시 승격층). 정식 승격은 파급 단계.
5. **자동 파이프라인에서 대화형 스킬의 위상 — 최선노력 단일 스킬.** 하나의 스킬이 사람이 있으면 대화 authoring을, 없으면 입력(Leader 요청문)에서 초안→PR을 낸다. "대화형"은 가능하면 대화, 아니면 PR 리뷰라는 뜻이다. 사람 개입의 불변 지점은 어느 경우든 **③ 사용자 게이트**다.
6. **Phase 6 이전 ② 공백 — 별도 장치 없음.** ② 자동 강제는 Phase 6 CI가 담당한다. 그전 공백에는 별도 장치를 두지 않는다 — 그 구간은 AXDT 자체 개발·테스트에만 걸리고(실사용 SoT는 완성 후 배포) 메울 실익이 없다. 애드혹 수동 `sot-readiness-review` 호출은 가능하나 게이트 인접 장치로 규정하지 않는다.
