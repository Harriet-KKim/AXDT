# 검토/보고용 HTML 스킬 (`review-report`) 설계

> 상태: **draft**. 사용자 검토 후 확정한다.
> 대응: 사용자 요청(검토/보고용 HTML을 체계적으로 쓰는 스킬), Backlog 373(용어집)·374(워크플로 도식) 연계.
> 성격: AXDT 산출물이 아니라 **개발 과정에서 쓰는 개인 도구**. 스킬 본체는 리포 밖(`~/.claude/skills/`), 프로젝트 데이터만 리포 안(`WIP/reviews/`).

---

## 1. 목표와 비목표

### 목표
- Spec·설계·결정사항을 **맥락이 충분히 담긴 HTML**로 보고하고, 사용자가 그 위에 **검토 의견을 주입**하며, 의견 수렴 후 재검토로 내용이 바뀌면 **같은 페이지(같은 URL)에 반영**하는 반복 루프를 스킬 하나로 고정한다.
- 어려운 용어는 **풀어서** 보이되, 풀이의 단일 진실원은 스킬이 소유하지 않고 **프로젝트 용어집 문서를 참조**한다.
- 공통 반복 요소(껍데기·배경·용어 풀이)는 **공용 파일에서 조립**하고, 매 라운드 새로 쓰는 것은 본문뿐으로 한정한다.
- 페이지에 **세션 이름·주제 요약·라운드 수**를 필수로 싣고, 빠지면 빌드가 실패하게 한다.

### 비목표 (하지 않음)
- **페이지→에이전트 자동 왕복** — Artifact의 CSP가 외부 통신(`fetch`/XHR/WebSocket)을 막으므로 실시간 전송은 성립하지 않는다. 의견은 클립보드(기본)·브라우저 되읽기(보조)로 회수한다.
- **용어집 신규 저술** — Backlog 373의 몫. 이 스킬은 소비자다.
- **다중 리뷰어·권한 모델** — 리뷰어는 사용자 1명.
- **라운드 간 본문 diff 렌더** — Artifact 버전 이력이 대신한다.
- **용어집·배경 캐시** — 매 빌드 원본을 다시 읽는다(드리프트 방지, 파싱 비용 무시 가능).

## 2. 핵심 설계 결정

### 2.1 산출 경로 = Artifact, 라운드 = 같은 URL 덮어쓰기
로컬 HTML 파일이 아니라 **Artifact로 게시**한다(개인 규칙: HTML 산출물은 Artifact). Artifact는 같은 파일 경로(또는 `url`)로 재게시하면 **같은 URL에 덮어쓰고 버전 이력**을 남긴다. 이것이 "같은 페이지에 반영"의 메커니즘이다. 2라운드부터 `meta.json`의 `artifact_url`을 `url`로 넘겨 덮어쓰고, `label`에 `"Round N"`을 붙여 이력에 남긴다.

### 2.2 공용은 파일로 분리, 조립기가 본문과 합성
Artifact는 자체 완결형이라 런타임에 외부 파일을 못 끌어온다. 따라서 "공용 참조"는 **게시 시점 인라인**이다. 껍데기(`shell.html`)와 조립기(`build.py`)를 스킬에 고정하고, 매 라운드 쓰는 것은 `body.html`뿐이다. 의견 UI의 `localStorage` 키 스코프·클립보드 직렬화처럼 **정확성이 중요한 코드**를 한 번 짜서 고정하면, 라운드마다 다시 쓰다 어긋나는 일이 없고 토큰도 준다.

### 2.3 용어집은 소유하지 않고 참조
스킬이 자기 용어집을 들면 손사본이 되어, Backlog 373이 완성되는 순간 두 벌로 갈라진다. 대신 `config.json`의 `glossary_sources`가 프로젝트 문서를 가리키고, 조립기가 빌드마다 파싱해 `{용어→풀이}` 맵을 만든다. 단일 진실원은 계속 `docs/sot/rule/` 쪽에 남는다. 지금은 `terminology.md` 하나로 시작하고, 373이 끝나면 배열에 경로 한 줄을 더한다.

파싱은 형식을 강제하지 않고 흔한 두 형태를 받는다: `- **용어** — 풀이`(정의 목록), `| 용어 | 풀이 |`(표). 못 찾은 용어는 조용히 넘기지 않고 빌드 로그에 **"풀이 없는 용어 K개"**로 남긴다 — 그 수가 곧 Backlog 373의 우선순위 근거가 된다.

### 2.4 필수 메타데이터는 문서가 아니라 빌드가 강제
`session`·`topic`·`round` 중 하나라도 `meta.json`에 없으면 조립기가 **exit 1로 멈춘다**. "필수"를 규약으로만 적으면 결국 빠뜨린다. 세 값은 페이지 최상단 히어로에 항상 렌더된다.

### 2.5 의견 앵커 = 본문 문장이 아니라 섹션 안정 ID
검토 페이지는 본문이 라운드마다 고쳐지는 게 전제다. 문장에 하이라이트를 걸면 그 문장이 다음 라운드에 바뀌는 순간 앵커가 깨진다. 그래서 앵커는 **섹션에 부여한 안정 ID**(`SP-3`·`D-2`…)다. 본문 최상위는 `<section data-anchor="SP-3" data-title="…">`의 반복이고, ID는 한 번 부여하면 라운드를 넘어 유지된다. `meta.json`이 ID 목록을 들고 있어 조립기가 대조하며, ID가 사라지면 그 섹션의 이전 의견이 고아가 되므로 경고한다.

### 2.6 의견에는 종류 태그
텍스트 뭉치로만 돌아오면 에이전트가 "반드시 고칠 것인지, 단순 질문인지"를 추측한다. **수정요청·질문·이견·승인** 네 종류로 태깅해, 다음 라운드 처리 방식을 명확히 한다. 페이지 하단엔 어느 섹션에도 안 붙는 **총평(GENERAL)** 칸을 둔다.

### 2.7 의견 회수 = 클립보드(기본) + 브라우저 되읽기(보조)
의견은 입력 즉시 `localStorage`의 **라운드별 스코프 키** `rr:<slug>:r<round>`에 저장된다. 라운드로 키를 나누지 않으면, 같은 URL 덮어쓰기라 origin이 같아 Round 2 페이지에 Round 1 초안이 되살아난다. 상단 고정 바의 `[의견 복사]` 버튼이 사람이 읽어도·기계가 파싱해도 되는 블록을 클립보드에 넣는다(§4.3). 붙여넣으면 회수 완료. Chrome 확장이 살아 있으면 에이전트가 탭을 열어 `localStorage`를 직접 읽어 붙여넣기를 생략한다. 이전 라운드 키는 지우지 않고 남긴다 — 사용자가 회수 전에 실수로 새 라운드를 열어도 초안이 살아 있고, 키가 라운드로 갈려 새 라운드 입력과 섞이지 않는다.

### 2.8 본문은 누적하지 않고 최신만, 이전 의견은 처리 결과 카드로
페이지 본문은 항상 **최신 안**만 담는다. 각 섹션에 그 섹션에 달렸던 이전 라운드 의견과 처리 결과가 접힌 카드로 붙는다: **반영 / 일부 반영 / 보류 / 다르게 감** + 한 줄 사유. 이전 라운드의 페이지 전체 모습은 Artifact 버전 이력이 보존한다. 결과적으로 사용자는 한 화면에서 (a) 최신 안, (b) 지난 지적이 어떻게 됐는지를 함께 본다.

### 2.9 시각·구조는 확립된 집 스타일을 코드로 고정
`shell.html`은 임의 디자인이 아니라 사용자가 명시 승인한 **HTML 산출물 집 스타일**(메모리 `html-deliverable-house-style`)을 구현한다. 이 스킬은 그 집 스타일을 매번 재지정하지 않고 한 파일로 굳히는 그릇이다.

- **구조(자립·해설형)**: 상단 eyebrow 라벨 + 제목 + lede(한 문단 도입) → 배경/읽는법 박스 → 목차(TOC) → 번호 매긴 장 → 하단 용어표 + 메타 푸터. 대화 밖에서 페이지만 봐도 읽히게 자립적으로 구성한다. 필수 메타(세션·주제·라운드)는 상단 히어로에, 출처 메타(대상 파일·PR·생성일)는 푸터에 둔다.
- **시각 시스템**: CSS 변수 토큰으로 팔레트를 정의하고 라이트·다크를 양방향 설계한다(`prefers-color-scheme` 기본 + `:root[data-theme]` 오버라이드가 양쪽에서 이긴다). 절제된 뉴트럴(살짝 hue bias) + 강조색 1 + 보조 강조 1 + 의미색(빨강=문제/치명, 초록=해결/통과, 호박=블로커). 시스템 폰트 스택만(웹폰트·CDN은 Artifact CSP가 차단), 식별자·해시·`file:line`은 monospace. 카드 + 콜아웃(note/warn) + 둥근 모서리 + 옅은 그림자, 표는 `overflow-x` 스크롤, 반응형, `prefers-reduced-motion` 존중. AI 기본 룩(크림+세리프+테라코타)은 피한다.
- **정합**: 이 집 스타일은 즉석 해설 HTML 요청에도 적용되는 범용 지침이라, 이 스킬로 완전히 대체되지 않는다(§9). 세부 디자인 판단은 `artifact-design` 스킬이 보강한다.

## 3. 파일 배치

```
~/.claude/skills/review-report/     (개인 스킬 — 모든 프로젝트 공용, worktree 복제 안 됨)
  SKILL.md            워크플로: 언제 쓰나 · 라운드 절차 · 본문 작성 규칙 · 앵커/용어 문법
  build.py            조립기 (py -3, 표준 라이브러리만)
  test_build.py       조립기 테스트
  shell.html          껍데기(집 스타일 §2.9): eyebrow+lede · 배경박스 · TOC · 토큰 라이트/다크 · 의견 UI · 메타 푸터 · 클립보드 직렬화

<프로젝트>/WIP/reviews/           (프로젝트 데이터 — git 추적)
  config.json         glossary_sources · context_file
  context.md          배경 박스 원문 (1회 작성, 프로젝트 소개)
  <slug>/             검토 세션 하나 = 아티팩트 URL 하나 = 라운드 시퀀스 하나
    meta.json         session · topic · round · artifact_url · anchors[]
    body.html         본문 (매 라운드 갱신하는 유일한 파일)
    comments.json     라운드별 의견 + 처리 결과 (진실원)
    out.html          생성물 (.gitignore)
```

### 배치 근거
- **개인 스킬**: AXDT는 worktree가 5개라 프로젝트 스킬은 그만큼 복제된다(`sot-readiness-review`가 그렇다). 개인 스킬은 복제 문제가 없고 다른 프로젝트에서도 쓴다.
- **`WIP/reviews/`**: `docs/interim/`은 AXDT가 **만들어낼 제품**의 산출물 스키마(ADR/plan/report/progress) 자리다. `rule-terminology`가 "AXDT 자체 설계 문서는 `WIP/`에 두며 본 분류와 별개"라고 명시하므로, 검토 데이터도 `WIP/` 아래가 맞다.
- **평면 `<slug>`**: 폴더가 갈리는 단위는 phase가 아니라 **검토 세션 하나**다(불변식: slug=URL=라운드 시퀀스). phase 접두(`phase3-git-isolation`)로 정렬은 자연스럽고, 여러 phase에 걸치는 주제를 어느 phase에 둘지 고민할 필요가 없다.
- **`comments.json` 단일 진실원**: 사람이 읽을 마크다운 사본을 따로 두지 않는다(TODO.md/TODO.html 미러가 겪는 드리프트 회피). git diff에서 JSON을 읽게 되는 비용은 감수한다. 사람이 보는 뷰는 렌더된 페이지가 제공한다.
- **`out.html`만 무시**: `body.html`의 git 이력이 라운드별 본문 이력, `comments.json`이 감사 기록. 생성물은 재현 가능하므로 추적하지 않는다.

## 4. 데이터 계약

### 4.1 `config.json` (프로젝트당 1개)
```json
{
  "glossary_sources": ["docs/sot/rule/terminology.md"],
  "context_file": "WIP/reviews/context.md"
}
```
경로는 프로젝트 루트 기준. `glossary_sources`는 순서대로 병합하며, 같은 용어가 겹치면 **뒤 소스가 이긴다**(더 전문화된 사전이 나중에 온다는 가정).

### 4.2 `meta.json` (검토 세션당 1개)
```json
{
  "session": "phase1 SoT 게이트 설계",
  "topic": "게이트 실패 시 재시도·재개 정책 확정",
  "round": 2,
  "artifact_url": "https://claude.ai/public/artifacts/…",
  "anchors": ["SP-3", "SP-5", "D-2"],
  "target_files": ["docs/sot/rule/sot-readiness.md"],
  "pr": "#12",
  "created": "2026-07-13"
}
```
`session`·`topic`·`round` 필수(없으면 빌드 실패) — 상단 히어로에 렌더. `artifact_url`은 1라운드 게시 후 채운다(없으면 신규 게시). `anchors`는 `body.html`과 대조용. `target_files`·`pr`·`created`는 집 스타일 메타 푸터(§2.9)에 렌더하는 출처 메타로, 있으면 표시하고 없으면 생략한다.

### 4.3 클립보드 직렬화 형식
```
### 검토 의견 — phase1 SoT 게이트 설계 / Round 2
<!--rr:meta slug=sot-gate round=2-->

[SP-3] 수정요청
3회는 너무 많음. 2회로 줄이고 실패 시 즉시 사용자에게.

[SP-5] 질문
트리 해시가 바뀌면 accepted가 무효화된다는데, rule 지문만 바뀐 경우도 그런가?

[GENERAL] 승인
나머지는 이대로 좋음.
```
HTML 주석의 `slug`·`round`가 회수한 의견이 어느 페이지 몇 라운드인지 확정한다. 같은 내용이 `localStorage`에 JSON으로도 있어, 브라우저 되읽기 경로도 동일 데이터를 쓴다.

### 4.4 `comments.json` (진실원)
```json
{
  "rounds": [
    {
      "round": 1,
      "items": [
        { "anchor": "SP-3", "kind": "수정요청", "text": "…",
          "resolution": "반영", "note": "2회로 축소, ADR-0012 갱신" }
      ]
    }
  ]
}
```
`resolution`·`note`는 에이전트가 다음 라운드에서 채운다. `kind` ∈ {수정요청, 질문, 이견, 승인}. `resolution` ∈ {반영, 일부 반영, 보류, 다르게 감}. 이 값이 §2.8 처리 결과 카드로 렌더된다.

## 5. 본문 작성 문법 (`body.html`)

```html
<section data-anchor="SP-3" data-title="게이트 실패 시 재시도 정책">
  <p>재시도는 최대 <b-term def="지정한 횟수만큼만 반복">3회</b-term>까지 하고…</p>
</section>
```
- 최상위는 `<section>` 반복(집 스타일의 "번호 매긴 장"), 각각 `data-anchor`(안정 ID)·`data-title` 필수. 장 번호와 TOC는 조립기가 순서대로 매긴다 — 본문에 번호를 박지 않는다.
- 용어 풀이는 **쓰기 전에** 준다(집 스타일). 두 경로: (a) 용어집에 있으면 첫 등장을 조립기가 자동으로 감싼다, (b) 용어집에 없으면 `<b-term def="…">`로 본문에서 직접 단다.
- 조립기가 감싸지 않는 곳: 코드 블록·제목·이미 `<b-term>`으로 감싼 곳.
- 워크드 예제는 집 스타일의 Q&A/before-after 패널을 쓴다: `<div class="rr-panel" data-kind="qa|beforeafter">`. 콜아웃은 `<div class="rr-note">`·`<div class="rr-warn">`.

## 6. 조립기 계약 (`build.py`)

`py -3 build.py WIP/reviews/<slug>` 의 절차:

1. 프로젝트 루트 탐색 → `reviews/config.json`·`<slug>/meta.json` 로드. 필수 필드 없으면 **exit 1**.
2. `body.html`의 `data-anchor` 수집 → `meta.json.anchors`와 대조. 소실 앵커는 경고(그 앵커의 이전 의견이 고아).
3. `glossary_sources` 파싱(정의 목록·표 두 형식) → `{용어→풀이}` 맵.
4. 본문 텍스트에서 용어 첫 등장 감싸기(코드·제목·기존 `<b-term>` 제외).
5. `<section>` 순서대로 장 번호·**TOC** 생성(집 스타일 §2.9).
6. `context.md` → 배경/읽는법 박스, `comments.json` → 이전 라운드 처리 카드, 용어 맵 → 하단 용어표, `meta.json`의 출처 메타 → 푸터.
7. `shell.html` 자리표시자 치환 → `out.html` 기록.
8. 요약 출력: `섹션 N · 용어 M개 감쌈 · 풀이 없는 용어 K개 · 소실 앵커 목록`.

의존성은 표준 라이브러리만(개인 규칙: `python`은 깨진 stub, `py -3` 사용). 게시는 조립기가 하지 않고, 에이전트가 `out.html`을 Artifact 툴로 올린다.

---

## 7. 라운드 절차 (SKILL.md가 규정)

1. **새 검토 시작**: `<slug>/` 생성, `meta.json`(round=1)·`body.html` 작성. 프로젝트에 `config.json`·`context.md` 없으면 먼저 만든다.
2. **빌드**: `py -3 build.py …` → `out.html`. 경고(소실 앵커·풀이 없는 용어) 확인.
3. **게시**: Artifact 툴로 `out.html` 게시. 1라운드는 신규, `artifact_url`을 `meta.json`에 기록. 2라운드부터 `url`로 덮어쓰기 + `label:"Round N"`.
4. **회수**: 사용자가 붙여넣거나(클립보드), 확장이 있으면 에이전트가 `localStorage` 되읽기.
5. **반영**: 회수한 의견을 `comments.json`에 적재하고 `resolution`·`note` 결정, `body.html` 갱신, `round`+1. 2로 돌아간다.

## 8. 테스트 (`test_build.py`)

- 필수 메타 누락 → exit 1 (session·topic·round 각각).
- 용어집 두 형식(정의 목록·표) 파싱 정확성 + 겹칠 때 뒤 소스 우선.
- 용어 감싸기: 첫 등장만·코드/제목/기존 `<b-term>` 제외.
- 앵커 대조: `meta.json`에 있으나 `body.html`에 없는 앵커 경고.
- 풀이 없는 용어 카운트 정확성.
- `comments.json` → 처리 카드 렌더(4종 resolution).
- 집 스타일 골격 렌더(§2.9): TOC 자동 생성·장 번호 순서, 출처 메타 푸터(필드 있음/없음), 라이트/다크 토큰 존재.
