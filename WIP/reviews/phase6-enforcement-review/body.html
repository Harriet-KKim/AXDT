먼저 세 줄로 요약한다.

  • 한 일: SoT 완료 강제 증분에서, 라이브 GitHub 없이 지금 확정할 수 있는 두 산출물을 작성했다 — 룰셋 선언(ENFORCEMENT_MATRIX.md)과 실행부 Phase 9 구현 계획서.
  • 현재 상태: Codex·Fable 두 모델의 독립 리뷰를 8라운드 반복해 지적이 0으로 수렴했다. 마지막 라운드에서 두 리뷰어 모두 "지적 없음"으로 확정했다.
  • 결정할 것: 실제 gh api 실행부 코드는 스펙이 Phase 9(라이브 도그푸딩) 몫으로 규정해 이번에 구현하지 않았다. 다만 검토 중 정본 규칙 2건의 드리프트를 발견했고, 강제 게이트 활성화 전에 사용자 게이트로 정정해야 한다(아래 "정본 드리프트" 참조).

이 검토의 두 대상 파일은 WIP/axdt/sot_gate/ENFORCEMENT_MATRIX.md와 WIP/plans/2026-07-19-phase6-enforcement-executor-live-impl.md이다. 작업은 격리된 git 워크트리(phase6-enforcement-executor 브랜치)에서 진행했고, 라운드마다 커밋해 반영 이력을 남겼다.

Phase 6 완료 정의는 스펙이 "(가) Git 호스트 클라이언트 ∧ (나) 강제 증분"으로 못박아 두었다. (가) 클라이언트는 이미 병합됐고, 남은 것은 (나) 강제 증분이다. 강제 증분의 게이트 판정 논리(순수 코어)는 이미 완성·검증됐고(테스트 152개 통과), 유일한 공백은 게이트가 GitHub과 실제로 대화하는 실행부(hosts/github.py의 7개 함수)가 빈 껍데기라는 점이었다.

그 실행부를 이번에 채우려 했으나, 조사 결과 스펙이 실행부의 gh api 구현을 명시적으로 잠정(provisional)으로, Phase 9 라이브 도그푸딩 몫으로 규정하고 있었다(스펙 §1 비목표·§8·§9). 지금 추측으로 구현하면 스펙의 "확정 전 구현 금지"를 어기고, 라이브에서 다시 뜯어야 한다.

정황상 닫힌 PR #22가 바로 이 벽으로 보인다 — 그 PR은 실행부 포트 5개를 구현했으나 병합되지 못하고 닫혔고, 본문 스스로 "승인 편집 탐지 = TODO(Phase 9)", "ENFORCEMENT_MATRIX.md 미작성", "glob 매칭기 = 로컬 임시 구현"이라 한계를 자인했다. 닫힘 사유 코멘트가 없어 단정은 아니다.

그래서 이번 범위를 "라이브 없이 확정 가능한 부분"으로 재정의했다(사용자 승인). 룰셋 파라미터는 2026-07-09 라이브 실증으로 이미 확정됐으니 선언 문서로 남기고, 실행부는 라이브 착수 시 그대로 실행할 수 있는 정밀 계획서로 남긴다.

ENFORCEMENT_MATRIX.md는 게이트의 verify_ruleset_config 함수가 라이브 호스트 룰셋을 대조할 선언 상태의 단일 진실원이다. 머지 컨트롤러는 기동 시와 매 머지마다 라이브 룰셋을 이 문서의 선언과 값 단위로 대조하고, 어긋나면 머지를 거부한다(fail-closed).

세 룰셋을 반드시 분리해 선언한다.

  • RS-A — main 갱신 제한: 컨트롤러 계정만 main을 갱신할 수 있게 하고, 나머지는 전부 막는다.
  • RS-B — 승인·감사: 필수 승인 1건·dismiss-stale·머지 커밋 방식만 허용·force-push 차단. 우회 신원을 비운다 — 그래야 컨트롤러도 승인 관문에 걸린다.
  • RS-C — sot/* 단조 전진: sot/* 브랜치의 force-push·삭제만 막아, 컨트롤러가 읽는 도중 브랜치가 되감기는 경합(ABA)을 닫는다.

세 룰셋을 하나로 합치면 안 되는 이유는 라이브 실증으로 확인됐다 — 우회가 룰셋 단위라, RS-A 통과용으로 컨트롤러를 우회 신원에 넣는 순간 같은 룰셋의 승인 요구까지 함께 건너뛰어 승인 0건 PR이 머지됐다. 문서는 사람이 읽는 표에 더해, 게이트가 파싱할 기계 판독 선언 블록과 값 정확 대조 규칙, 문법 위반 시 fail-closed 조건을 함께 싣는다.

계획서는 hosts/github.py의 7개 포트를 실제 gh api로 구현하고 두 사전 관문(룰셋 점검·강제-필수 경로 판독)을 배선하는 작업을, 라이브 착수 시 그대로 실행할 수 있도록 Task 단위로 나눈 것이다.

핵심 원칙 두 가지를 문서 전체에 관철했다.

  • 라이브 스키마 캡처 우선: 각 포트는 실제 gh api 응답을 먼저 캡처해 확인한 뒤, 그 캡처를 시험 데이터로 삼는다. 추측 데이터로 시험을 먼저 굳히면 Phase 9가 확정해야 할 스키마를 미리 동결하기 때문이다.
  • 동결 계약 확장은 조용히 하지 않는다: 순수 코어·포트·컨트롤러의 시그니처는 동결 상태다. 이를 바꾸는 항목(머지 반환형·두 키 취득 방식·감사 저장소·기동 점검 등)은 전부 착수 전 승인 대상(P1~P3)으로 열거해, 계획 실행 중 몰래 바꾸지 못하게 했다.

실제 GitHub 룰셋 적용·② 검토 CI 생산자 준비·초기 마이그레이션 실행·정본 개정은 이 계획의 코드 범위 밖이므로, 착수 전제(P1~P3)와 활성화 절차(P4)로 명시했다. P4는 활성화 순서(RS-C→RS-B→컨트롤러 배포→마이그레이션→RS-A)와 비상 되돌리기(rollback)까지 담는다.

두 산출물을 Codex(gpt-5.6-sol)와 Fable 두 모델의 독립 리뷰에 병렬로 걸고, 각 라운드의 지적을 스펙·정본 규칙·동결 계약 코드와 직접 대조해 검증한 뒤 반영하기를 8라운드 반복했다. 막을 수 없는 지적(예: 스펙 §4.1 감시 열거 자체를 넓히라는 요구)은 물타기 대신 "정본 개정 전제조건"으로 정직하게 명시했다.

두 모델의 궤적이 달랐다. Fable은 국소 오류를 짚으며 일찍 수렴했고, Codex(고강도 추론)는 3·4라운드에 오히려 지적이 늘었다 — 계획서를 정밀화할수록 더 깊은 계약 층(감사 멱등 복구·값 도메인 검증·활성화 순서 등)을 새로 파냈기 때문이다. 이 발산을 사용자에게 보고했고, "완전 0까지 반복"이라는 지시에 따라 계속 반영한 결과 5라운드부터 Codex도 꺾여(11→7→5→2→0) 수렴했다. 매트릭스는 6라운드부터 두 리뷰어 모두 지적 없음이었다.

반영 방향의 핵심은 "미정 항목을 모두 명시적 라이브 결정·전제조건으로 라벨링"하는 것이었다. 그래야 리뷰어가 "빠뜨렸다"고 지적할 여지가 사라진다.

이 두 건은 산출물의 오류가 아니라, 정본 문서가 스펙·ADR 최신본을 반영하지 못한 드리프트다. 강제 게이트 활성화 전에 사용자 게이트 SoT PR로 정정해야 한다.

  • 규칙 sot-readiness.md의 RS-C 부정: 정본 규칙(status: active)의 강제 매핑 표(92행)와 근거(105행)가 아직 "sot/* 소스 브랜치 보호는 불요"라고 적혀 있다. 그러나 ADR-0009 결정 8은 2026-07-18에 "요구"로 뒤집혔고(RS-C 신설), 스펙 §4.1도 RS-C를 강제한다. 영향: 이 드리프트가 남으면 규칙을 따르는 셋업은 RS-C를 안 걸고, 게이트는 RS-C 부재를 불일치로 보아 모든 머지를 막는다. 조치: 규칙 두 문장을 결정 8에 맞추는 사용자 게이트 PR이 활성화보다 먼저다.
  • ADR-0009의 단일 키 문면 잔존: ADR 결정 3·4·9·10과 대가 절이 아직 "판정 키" 단수로 적혀 있으나, 스펙은 2026-07-17에 "두 키(판정 키 + 완전성 스윕 키)"로 개정됐다. 조치: ADR 전체의 단일 키 문면을 두 키 계약에 맞춰 스윕한 뒤에야 ADR-0009를 accepted로 승급한다.

산출물 2건은 이 검증 기준으로 확정 가능한 상태다(두 모델 지적 없음). 남은 것은 Phase 9 라이브 도그푸딩에서의 실제 구현이며, 계획서가 그 착수 전제를 P1~P4로 열거한다.

  • P1 — 정본 정정: 위 "정본 드리프트" 두 건(규칙 sot-readiness.md·ADR-0009)을 사용자 게이트 PR로 정정.
  • P2 — 감시 범위 정본 개정: 매트릭스가 스펙 §4.1 감시 열거보다 넓게 대조하므로, 정본 §4.1을 이 블록에 맞추는 개정.
  • P3 — 동결 계약 확장 승인: 머지 반환형·두 키 취득·감사 저장소 등 동결 시그니처 변경 8건을 착수 전 승인 또는 호스팅 래퍼로 결정.
  • P4 — 활성화 절차: ② 검토 CI 생산자 준비, 초기 마이그레이션, 룰셋 적용 순서, 비상 rollback.

이 전제들이 처리되면 계획서의 Task 0~9를 순서대로 구현해 게이트 실행부를 라이브로 완성하고, 그 시점에 ADR-0009를 accepted로 승급해 Phase 6 강제 증분을 닫는다.
