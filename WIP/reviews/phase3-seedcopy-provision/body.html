Round 1은 신뢰·온보딩 설정을 이미지에 굽기 대신 seed+copy로 바꾸는 설계를 실측 검증했다. Round 2는 그 뒤 세 가지가 더해진 결과다 — (1) 두 모델에 "더 나은 해결안이 있나"를 자문받았고, (2) 그 사이 새로 잰 실측(R7)이 토큰 갱신 그림을 뒤집었으며, (3) 사용자가 "무중단 장기세션의 뜻"을 확정해 설계를 다시 5라운드(v6→v10) 수렴시켰다.

핵심 결론. 골격(seed+copy + 중앙 lease 갱신기 + accessToken만 컨테이너 지급)은 두 모델이 현재 증거에서 최선으로 확인했다(공식 문서 교차검증 포함). 바꿀 곳은 골격이 아니라 토큰을 컨테이너에 전달·교체하는 마지막 단계였다. v6→v10 재수렴으로 Codex-Sol·Fable 두 모델 모두 blocking 0에 도달했고, "검토용 초안으로 발의 대기 수렴"으로 판정했다. 값 확정·구현·Codex 확장은 아래 R1–R15 측정 게이트 뒤로 둔다.

이 페이지 구성. 새로 확정된 실측(R7 포함) → 자문이 바꾼 것 → seed+copy 기본 → 토큰 수명 아키텍처(이번의 핵심) → 자격증명 전달·감독자 배치 → ADR 골자 → 측정 게이트(R1–R15) → 수렴 여정 → 다음 단계.

Round 1의 세 항목(무프롬프트 도달·/tmp·임의 사용자)에 더해, 자문 직전 리눅스 호스트에서 실토큰으로 다섯 가지를 더 쟀다(값은 노출하지 않고 수정시각·크기·화면만).

R7a가 핵심이다 — 컨테이너 claude는 스스로 토큰을 갱신하지 못한다. accessToken을 서버가 거부하도록 훼손하고 유효한 refreshToken을 파일에 함께 둔 채 띄웠더니, 화면은 Login expired · Please run /login에서 멈췄고 자격증명 파일을 다시 쓰지 않았다(수정시각 불변). 헤드리스 /login도 실패. 두 번 독립 관측이 일치했다.

정직한 단서: 만료를 "토큰 훼손 + 만료시각 과거화"로 흉내낸 것이라(자연 8시간 경과 아님), "갱신을 시도조차 안 한다"와 "시도했으나 이 조건에서 실패한다"는 화면으로 구분되지 않는다. 그래서 결론은 "자가 갱신 불가"가 아니라 "미보장 — AXDT는 의존 금지"다. 어느 쪽이든 갱신은 중앙(호스트)에서 해야 한다는 설계 귀결은 같다.

값(수명 약 8h/29d 등)은 여전히 관측값이지 계약값이 아니다 — 설계 상수로 얼리지 않고, 아래 측정 게이트로 확정한다.

수렴한 설계와 R7 실측을 두 모델에 공유하고 "이 방향이 최선인가, 더 나은 아키텍처가 있나"를 물었다(수렴 리뷰가 아니라 대안 탐색). 두 모델의 답이 놀랍도록 일치했고, Codex는 리눅스 매뉴얼·도커 공식 문서로 교차검증까지 했다.

골격은 옳다 — 고칠 곳은 "마지막 홉"

중앙 갱신기 + 컨테이너엔 accessToken만 + seed+copy 전달이라는 골격은 유지가 맞다(refreshToken은 유출 시 피해가 큰 고가치 표적이므로 한 곳에만 두는 것이 최소권한 원칙에 맞다). 바꿀 것은 골격이 아니라 토큰을 컨테이너에 전달하고 교체하는 마지막 단계다.

두 모델이 짚은 결함 두 가지.

  • 단일 파일 읽기전용 마운트는 토큰 교체(회전)를 못 받는다. 파일 하나를 마운트하면 그 순간의 inode에 고정돼, 호스트가 새 토큰 파일로 바꿔치기해도 컨테이너 안에는 반영되지 않는다(매뉴얼·도커 문서로 확인). 실행 중 컨테이너에 새 토큰을 넣으려면 다른 방법이 필요하다.
  • 중앙 갱신기는 "토큰 발급기"가 아니라 "lease 관리자"여야 한다. 갱신이 옛 토큰을 무효화하는 정책이면(R8), 새 토큰을 발급하는 행위 자체가 아직 옛 토큰을 쓰는 컨테이너를 죽인다. 그래서 갱신기는 세대·만료를 추적하는 lease 관리자가 된다.

또한 두 모델은 새 OAuth 클라이언트를 직접 만들지 말라고 했다 — 호스트에 이미 로그인된 정품 claude의 갱신 경로를 구동해 토큰만 읽어 쓰는 편이 버전 변화에 안전하다. 헤더를 갈아끼우는 중계 프록시 같은 대안은 프롬프트 트래픽 노출·가용성 결합 때문에 비권장으로 정리됐다.

부팅 부분은 Round 1에서 확정된 그대로다. 신뢰·온보딩 설정을 담은 읽기전용 HOME provision seed를 마운트하고, entrypoint가 그것을 HOME으로 복사한 뒤 에이전트를 실행한다.

부팅 흐름

[읽기전용 설정 seed 마운트]  +  [자격증명 파일 마운트]
        -->  [entrypoint: HOME 생성 + 설정 복사 + 자격증명 600 설치]
        -->  exec 에이전트  (R9로 실측 통과)

이렇게 하면 온보딩 키 이름을 몰라도 되고(HOME 통째 복사), /tmp가 임시 파일시스템이어도 무해하며(런타임 복사라 이미지 레이어에 의존 안 함), 공유 쓰기 충돌이 사라지고(읽기전용 원본 + 컨테이너별 복사본), 임의 사용자 문제도 풀린다. 이 부팅 경로는 R9로 실측됐다(단 파일 마운트 방식 한정).

남은 진짜 문제는 토큰 수명이다. accessToken은 수명이 짧고(관측 약 8시간), 컨테이너는 스스로 갱신하지 못한다(R7a). 그런데 AXDT는 8시간을 넘는 세션이 필요하다(사용자 확정).

사용자가 정한 "무중단"의 뜻. "중단이 0"이 아니라 "세션의 작업·대화가 토큰 경계를 넘어 지속"이다 — 8시간마다 컨테이너 안 claude를 잠깐 재시작하고 대화 이어가기(--continue)로 잇는 것은 허용된다(진행 중이던 도구 호출은 유실될 수 있음). 이 정의가 설계를 크게 단순화한다.

그래서 두 층으로 나뉜다.

바닥(요구 충족) = 재시작 롤오버

재시작 롤오버: 만료 전에 push 전달로 실행 중 컨테이너에 새 토큰을 넣고, 감독자(supervisor)가 claude를 재시작한다. 새 claude는 시작할 때 새 토큰을 읽으므로, 실행 중 프로세스가 파일을 다시 읽는지(미측정)에 의존하지 않는다. 이 경로는 어떤 측정 결과에서도 살아남는 바닥이다.

최적화(요구 아님) = hot-handoff

hot-handoff: 재시작 없이 새 토큰을 집게 한다 — claude가 파일을 다시 읽거나(R14), 인증 헬퍼 훅이 주기적으로 재호출될 때(R14d). 실패해도 바닥(재시작)이 요구를 충족하므로 이건 blocker가 아니라 최적화다.

제어 채널 핸드셰이크. 재시작은 컨테이너 밖(호스트 갱신기)과 안(감독자) 두 프로세스가 번갈아 움직이므로 순서 경쟁을 막는 핸드셰이크를 둔다. 낡거나 실패한 토큰으로는 재시작하지 않는다(fail-closed).

두 개의 고위험 미측정.

(1) --continue 복원(R14c-복원)은 재시작 바닥의 공통 급소다 — 죽인 세션이 대화 이어가기로 정말 복원되는지가 감독자·재생성 두 형상 모두의 단일 미측정 가정이다. 실패하면 요구를 못 채우므로 가장 먼저·싸게 측정한다.

(2) R8(갱신이 옛 토큰을 무효화하나)은 AXDT 정상 토폴로지(한 계정을 여러 컨테이너가 공유)에서 아키텍처를 가른다. R8이 참이면 한 컨테이너의 갱신이 형제 컨테이너의 토큰을 끊는 동족 살해가 되고, 갱신 구동기가 사용자 호스트 로그인이면 사용자가 호스트에서 claude를 쓰는 것만으로도 컨테이너 토큰이 무효화된다. → 전용 계정 분리나 컨테이너별 격리 로그인이 필요할 수 있다. 무거운 구현 전에 측정한다.

토큰을 컨테이너에 넣는 경로가 부팅과 재시작 두 가지로 갈린다.

  • 부팅 전달 = 읽기전용 파일 마운트(R9로 실측) + 전용 그룹(GID)으로 임의 사용자 읽기 문제(R13)를 푼다.
  • 재시작 전달 = push(docker exec로 실행 중 컨테이너에 절대경로로 기입). 단일 파일 마운트가 inode 고정으로 못 하는 일을 이 경로가 맡는다.

감독자 계약(구현 의무). claude를 PID 1이 아니라 감독자의 자식으로 두면, 감독자가 (i) 좀비 회수·신호 전달, (ii) 만료 트리거 감지·재시작, (iii) 터미널·신호(Ctrl-C 등)·claude 종료코드 전달, (iv) 감독은 실 에이전트에만(테스트·placeholder 명령은 감싸지 않아 무한 재시작을 막음)을 책임진다.

공유 계약 변경(소비자 통보 필요). 감독자를 끼우면 claude가 죽거나 재시작 중이어도 컨테이너는 살아 있으므로, 세션 생존·종료코드를 읽는 backend.py의 계약(is_alive·exit_code·status)과 입출력(send_text는 재시작 창에서 버퍼링/거부, read_new_output은 재시작 흔적 경계 표시)의 의미가 바뀐다. 상위 계층에 통보하고 계약 테스트를 갱신해야 한다.

주 경로 확정. 감독자(A)를 기본으로 하고, A의 전달·보안 측정이 실패하면 컨테이너 재생성(B)을 게이트된 폴백으로 둔다(B는 backend 생존 계약을 안 바꾸는 대신 세대마다 컨테이너를 교체하고 대화 기록을 별도 볼륨에 남긴다). 자격증명 잔존은 HOME을 임시 파일시스템(tmpfs)으로 올려 완화한다(정지 시 회수 차단; 단 swap으로 샐 수 있어 별도 검토).

제목(안). 컨테이너 신뢰·자격증명 provision을 "이미지 굽기"에서 "seed+copy + 중앙 lease 갱신"으로 (Claude arm, 잠정). 상태: Proposed — R1–R15 종료 전 값 동결·Codex 확장 금지.

결정

  1. 굽지 않고 읽기전용 seed 마운트 → HOME 복사. 무프롬프트 도달은 파일 마운트 부팅 R9로 충족(push 부팅은 재측정 대기).
  2. 자격증명 분리, accessToken만 지급(refreshToken 미포함). 부팅=파일 마운트+전용 GID, 재시작=push. R2 실패 시 컨테이너별 격리 로그인 폴백(그때는 컨테이너 자기 갱신 허용 — 일반 규칙의 명시적 예외).
  3. 실행은 이미지 entrypoint, 실 에이전트는 감독자(claude≠PID 1) 기본·재생성은 폴백. 기본 명령 claude 굳히기는 이미지 베이스(R6)·Phase 5 이후.
  4. 토큰 수명 = 중앙 lease 갱신기(정품 claude 구동·원자적 저장). 요구 충족은 재시작 롤오버(R14c), hot-handoff는 최적화. R15(갱신 유발)는 공급 전제 게이트.
  5. HOME 경로 유지(위치는 파일시스템 독립) + 별도로 잔존 완화용 tmpfs.

결과(정직 표기)

  • 공유 계약 변경: run_args에 seed·자격증명 마운트+tmpfs+감독자 스크립트; backend.py 생존·입출력 계약 의미 변화. 소비자 통보·계약 테스트.
  • 비용: seed 생성·갱신·무결성 파이프라인 + 중앙 갱신기(단일 장애점·최고가치 표적). 격리 로그인 폴백 시 로그인 N회·refreshToken N개 관리.
  • 보안 표면: accessToken은 만료 전 탈취 유효, tmpfs swap 유출, 재생성 폴백의 대화 내용 잔존(자격증명 아님), 전용 그룹 구성원 경계. seed 무결성은 이미지 digest보다 약함 → 호스트 신뢰 경계 의존.

설계 메커니즘은 정했고 값은 측정 뒤로 둔다. 요구 충족을 직접 좌우하는 것(blocker)과 나머지를 구분한다.

측정 도구는 대상에 자동 합격/불합격을 찍지 않는다 — 증거만 붙잡고 판정은 사람이 한다. 실토큰·계정 식별자(PII)를 만지는 측정은 로그에서 값을 가린다.

Round 1은 초기 설계를 5라운드로 수렴시켰다. Round 2는 그 위에 대안 자문 + 재수렴 5라운드를 얹었다.

재수렴에서 잡힌 blocking(총 12건): 무중단 정의의 모호함 · R8=참일 때 hot-handoff 불성립 · ADR 정합 · PID 1과 재시작·tmpfs 충돌 · 재시작 바닥이 전달 채널에 은밀히 의존 · backend.py 생존 계약 누락 · 주 경로 문서 불일치 등. 모두 닫혔고, 두 모델은 "발의 대기 수렴"으로 판정했다. 골격은 자문 때부터 옳았고, 다섯 라운드는 토큰 수명·재시작이라는 새 요구가 연 표면을 정합시킨 과정이었다.

설계는 발의 가능 상태다. 진행은 측정이 앞서고, 가장 싸게 무효화 가능한 것부터 잰다.

  1. 측정(AX-DEV): R14c-복원(--continue) 최우선 → R2(accessToken만) → R14c-전달(push 핸드셰이크) → R15(갱신 유발) → R8(무거운 구현 전, 함대 토폴로지 요구-임계) → R1·R4 → 최적화 R14·R14d → 나머지. 로그는 값·PII 가림.
  2. 기록: 결과를 Phase 3 매트릭스에 CLI 버전과 함께.
  3. 구현(TDD): run_args 확장(seed·자격증명 마운트+tmpfs+감독자) + entrypoint/감독자 스크립트. 현행 테스트·placeholder 공존. backend.py 계약 갱신·소비자 통보.
  4. 중앙 lease 갱신기(정품 claude 구동·원자적 저장·핸드셰이크 호스트측) + seed 생성/갱신/무결성 도구.
  5. ADR 발의: Claude 한정 잠정 → sot/<slug> 사용자 게이트. 발의에 두 경고(공통 급소 --continue·함대 토폴로지 R8)를 싣는다.

사용자 결정이 필요한 지점. (1) 측정을 지금 진행할지, 발의를 먼저 할지의 순서. (2) ADR 번호·slug와 sot 브랜치를 여는 시점. (3) 설계 초안(현재 job 임시본)과 이 검토 소스(WIP/reviews/)를 phase3 브랜치에 커밋할지. 어느 것도 아직 하지 않았다 — 지시를 기다린다.
