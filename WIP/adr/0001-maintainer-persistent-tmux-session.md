# ADR-0001: Maintainer는 상시 tmux 세션으로 실행한다

## 상태
Accepted (2026-06-26) · 관련 결정 D1

## 맥락
Maintainer는 전체 진척 관리, Leader 생성·배치, tmux send-keys 주입, progress 단독 기록, 사용자 결정 지점에서의 메신저 발송을 맡는 오케스트레이터다. 이 역할을 **어떤 실행 형태**로 둘지 정해야 했다.

- 진행 중 오케스트레이션 맥락(어느 Leader가 무엇을 하고 무엇을 기다리는지)이 프로젝트 내내 이어져야 조율이 끊기지 않는다.
- 사용자 결정·블로커 알림은 **발생 즉시** 나가야 한다(지연되면 파이프라인이 멈춘다).
- 반면 장기 세션은 context가 무한히 커지는 문제를 안는다.

## 결정
Maintainer를 **프로젝트 시작~종료까지 살아있는 장기 tmux 세션**으로 실행한다. context 비대화는 Maintainer를 죽이는 대신 **Watcher(Cron 주기 호출)가 context를 압축·정리**하도록 분리해 해결한다.

## 결과
**좋은 점**
- 상주 프로세스라 **결정 지점에서 즉시** Leader 주입(통신 경로는 `ADR-0003`)·후속 조율을 이어간다(tick 대기·재기동 없음).
- 결정 지점에서 **즉시 메신저 발송** 가능 → 이벤트 큐 불필요(별도 DB/큐 없음, `ADR-0002`).
- 런타임 context가 살아 있어 매 사이클 상태를 파일에서 재구성할 필요가 없다(영속 권위는 progress·report 파일에 그대로 둔다).

**대가 / 주의**
- 장기 세션의 context 누적은 **상시 관리 대상** → Watcher로 위임(관심사 분리).
- 세션이 죽으면 진행이 멈춘다 → 영속 권위는 progress·report 파일에 남으므로 **재기동 후 복원 가능**. 단 사망 직전의 미승격 report·진행 중 판단의 복원 범위는 별도 복원 절차가 정한다(Phase 4, `ADR-0004`).

## 검토한 대안
### 대안 A — 무상태 + Cron 주기 호출 Maintainer
Maintainer 자체를 Cron으로 매번 새로 깨워 progress·report에서 상태를 재수화하는 stateless 방식(안정된 tmux target만 있으면 send-keys 자체는 보내는 쪽 소유 여부와 무관하게 가능). · **기각 사유**: ① **알림이 다음 tick까지 지연**돼 사용자 결정·블로커가 즉시 못 나가고 파이프라인이 멈춘다. ② 매 호출마다 **전체 오케스트레이션 context를 파일에서 재구성**해야 해 비용·복잡도가 커진다. ③ 진행 중 판단·미승격 report 등 live 맥락이 호출 사이에 끊겨 연속 조율이 어렵다.

### 대안 B — 상주 데몬 + 외부 이벤트 큐/DB
상주시키되 상태·이벤트를 별도 큐/DB에 두는 방식. · **기각 사유**: 영속 상태는 이미 interim 파일에 있다. 큐/DB는 **중복 진실원**을 만들고 운영 부담을 늘린다(별도 저장소 불채택은 `ADR-0002`로 분리 기록).
