# ADR-0008: 테스트 설계는 SoT 4번째 타입으로 요구·사양과 동급으로 다룬다

## 상태
Accepted (2026-07-07) · 관련 결정 D16

## 맥락
SoT 완료 판정(`rule-sot-readiness`)은 지금 requirements·specification만 대상으로 한다. 그러나 개발 착수는 "무엇을 만들지"뿐 아니라 "무엇을 어떻게 검증할지"가 정해져야 품질이 성립한다 — V-모델에서 요구·사양의 각 층에는 대응하는 검증 설계가 있다. 검증 설계가 SoT 밖에 있으면 완료 판정이 그 부분을 보지 못해, 검증 근거가 빈 채로 개발이 출발하고 완료로 간주된다.

테스트 설계 문서는 테스트를 수행하는 코드가 아니라, 요구·사양에서 도출한 테스트 조건·커버리지 목표·스위트 구조·추적성·절차 원칙을 담는 문서다(ISTQB 테스트 분석·설계 기준). 목적이 요구·사양과 달라 별도 타입으로 둔다.

## 결정
`docs/sot/test-design/`를 requirements·specification과 **동급인 4번째 SoT 타입**으로 둔다. 요구·사양과 마찬가지로 Agent가 작성하고 변경은 사용자 게이트 PR로만 한다(`rule-sot-change-user-gate`).

설계와 구현을 나눈다.

- **테스트 설계(SoT)** 는 코드·실행 없이 요구·사양에서 도출되는 부분만 담는다 — 테스트 조건, 블랙박스 커버리지 목표(동등분할·경계값·결정표·상태전이), 스위트 구조, 요구·사양↔테스트 조건 추적성, 절차 원칙(검증 방법·관찰점·합격 판정의 원칙).
- **테스트 구현(`test/` 코드, Tester 소관, D12)** 은 구체 케이스·데이터·픽스처·실행 스크립트·환경을 담는다.

완료 판정 대상은 테스트 설계(SoT)까지다. 요구·사양이 완료로 판정되려면 대응하는 테스트 설계가 함께 완료되어야 한다.

## 결과
**좋은 점**
- 요구↔테스트가 대칭을 이룬다 — requirements→`src/`, test-design→`test/`, 그 사이를 specification이 잇는다. 완료가 "검증 설계까지 끝난 상태"를 보장한다.
- 검증 설계가 SoT라 변경통제·추적을 받는다. 코드 산출물 품질의 근거가 문서로 고정된다.

**대가 / 주의**
- 전파 무게: 테스트 설계는 베이스 규칙을 따르므로 모든 신규 프로젝트가 최소 하나의 테스트 설계 SoT를 갖춰야 완료가 성립한다. opt-out은 두지 않는다 — V-모델상 검증 설계를 선택 사항으로 두면 완료의 의미가 흔들린다.
- 마이그레이션: 이 규칙이 활성화되면(강제 = Phase 6, 또는 뒤늦은 채택) 그 이전에 요구·사양만으로 완료됐던 문서는 테스트 설계가 없어 fail-closed(미완료)가 된다. 전량 스윕 또는 전환 계획이 필요하다.
- PR 범위 확대: 요구·사양 변경 PR이 대응 테스트 설계까지 함께 다루게 된다.
- 규칙 동시 갱신 필수: 타입을 등록하면서 완료 규칙(`rule-sot-readiness`)·검토 스킬(`sot-readiness-review`)을 같은 패스에서 갱신하지 않으면 완료 트리가 불일치한다.

**못박은 설계 제약**
1. 화이트박스 커버리지(결정/분기/MC-DC)는 테스트 설계 SoT에서 제외한다. 이 커버리지는 코드가 있어야 측정·명세되므로, SoT에 넣으면 개발 전 완료가 영구히 미충족이 되어 게이트가 영구 차단된다. 화이트박스는 구현 계층에서 측정하며 완료 판정 대상이 아니다.
2. 조건(SoT)과 케이스(코드)의 절단선을 템플릿과 규칙에 명시한다. 요구→코드는 종류가 다르지만, 조건→케이스는 정도의 차이(추상↔구체)라 경계가 연속적이다. 명시하지 않으면 무엇까지 SoT인지가 흔들려 완료 판정이 불안정해진다. 절단 기준: "코드·실행 없이 문서만으로 확정되고 요구·사양에서 바로 도출되면 SoT, 코드·데이터·실행 환경이 있어야 확정되면 `test/`."

## 검토한 대안
### 대안 A — interim으로 둔다
테스트 설계를 `docs/interim/`의 가변 산출물로 둔다. · **기각**: interim은 완료 판정 대상이 아니라 검증 설계가 완료에서 빠진다. 변경통제도 없어, 요구·사양이 바뀌어도 테스트 설계와의 정합을 게이트가 강제하지 못한다.

### 대안 B — requirements 문서 안의 한 절로 둔다
요구사항 문서에 테스트 절을 추가한다. · **기각**: 요구(무엇이 왜)와 테스트 설계(어떻게 검증)는 목적이 다르다. 한 문서에 섞으면 문서가 비대해지고, 요구↔테스트의 다대다 추적이 한 파일 안에서 꼬인다.
