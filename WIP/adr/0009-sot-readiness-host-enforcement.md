# ADR-0009: SoT 완료 강제(①②③)는 머지 컨트롤러가 유일 갱신 권한자로서 머지 직전에 판정해 실현한다

## 상태
Proposed (2026-07-08, 2026-07-09·2026-07-13 개정) · 관련 결정 D6

이 증분은 `ADR-0007`(층 강제, proposed) 토대 위에 얹힌다 — 허브 콘텐츠·경로 게이트 CODE가 아직 후속이라 `ADR-0007`도 proposed다. 본 ADR도 강제 CODE가 구현될 때까지 `proposed`로 둔다. GitHub 전용이며, GitLab/Forgejo 강제는 별도 멀티호스트 Phase, 라이브 검증은 Phase 9다. 이 증분이 Phase 6를 닫는다(= (b) 클라이언트 증분 ∧ 강제 증분, `ADR-0010`).

## 맥락
`rule-sot-readiness`는 SoT 완료를 **① 형식 ∧ ② 검토 ∧ ③ 승인**의 논리곱으로 정의하고, 완료를 실제로 강제하는 것은 규칙문이 아니라 머지 게이트라고 명시한다. 정책·신뢰 모델은 규칙이 고정했고, 남은 것은 **그 강제를 호스트에서 거는 메커니즘**이다.

네 가지 제약이 메커니즘을 좁힌다.

- **무인증 허브로는 불가**(`ADR-0006`). 허브는 PR·승인·머지결과 상태가 없어 ①②③ 결속을 표현할 능력이 없다.
- **네이티브 필수검사는 통과/실패만 표현한다.** "각 open blocking이 accepted/rejected로 닫혔는가"는 필수검사 하나로 표현되지 않는다. 사용자 결정을 검사 산출물 위에 얹어 계산하는 조각이 필요하다.
- **사용자 결정은 파일이 아니라 호스트 채널에 결속된다**(규칙). 위조 가능한 파일에 완료를 여는 권한을 두지 않는다.
- **이식성.** AXDT는 GitHub·GitLab·Forgejo를 지원한다(`ADR-0010`). 강제 메커니즘이 한 호스트의 상위 요금제 기능에 묶이면 나머지 두 호스트에서 재설계해야 한다.

머지 게이트를 실현하는 길은 둘이다.

- **(가) 필수 검사 오버레이** — 게이트가 판정을 검사 상태로 게시하고 브랜치 보호가 그 검사를 요구한다.
- **(나) 머지 컨트롤러** — `main`을 갱신할 권한을 지정 신원 하나로 제한하고, 그 신원이 머지 직전에 판정을 계산해 통과할 때만 머지한다.

(가)는 GitHub에서 위조 내성을 갖추려면 전용 App(check run은 App만 생성 가능)과 조직 룰셋의 required workflow(조직 설정·Enterprise 전용)에 의존한다. GitLab의 external status check는 Ultimate 티어, Forgejo는 상태검사가 GitHub에 근접하나 표현력이 다르다. 반면 (나)가 쓰는 "누가 브랜치를 갱신할 수 있는가"는 세 호스트에 모두 있는 공통 기능이다.

## 결정
**(나) 머지 컨트롤러**로 강제한다. 강제 계산은 호스트 무관 순수 코어에, 호스트에서 읽고 머지하는 일은 주입 포트에 둔다((b)의 어댑터+백엔드 분리 계승).

1. **`main` 갱신 권한을 머지 컨트롤러 신원 하나로 제한한다.** 사람·CI·다른 봇은 `main`을 갱신하지 못한다. 컨트롤러는 대상 저장소 **밖**에 살고, 머지 토큰은 컨트롤러에만 두며 저장소·브랜치·동작 단위 최소 권한으로 발급한다. 일반 CI 시크릿에 넣지 않는다 — 같은 저장소 브랜치 PR에는 시크릿이 전달되므로 저장소 안에 두면 PR이 토큰에 닿는다.

2. **브랜치 규칙을 두 벌로 분리한다(불변식).** 규칙 묶음 A는 "지정 신원만 `main` 갱신"만 담고 컨트롤러를 우회 신원으로 지정한다. 규칙 묶음 B는 네이티브 필수 승인·CODEOWNERS·허용 머지 방식을 담고 **우회 신원을 비운다**. GitHub 룰셋에서 우회(bypass)는 **룰셋 단위**임이 실증됐다 — 둘을 한 룰셋에 합치면 컨트롤러가 승인 요구까지 함께 건너뛴다. 분리해야 컨트롤러가 유일 머지자이면서도 승인 관문에 걸린다.

3. **머지 게이트 코어(순수).** 주입된 입력(② 검토 CI 산출물 · 호스트 채널의 사용자 결정 · 승인 이벤트 · PR 메타데이터)으로 `①형식 ∧ ②성립 ∧ ③승인`을 **같은 판정 키**에서 계산한다. 부작용이 없어 라이브 호스트 없이 결정적으로 테스트된다. 쓰기 포트는 검사 상태 게시가 아니라 **머지 실행**이다.

4. **머지 전역 직렬화 + 머지 직전 재평가.** 컨트롤러는 한 번에 하나의 머지만 수행하고, 머지 직전에 판정을 처음부터 다시 계산한다. 이로써 "착지하는 판정 키에 신선한 평가 없이 머지되는 경로는 없다"는 불변식을 얻는다. 결정 취소·코멘트 삭제·권한 변경·base 전진을 이벤트로 좇아 이전 판정을 무효화하는 방식에 기대지 않는다 — 이벤트 연쇄는 머지 전 사람이 보는 표시를 갱신하는 용도로만 쓴다.

5. **호스트 채널 = 구조화 PR 코멘트(append-only).** 결정권자가 완전 결속 키를 참조한 기계판독 코멘트로 finding별 accepted/rejected를 남긴다. **결정권 = 저장소 permission `admin` ∧ 지정 명단 등재 ∧ 사람 계정.** 명단은 게이트가 읽는 입력이며, `.github/CODEOWNERS`는 경로별 승인을 강제하는 **추가 관문**이지 명단의 대체물이 아니다. 판별은 `role_name`으로 한다(레거시 `permission` 필드는 maintain을 write로 뭉갠다). 권한은 **평가 시점의 현재 값**으로 판정한다 — 강등은 즉시 반영되고, 승격은 소급한다. 코멘트는 편집·삭제하지 않고 새 코멘트로만 번복한다.

6. **완료 강제는 SoT를 바꾸는 PR에만 적용한다 — 단 강제 장치 자체는 별도 관문을 둔다.** 컨트롤러가 저장소의 모든 머지를 수행하므로, 제안된 머지 결과가 `docs/sot/{requirements,specification,test-design}`를 바꾸지 않는 PR은 ①②③ 대상이 아니다. 그러나 **강제 장치 자체를 바꾸는 PR**(규칙 지문 원천·검토 CI·CODEOWNERS·게이트 코드 = "강제-필수 경로")은 ①②③ 대상은 아니되 그대로 머지하지 않고 **포크 거부 + 결정권자 승인**을 요구한다(sub-spec §2.6 세 번째 분기). 두 경로 어디에도 없는 PR만 그대로 머지한다. 판단은 머지 결과의 변경분으로 하고 브랜치 이름·커밋 메시지로 하지 않는다.

7. **PR의 head 저장소가 대상 저장소와 다르면 거부한다.** 브랜치 이름 규약(`^sot/[a-z0-9]+(?:-[a-z0-9]+)*$`)만으로는 포크가 흉내 낼 수 있다.

8. **감사 이력 보존은 호스트가 강제한다.** 우회 불가 규칙 묶음 B가 허용 머지 방식을 merge commit으로 제한하고, `main`의 force-push·브랜치 삭제를 차단한다. 컨트롤러가 올바른 방식을 고르리라는 가정에 감사 이력을 얹지 않는다. "선형 히스토리 요구"는 머지 커밋을 금지하므로 켜지 않는다. **`sot/*` 소스 브랜치 자체의 force-push 차단은 요구하지 않는다** — merge commit이 승인 head를 `main` 이력에 부모로 보존하고, 승인~머지 사이 창에서의 head 교체는 dismiss-stale과 컨트롤러의 head SHA 고정(sub-spec §2.5)이 막는다. (초기 규칙 문언은 squash 방지 근거에 `sot/*`를 함께 실었으나, squash 강제의 주체는 `main` 룰셋의 `allowed_merge_methods`임이 실증돼 그리로 이관됐다.)

9. **컨트롤러 도메인의 불변 감사 기록.** 모든 머지의 `merged_by`가 컨트롤러이므로 호스트 이력만으로는 누가 무엇을 승인했는지 남지 않는다. 컨트롤러는 머지마다 판정 키·결정 스냅샷·승인 이벤트를 자기 도메인에 추가 전용으로 기록한다.

10. **초기 마이그레이션 = 축3 1회 스윕 + fail-closed.** 활성화 시 기존 완료 문서 전량을 교차 정합성(축3)만으로 스윕해 `rules` 선언 완전성을 확립한다(규칙의 재검토 트리거). 판정 키에 대한 산출물이 없으면 미완료 처리한다. requirements+specification만으로 완료됐던 문서는 test-design(`ADR-0008`)이 빠져 fail-closed로 떨어지며, 유예 없이 백필한다. 마이그레이션 baseline finding의 결정도 파일이 아니라 전용 마이그레이션 PR의 구조화 코멘트로 닫는다.

11. **범위 = GitHub 전용.** GitLab/Forgejo 강제는 별도 멀티호스트 Phase로 분리한다. 3-호스트 강제 매트릭스는 그 Phase를 위한 선행 스케치로 sub-spec에 첨부한다.

12. **전제조건 = 브랜치 규칙을 걸 수 있는 저장소 구성.** 개인 소유 private 저장소 + 무료 플랜에서는 룰셋도 구식 브랜치 보호도 거부된다(실증: `403`). AXDT 저장소를 공개로 전환해 이 전제를 충족했다(2026-07-09). private 유지가 필요해지면 GitHub Pro 이상이 필요하다.

## 실증 (2026-07-09, GitHub 개인 저장소)
임시 저장소에 룰셋을 걸고 PR을 머지해 확인했다. 아래는 관찰이며, sub-spec §4.1의 설정이 여기서 나온다.

| 확인 | 결과 |
|---|---|
| 개인 private 저장소 + 무료 플랜의 룰셋·브랜치 보호 | 둘 다 `403`. 공개 전환 또는 Pro 필요 |
| 룰셋이 저장소 관리자에게도 적용되는가 | 적용됨 — 우회 목록에 없으면 소유자의 머지도 거부 |
| 우회 신원에 개별 계정 지정 | 가능(`actor_type: "User"`). 전용 App 불필요 |
| 우회의 범위 | **룰셋 단위** — 한 룰셋에 승인 요구를 같이 담으면 그것까지 우회(승인 0으로 머지됨) |
| 룰셋 분리 시 컨트롤러 머지 | 승인 부족으로 `405` 거부 — 승인 관문이 독립 유지됨 |
| merge queue | 개인 저장소에서 사용 불가(룰 생성 `422`) |
| 허용 머지 방식 | 룰셋의 `allowed_merge_methods`가 강제. 저장소 전역 설정이 squash를 허용해도 룰셋이 막는다 |

## 결과
**좋은 점**
- 강제의 신뢰 뿌리가 저장소 밖에 있다. 커밋 상태 위조·워크플로 자체수정·App 정체성 같은 문제군이 발생하지 않는다.
- "누가 갱신할 수 있는가"만 쓰므로 GitLab·Forgejo로 옮길 때 재설계가 아니라 설정 번역이다.
- 머지 직전 재평가 + 전역 직렬화가 낡은 판정과 base 전진 경합을 구조적으로 닫는다.
- 강제 계산이 순수 코어라 라이브 호스트 없이 결정적으로 테스트된다.
- 규칙 묶음 B가 우회 불가라, 컨트롤러가 침해되거나 버그가 나도 네이티브 승인·머지 방식 제한이 남는다.

**대가 / 주의**
- **피해 반경.** 컨트롤러 토큰이 유출되면 승인 관문은 남으나 머지 실행은 공격자 손에 든다. 토큰을 저장소 밖에 두고 최소 권한으로 좁히는 것이 유일한 방어다.
- **가용성.** 컨트롤러가 멈추면 저장소의 모든 머지가 멈춘다. fail-closed의 귀결로 수용한다.
- **감사 주체 소실.** `merged_by`가 항상 컨트롤러다. 결정 9의 불변 기록이 이를 보상한다.
- **비-SoT PR까지 컨트롤러를 지난다.** 결정 6의 분류가 틀리면 무관한 PR이 막히거나 SoT PR이 검사를 건너뛴다. 분류 근거는 머지 결과 변경분 하나뿐이다.
- **규칙 묶음 분리가 조용한 실패점이다.** 둘을 합치면 경고 없이 승인 관문이 사라진다. 룰셋은 admin이 PR 없이 바꿀 수 있어 PR 게이트 밖의 붕괴 지점이므로, 컨트롤러가 기동 시와 매 머지의 직렬화 잠금 안에서 라이브 룰셋 구성을 선언 상태와 대조해 불일치면 fail-closed한다(sub-spec §4.1).
- **② 산출물의 신뢰는 저장 위치의 쓰기 통제에 걸린다.** 판정 키는 공개 콘텐츠(제안된 머지 결과)의 결정적 함수라 위조 방지가 아니다 — 작성자는 자기 PR의 판정 키를 계산할 수 있다. 신뢰된 ② CI 신원만 산출물을 쓸 수 있어야 하며(또는 CI 신원 서명), 판정 키 대조는 재전송 방지용이다. 산출물 저장소의 쓰기 신뢰 모델은 하중을 받는 보안 요소다(sub-spec §4.2·§8).
- **승격이 소급한다.** 권한을 평가 시점의 현재 값으로만 판정하므로, 명단에 없던 계정이 미리 남겨 둔 표시가 등재와 동시에 유효해진다. 명단 등재 자체가 결정권자의 행위이고, 등재는 그 계정의 기존 표시를 인정하는 것으로 보아 의도된 동작으로 둔다. 표시 시점 권한을 함께 요구하는 방식은 검토 후 채택하지 않았다.
- 포트의 GitHub 구현·CI 워크플로는 provisional이라 Phase 9 라이브 검증 전까지 위험을 안는다.
- 강제 CODE가 아직 없어 status는 proposed다(`ADR-0007`과 동일).

## 검토한 대안
### 대안 A — 사용자 결정을 감사 파일에 기록
결정을 저장소 감사 경로에 커밋하고 게이트가 그 파일을 읽는다. · **기각**: 규칙이 명시적으로 금한다 — 위조 가능한 파일에 완료를 여는 권한을 둘 수 없다.

### 대안 B — 결정권을 저장소 `write`(또는 `maintain`)까지 허용
· **기각**: `write`는 PR을 여는 봇/머신 계정이 대개 가져 자기 PR의 blocking을 스스로 닫을 수 있다. `maintain`도 결정권자 집합을 넓히고 호스트 간 경계가 흐릿하며, 레거시 permission API가 maintain을 write로 뭉갠다.

### 대안 C — 필수 검사 오버레이 ((가) 방식)
게이트가 `sot-merge-gate` check run을 게시하고 브랜치 보호가 그것을 필수 검사로 요구한다. 다중 리뷰(Codex·Fable)가 권고한 안이다. · **기각**: 이식성. 위조 내성을 갖추려면 GitHub에서 전용 App과 조직 룰셋 required workflow가 필요하고(check run은 App만 생성 가능, required workflow는 조직·Enterprise 전용), GitLab의 external status check는 Ultimate 티어다. 세 호스트 공통 원시 기능은 "누가 갱신할 수 있는가"뿐이므로 그것을 강제의 토대로 삼는다. (가)의 이점(호스트가 게이트를 대신 강제, 컨트롤러 가용성 불필요)은 인정하나 이식성을 우선한다.

### 대안 D — GitLab/Forgejo를 이번 증분에서 함께 강제
· **기각**: 강제 표현력이 호스트별로 다르고 Phase 9(GitHub 라이브 도그푸딩)와 맥락이 맞지 않는다. 매트릭스만 선행 스케치로 남긴다.

### 대안 E — 마이그레이션 시 기존 완료를 재검토 없이 통과
· **기각**: test-design 공백을 유예하면 fail-closed를 우회해 미검증 완료가 남는다.

### 대안 F — 규칙을 한 묶음으로 두고 컨트롤러를 우회 신원으로 지정
설정이 단순하다. · **기각**: 실증 결과 우회는 룰셋 단위다. 컨트롤러가 승인 요구까지 건너뛰어 ③이 사라진다. 분리는 선택이 아니라 정확성 조건이다.

### 대안 G — merge queue로 머지를 직렬화
호스트가 큐를 관리하고 머지한다. · **기각**: 개인 소유 저장소에서 merge queue 룰이 거부된다(실증). 또한 큐가 머지를 수행하면 "컨트롤러가 유일 머지자"가 깨져 머지 직전 재평가 지점이 사라진다. 직렬화는 컨트롤러가 담당한다.
