# ADR-0002: 별도 DB·큐·이벤트 로그를 두지 않고 interim 파일로 상태를 보유한다

## 상태
Accepted (2026-06-26) · 관련 결정: 상태 저장 정책(TODO "상태 저장 — 별도 DB/큐 없음" 섹션)

## 맥락
시스템은 진척·보고·알림 상태를 어딘가 보존해야 한다. 흔한 선택은 DB·메시지큐·이벤트 로그다. 그러나 AXDT에는 이미 상태를 담는 매체가 있다 — interim 파일(plan·report·progress). 또한 Maintainer는 상시 세션이라(`ADR-0001`) 알림을 큐에 쌓지 않고 즉시 보낼 수 있다.

## 결정
새로운 DB·큐·이벤트 로그를 도입하지 않는다. 영속 상태는 **interim 파일**에 둔다. 메신저 outbound 알림은 상시 Maintainer가 **결정 지점에서 직접 발송**하고, inbound는 **stateless webhook 브릿지**가 받아 tmux send-keys로 Maintainer에 전달한다. Web 브리핑은 파일을 **read-time 렌더**만 한다.

## 결과
**좋은 점**
- 상태 진실원이 **파일 한 곳**(권위는 progress)으로 유지돼 중복·동기화 문제가 없다.
- 운영 구성요소가 줄어 셋업·장애면이 단순해진다.
- progress의 git 이력(D10 마일스톤 커밋)이 사실상 변경 이력 역할을 한다.

**대가 / 주의**
- 동시 쓰기 통제는 파일 규범(`rule-progress-single-writer` 등)이 책임진다.
- 크래시·재시작 후 상태 복원은 progress·report를 다시 읽어 재구성해야 한다(별도 복원 절차 — Phase 4, `ADR-0004`).
- 대규모 이벤트 이력·고빈도 조회가 필요해지면 재검토 대상(현재 범위 밖).

## 검토한 대안
### 대안 A — 이벤트 큐 / 메시지 브로커
알림·작업을 큐로 중계. · **기각 사유**: 상시 Maintainer가 즉시 발송 가능해 큐가 불필요하다(`ADR-0001`). 인프라와 상태가 이원화돼 복잡도만 늘어난다.

### 대안 B — 상태 DB
진척·상태를 DB에 저장. · **기각 사유**: interim 파일과 **중복 진실원**. interim 파일은 이미 사람이 읽고 git으로 이력화되는 **운영상 단일 진실원**이다(권위본=SoT와는 구분; 시스템 권위는 progress).

### 대안 C — append-only 이벤트 로그
모든 상태 전이를 로그로 적재. · **기각 사유**: progress의 git 이력이 이미 전이 이력을 보존한다. 별도 로그는 중복.
