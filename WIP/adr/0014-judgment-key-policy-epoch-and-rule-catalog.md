# ADR-0014: 완료 판정을 두 키로 결속하고(정합성 판정 키 + 선언 완전성 스윕 키), 판정 키에 검토 정책 epoch·규칙 카탈로그 digest를 추가한다

## 상태
Accepted (2026-07-13) · 관련 결정 D28

## 맥락
완료 판정 키(judgment key)는 ②(정합성·공백 검토)와 ③(사용자 승인)의 결과를 내용에 결속한다 — 키는 SoT 트리 해시와 적용 규칙 지문으로 이뤄지고, 내용이 바뀌면 키가 바뀌어 기존 판정이 무효가 된다(fail-closed). **무효화·재검토는 이 단일 키 단위**이며, 키가 하나라도 바뀌면 projection 전량을 홀리스틱 재검토한다(부분 문서 타겟팅 없음 — `rule-sot-readiness` ②). 규칙 frontmatter `scope`는 '어느 문서를 다시 볼지'가 아니라 **어느 규칙 변경이 적용 규칙 지문(성분(2))에 들어가 무효화를 유발하는가**를 정한다: `global`은 항상, `local`은 그 규칙을 `rules`에 선언한 문서가 있을 때.

phase1 다중모델 리뷰(Codex-Sol·Fable)에서 이 결속에 두 공백이 드러났다.

1. **완료 판정 규칙 자신의 변경이 무효화를 촉발하지 않는다.** `scope`는 어느 규칙 변경이 적용 규칙 지문(성분(2))에 들어가는지만 제어한다. `rule-sot-readiness`(완료 판정 절차)나 `rule-sot-change-user-gate`(게이트 절차)는 대상 SoT 콘텐츠를 제약하는 콘텐츠 규범이 아니라 거버넌스라 완료 문서가 이들을 `rules`에 선언할 일이 없다 — 그래서 이들이 바뀌어도 적용 규칙 지문이 안 움직여 기존 완료가 그대로 유효한 것으로 남는다. 판정 기준이 바뀌었는데 옛 판정이 살아 있는 상태다.
2. **규칙 목록의 변화가 감지되지 않는다.** `scope: local` 규칙을 새로 추가하면, 아직 어떤 문서도 그 규칙을 `rules`에 선언하지 않아 적용 규칙 지문이 안 바뀐다. 규칙 삭제·이동·`scope` 전환·활성 전이(`status`)도 마찬가지로 키에 반영되지 않는다.

`scope`를 `global`로 돌려 1번을 막으려는 접근은 어긋난다(ADR-0015 참조): 완료 판정 규칙은 대상 SoT 콘텐츠를 횡단 제약하는 콘텐츠 규범이 아니라 거버넌스라 `scope` 축과 성격이 맞지 않고, 무엇보다 `global`은 규칙 **파일 본문**만 지문에 넣을 뿐 검토 스킬·모델·렌더 프롬프트·추론 설정 같은 **실행 정책 전체**를 결속하지 못한다. 정책 결속을 `scope`가 아니라 키에 명시 성분으로 넣어야 한다.

## 결정
**정합성 판정 키**에 두 성분을 추가하고, **선언 완전성**을 위한 **별도 키**를 둔다(무효화·재검토는 각 키 단위, 키 변경 시 projection 전량 홀리스틱).

1. **`review_policy_epoch`**(정합성 판정 키 성분; 완전성 스윕 키와 공유) — 검토 정책·실행기 버전. **(저장소 파생) 완료 판정 규칙(`rule-sot-readiness`·`rule-sot-change-user-gate`)·검토 스킬의 내용 해시, (실행 환경 파생) 검토 실행 모델 revision + 신뢰 실행기 버전 digest(CI 하니스·직렬화기·응답 파서·키 계산기·프롬프트 스캐폴드; 정본 열거는 `rule-protected-paths` 실행기 manifest) + 결과에 영향을 주는 추론 설정 digest**를 포함한다. 스킬 해시만으로는 호스트 조립 프롬프트·추론 설정·base 실행기 코드를 결속하지 못하므로 이들을 함께 넣어 "키가 가리키는 정책·실행기 = 실제 실행된 것"을 보장한다. 프롬프트 digest는 **정책 불변부만**(실행별 가변 데이터 = 대상 SoT·rule 입력·키 값·ref·시각은 제외 — 안 그러면 카탈로그 본문이 epoch로 판정 키에 새어들거나 프롬프트가 실행마다 달라져 재사용이 죽는다). 판정 절차·게이트 절차·검토 스킬·검토 주체·실행기가 바뀌면 epoch가 바뀌어 기존 완료가 전량 무효가 된다.
2. **규칙 카탈로그 manifest digest**(정합성 판정 키 성분) — 규칙 카탈로그의 `{path, id, scope, status}` 집합에 대한 다이제스트(카탈로그 대상 = `docs/sot/rule/`의 규칙 파일; `README`·`_TEMPLATE` 제외). 규칙의 **추가·삭제·이동(`path`)·`id` 변경·`scope` 전환·`status` 변경**을 키가 감지해, `scope: local` 규칙 신설로 정합성 재검토 대상이 0건이 되는 공백과 활성 전이(`deprecated`·`superseded`로 적용 집합이 바뀜)를 닫는다.
3. **완전성 스윕 키**(별도 키) = (projection 트리 해시, **활성 rule 카탈로그 입력 digest**, `review_policy_epoch`). 카탈로그 입력 digest = 완전성 검사에 실제 직렬화돼 들어가는 바이트의 digest(정규화 repo-relative `path`로 정렬한 `{path, frontmatter 포함 전체 내용}` 레코드의 canonical serialization; 활성 rule만, README·`_TEMPLATE` 제외 — `path` 포함이라 rename도 결속). 결과에 영향 주면 안 되는 메타데이터(PR·ref·시각)는 LLM 컨텍스트에서 제외한다. **선언 완전성 검사**(문서가 실제 의존하는 rule을 `rules`에 빠짐없이 선언했는가)만 결속한다. 이 검사는 활성 rule 본문 전량을 봐야 미선언 의존을 탐지하는데, 그 본문을 정합성 판정 키에 넣으면 미선언 rule(대개 오케스트레이션) 편집마다 정합성이 헛발화해 ADR-0015와 어긋난다 — 그래서 **입력이 다른 이 검사를 별도 키에 결속**해, 미선언 rule 본문 편집이 정합성은 안 건드리고 완전성 검사만 재실행시키게 한다(같은 키가 다른 실제 입력을 대표하는 캐시 우회 차단). 라운드4 다중모델 리뷰에서 "완전성 입력이 판정 키 밖이면 캐시 우회로 완료를 잘못 열 수 있다"는 blocking 지적을 이 별도 키로 닫았다.

정합성 판정 키 성분 1·2는 기존 트리 해시·적용 규칙 지문과 함께 판정 키를, 성분 3은 완전성 스윕 키를 구성하며, `rule-sot-readiness`의 키 정의에 명문화한다.

## 결과
**좋은 점**
- 완료 판정 규칙·게이트 절차·검토 주체·실행기가 바뀌면 옛 완료가 자동으로 무효가 된다 — 판정 기준과 판정 결과의 결속이 fail-closed로 유지된다.
- 규칙 목록의 구조 변화(추가·삭제·이동·id·scope·status)가 키에 잡혀, `scope: local` 규칙을 안전하게 신설할 수 있다(ADR-0015의 "전부 local" 결정을 뒷받침).
- 선언 완전성 검사가 활성 rule 본문 전량을 **별도 키(완전성 스윕 키)**에 결속해, 미선언 의존을 실제 본문으로 탐지하면서도 그 편집이 정합성 재검토를 헛발화시키지 않는다 — 정합성 헛발화 방지(ADR-0015)와 완전성 입력 소여 결속을 두 키로 분리해 둘 다 얻는다.

**대가 / 주의**
- `review_policy_epoch`가 바뀌면 완료 문서 전량이 ②(정합성·완전성 모두) 재검토 대상이 된다 — 완료 판정 규칙·리뷰 스킬/모델/실행기를 손대면 재검토 비용이 큰 것이 정상이며, 그 비용이 곧 fail-closed의 값이다.
- epoch에 무엇을 넣는가(규칙·스킬 해시·모델 revision·실행기 버전·정책 불변 프롬프트·설정)와 프롬프트 digest의 경계(가변 데이터 제외)를 규칙에 정확히 못박아야 한다. 범위가 모호하면 무효화가 과하거나 모자란다. 카탈로그 manifest digest는 `scope`·`status`까지 포함해 scope 전환·활성 전이도 잡는다.
- 두 검토(정합성=판정 키, 선언 완전성=완전성 스윕 키)는 **비필수 증거 검사**이고, 머지를 막는 **필수 검사는 집계 게이트(`sot-readiness-gate`) 하나**뿐이다(그것이 **① 통과·두 검토 증거·사용자 override·③ 승인·현재 두 키 일치**를 모아 성공/실패 계산 — rule 강제 매핑과 동일 열거). 검사·키가 하나 늘지만, 단일 키에 카탈로그 본문을 욱여넣어 헛발화를 부르거나 완전성 입력을 키 밖에 둬 캐시 우회를 남기거나, 증거 검사를 필수로 걸어 override를 죽이는 것보다 건전하다.

## 검토한 대안
### 대안 A — 완료 판정 규칙을 `scope: global`로 두어 무효화 유도
`rule-sot-readiness`·`rule-sot-change-user-gate`를 global로 선언해 변경 시 전량 재검토. · **기각**: 완료 판정 규칙은 대상 SoT 콘텐츠를 제약하는 콘텐츠 규범이 아니라 거버넌스·완료정책이라 `scope` 축과 성격이 맞지 않는다(ADR-0015). 또한 `global`은 규칙 **파일 본문**만 지문에 넣을 뿐 검토 스킬·모델·렌더 프롬프트·추론 설정 같은 **실행 정책 전체**를 결속하지 못한다 — 정책 결속을 scope로 우회하면 모델·프롬프트 변경이 키에 안 잡힌다. 그건 epoch의 몫이다.

### 대안 B — epoch만 두고 카탈로그 digest는 생략
정책 버전만 수동으로 올린다. · **기각**: epoch는 완료 판정 규칙 변경은 잡지만, 다른 규칙의 추가·삭제·scope 전환은 못 잡는다. `scope: local` 규칙 신설이 재검토 0건이 되는 공백이 그대로 남는다.

## 촉발 근거 (rule-adr-recording 도그푸딩)
이 결정은 `rule-adr-recording`의 촉발 조건 중 **완료 판정·무효화 메커니즘(핵심 계약) 변경**과 **대안 기각**에 걸린다. 판정 키 정의를 바꾸는 결정이므로 ADR로 기록한다. ADR-0015(규칙 scope 분류)와 짝을 이룬다 — 완료정책 규칙을 local로 두어도 무효화를 잃지 않는 근거가 이 epoch다.
