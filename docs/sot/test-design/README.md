# docs/sot/test-design/ — 테스트 설계 (Test Design)

> 요구·사양을 **무엇을 어떻게 검증하는가** — 테스트 조건·커버리지·스위트의 권위 정의.

## 목적
요구·사양이 충족됐음을 어떻게 검증할지 확정한다. 요구·사양에서 테스트 조건을 도출하고, 블랙박스 커버리지 목표·스위트 구조·추적성·절차 원칙을 정한다(ISTQB 테스트 분석·설계 기준). 완료 판정에서 요구·사양과 함께 다뤄져(`rule-sot-readiness`), 검증 설계까지 끝나야 개발에 착수한다. Tester가 `test/` 코드를 짜는 상위 근거다. (요구 = 왜/무엇, 사양 = 무엇/어떻게, 테스트 설계 = 어떻게 검증.)

## 설계와 구현의 절단선
이 디렉터리는 **테스트 설계(SoT)** 다. 구체 테스트 코드는 `test/`(Tester 소관, D12)에 둔다. 경계 기준:

> 코드·실행 없이 문서만으로 확정되고 요구·사양에서 바로 도출되면 SoT, 코드·데이터·실행 환경이 있어야 확정되면 `test/`.

경계 사례(값 수준)의 타이브레이커: 구체 입력값·기대 출력은 문서에서 도출 가능해도 `test/`의 몫이다. SoT는 **기법·분할·경계의 식별**까지만 담는다 — 예: "상한 100에 경계값 기법 적용"은 SoT, 거기서 뽑은 실제 값(99·100·101)과 기대 결과는 `test/` 코드가 정한다.

| 여기 (SoT) | `test/` 코드 |
|---|---|
| 테스트 조건 (무엇이 참이어야 검증되나) | 구체 케이스 (실제 입력값·기대 출력) |
| 블랙박스 커버리지 목표 (동등분할·경계값·결정표·상태전이) | 테스트 데이터·픽스처 |
| 스위트 구조 | 실행 스크립트·자동화 |
| 요구·사양 ↔ 테스트 조건 추적성 | 환경 설정 |
| 절차 원칙 (검증 방법·관찰점) | 화이트박스 커버리지 측정 (결정/분기/MC-DC) |

화이트박스 커버리지를 SoT에 두지 않는 이유: 코드가 있어야 측정되므로, 개발 전 완료 판정의 조건으로 삼으면 영구히 미충족이 된다(`ADR-0008`).

## 필수 내용
- **범위** — 검증하는 요구·사양의 경계
- **테스트 조건** — 검증되어야 할 조건(`TD-n`)
- **커버리지 목표** — 블랙박스 기법과 목표 수준
- **스위트 구조** — 조건의 레벨·그룹 조직
- **추적성** — 요구·사양 항목 ↔ 테스트 조건
- **절차 원칙** — 검증 방법·관찰점
- **수용 기준** — 각 조건의 합격 판정(관찰 가능한 통과 기준)
- **참조** — 짝 요구·사양, 관련 rule·외부 표준

## 네이밍
- 파일명: 주제 기반 kebab-case, `<topic>.md` (예: `auth-login.md`). 번호 미사용. 짝 요구·사양과 같은 topic을 쓰면 추적이 쉽다.
- frontmatter `id`: 소문자 `td-<topic>` (예: `td-auth-login`).
- 항목 ID: `TD-n` (테스트 조건). 구체 test case는 `test/` 코드의 몫이라 문서 항목에 쓰지 않는다.
