# SoT 완료(개발 착수 준비) 판정 기준

## 규칙문
> `docs/sot/`의 requirements·specification·test-design이 개발을 자동으로 시작해도 되는 상태("완료")인지는 **① 형식 검증 ∧ ② 정합성·공백 검토 ∧ ③ 사용자 승인**이 **모두** 성립할 때만 참이며, 셋은 **동일한 SoT 콘텐츠**(같은 requirements·specification·test-design 트리)에 대해 성립해야 한다. ③은 마지막 관문일 뿐 ①②를 대신하지 못한다.

완료는 특정 SoT 콘텐츠 상태의 속성이다. requirements·specification·test-design에 새 변경이 들어오면 이전 ①②③ 결과는 무효가 되며, 바뀐 콘텐츠에 대해 다시 성립해야 한다. 완료는 그 req·spec·test-design이 의존하는 rule에도 결속된다 — 관련 rule이 바뀌면 기존 완료의 ② 검토가 무효가 되어 재검토가 필요하다(아래 적용범위의 재검토 트리거). 완료를 실제로 강제하는 것은 규칙문이 아니라 **머지 게이트**이며(아래 강제 매핑), 그 결속은 호스트 브랜치 보호(Phase 6)에 의존한다. 본 규칙은 "무엇이 완료인가"와 "무엇이 무엇을 강제하는가"를 정의한다.

### ① 형식 (기계 검증 · 필요조건)
- requirements·specification·test-design 각각, 항목이 담긴 문서가 최소 1개(README·`_TEMPLATE` 제외).
- 각 문서에 항목 ID가 있음 — 요구는 `FR-n`·`NFR-n`, 사양은 `SP-n`, 테스트 설계는 `TD-n`. test-design의 `covers`는 자신이 검증하는 요구·사양 항목을 가리킨다(참조 무결성은 아래 항목이 검사).
- **참조 무결성**: `covers`·`rules`가 가리키는 항목·rule id가 실재한다(dangling 참조 없음). 결정적으로 검사할 수 있어 ①에 둔다. (`related`는 힌트이므로 무결성 검사 대상이 아니다.)
- 미치환 플레이스홀더(`<...>`) 없음 — 코드 블록·인라인 코드는 검사에서 제외.
- 금지어 없음 — `TBD`·`TODO`·`FIXME`·`미정`·`추후`·`나중에` 등(대소문자·한글 변형 포함; 목록은 검사기가 관리).
- 각 요구·사양·테스트 설계 항목의 수용 기준 자리에 플레이스홀더·금지어가 아닌 문장이 있음. 기재된 내용이 실제로 검증 가능한지, 무의미한 filler인지는 기계가 판정하지 않는다 — ②의 몫이다.
- 형식 검증은 결정적이다. 같은 콘텐츠에는 항상 같은 결과를 낸다.

### ② 정합성·공백 (LLM 검토 · 필요조건)
- 스킬 `sot-readiness-review`가 requirements·specification·test-design을 항목 단위로 검토한다(검토 축·기록 포맷은 스킬이 규정).
- **호스트 CI가 콘텐츠당 1회 자동 실행**: SoT 콘텐츠가 바뀔 때마다 호스트 CI가 검토를 자동으로 한 번 실행하고, 판정을 **머지 게이트가 소비하는 검사 산출물**로 낸다. 그 산출물을 호스트 필수 검사 상태에 결속할지, 머지 게이트가 머지 직전에 직접 읽을지는 강제 메커니즘이 정한다(아래 강제 매핑). CI는 `open` blocking이 하나라도 있으면 `review_blocked`, 없으면 `review_clear`만 낸다 — 사용자 결정(`accepted`/`rejected`)은 verdict에 넣지 않는다(아래 강제 매핑에서 게이트가 얹는다). 같은 판정 키에 대해 **이미 완료된 산출물**이 있으면 재실행하지 않는다 — `review_clear`든 `review_blocked`든 완료된 검토다. 차단 결과를 재실행하면 비결정적 검토기가 매번 다른 finding을 내어 사용자 결정이 무효화되는 순환에 빠진다.
- **결속 키와 무효화 축**: ② 판정·각 finding·사용자 수용/기각은 **(SoT 트리[requirements·specification·test-design 콘텐츠] 해시 + 그 문서에 적용되는 rule 집합의 지문)을 키로** 결속한다. rule 지문은 범위별로 넣는다 — `scope: global` rule은 모든 완료 문서의 키에, `scope: local` rule은 그 rule을 `rules`에 선언한 문서의 키에만. 판정 재사용·전파는 **키의 두 성분이 모두 같을 때만** 한다(트리가 같고 SHA만 다른 rebase·merge 커밋 포함). 무효화는 어느 성분이 바뀌었는지로 갈린다 — **req·spec·test-design 트리 변경**이면 그 문서의 판정·finding·수용/기각 전부 무효 + ② **전체 재검토**, **적용 rule 지문만 변경**(트리 불변)이면 판정은 무효가 되나 재검토는 **축3(교차 정합성)에 한정**한다(아래 적용범위). 두 성분 어느 쪽도 바꾸지 않는 변경(예: 감사 로그 커밋 자체)은 ②를 재발화하지 않는다. 이로써 "콘텐츠당 1회"와 "SHA에 붙는 검사 결과"가 정합하고, rule 변경이 키를 바꿔 재검토를 실제로 발화시킨다. (이 2성분 `(트리 해시 + 적용 rule 지문)`을 **판정 키**라 하고 — 재사용·무효화·fail-closed·③ 승인 stale이 이 키를 쓴다 — finding 단위 사용자 표시·대조에는 여기에 `F-n + 내용 digest`를 더한 **완전 결속 키**를 쓴다.) **적용 rule 지문 = 적용되는 각 rule 파일 전체 내용의 해시를, 호스트가 그 PR의 제안된 머지 결과 상태에서 계산**한다(에이전트 산출을 신뢰하지 않는다). 이로써 한 PR이 규칙과 req·spec·test-design을 함께 바꿔도 req·spec·test-design을 **새 규칙 기준으로** 판정한다(옛 규칙 기준으로 통과하는 구멍을 막는다). 검사 코드·정책 자체는 `rule-protected-paths`대로 신뢰 ref(base)에서 읽어 후보 브랜치가 검사기를 바꾸지 못하게 한다. 비규범 부분(근거·예시) 편집도 지문을 바꿔 축3 재검토를 유발할 수 있으나, 축3 한정·저빈도라 보수적으로 수용한다. 지문이 머지 결과에 의존하므로, PR이 그대로여도 base(`main`)가 다른 rule 변경으로 전진하면 지문이 바뀌어 그 커밋의 판정 키가 달라질 수 있다(정확성상 의도된 동작 — 새 rule 반영).
- **게이트가 신뢰하는 산출물**: 게이트가 "남은 blocking이 모두 처리됐는가"를 계산하려면 blocking finding **목록과 각 상태**가 필요하다. 그 신뢰 출처는 **감사 로그 파일이 아니라 ② 검토 CI의 산출물**이다 — CI는 verdict와 함께 **판정 키(SoT 트리 해시 + 적용 rule 지문)와 각 open blocking finding의 `(F-n + 내용 digest)` 목록**을 검사 산출로 낸다. 게이트는 이 신뢰 목록을 호스트 채널의 사용자 표시(아래 스키마)와 **같은 완전 결속 키로** 대조하고, CI는 이 산출에 실린 판정 키로 재실행/스킵을 판정한다(감사 로그는 비신뢰라 읽지 않는다). 감사 로그는 사람이 읽는 사본일 뿐 게이트의 입력이 아니다. 여기서 **finding 내용 digest = 그 finding의 (축 + 참조 문서·항목 ID + 심각도 + 설명 본문)을 정규화해 취한 해시**로 결정적으로 정의한다 — CI 산출과 사용자 표시가 같은 값을 재현해야 대조가 안정적이다. 정규화 규칙(유니코드 NFC · 개행 LF 통일 · 앞뒤 공백 제거와 연속 공백 축약 · 참조 목록 정렬)은 검사기가 단일 구현으로 고정한다(형식 금지어 목록과 같은 방식). 마크다운 구조는 정규화하지 않는다 — "무엇이 같은 마크다운인가"는 정의되지 않아 결정성을 해친다.
- 검토는 CI 실행이라 문서를 작성한 세션과 자연히 분리된다(자기검토 편향 방지). 실행 모델·프롬프트는 검사 산출과 감사 로그에 기록한다.
- `blocking` finding은 두 갈래로만 닫힌다 — (a) SoT 변경으로 실제 해소(`resolved`; 트리가 바뀌므로 CI 재실행이 확인), (b) **사용자 채널 결정**으로 `accepted`(위험 인지 후 수용) 또는 `rejected`(오판). CI는 같은 콘텐츠를 재실행하지 않으므로, 오판도 검토 산출이 아니라 사용자가 닫는다.
- **사용자 채널 스키마**: `accepted`·`rejected`는 감사 로그 표기가 아니라 **호스트 채널**(그 커밋 PR)에서 **완전 결속 키 `(SoT 트리 해시 + 적용 rule 지문 + finding ID F-n + finding 내용 digest)`**를 참조해 표시한다 — 위 CI 신뢰 산출과 같은 키다. 위조 가능한 파일에 완료를 여는 권한을 두지 않고, 콘텐츠·rule 경계를 넘어 이월되지 않게 하기 위함이다. **판정 키 성분(트리 또는 적용 rule 지문) 어느 쪽이든** 바뀌면 그 표시는 무효가 되어 재확인이 필요하다.
- 검사 산출물이 없거나 현재 **판정 키(SoT 트리 + 적용 rule 지문)**에 대한 것이 아니면 **미완료로 처리한다(fail-closed)** — 트리든 적용 rule 지문이든 어긋나면 산출물은 무효다.

### ③ 사용자 승인 (최종 게이트)
- 사용자가 `rule-sot-change-user-gate`의 게이트 PR을 승인해야 완료가 확정된다.
- 승인은 승인 시점의 **판정 키(SoT 트리 + 적용 rule 지문)**에 대해서만 유효하다. requirements·specification·test-design push든 적용 rule 변경(지문 변화)이든 판정 키가 바뀌면 승인은 무효가 된다(호스트의 "최신 push 재승인" 보호 = dismiss-stale; ②의 fail-closed와 같은 키 기준). 승인이 걸린 판정 키는 **승인 시점 상태에 고정**하며 평가 시점에 다시 계산하지 않는다 — 재계산하면 base 전진으로 rule 지문이 바뀌었을 때 낡은 승인이 새 상태의 승인으로 되살아난다.
- 승인자·② 결정권자는 **지정 명단에 오른 사람 계정**이어야 한다. 저장소 권한만으로는 부족하고, 명단은 게이트가 읽는 입력이다. 기계 계정은 명단에 두지 않는다 — 사람이 자기 기계 계정으로 PR을 열고 사람 계정으로 승인하면 계정 이름 비교만으로는 자기승인을 걸러내지 못한다.
- 권한은 **평가 시점의 현재 값**으로 판정한다. 강등은 즉시 반영되고, 승격은 소급한다 — 지금 결정권자인 계정이 과거에 남긴 표시는 유효하다. 명단 등재는 결정권자의 명시적 행위이므로, 등재는 그 계정의 기존 표시를 함께 인정하는 것으로 본다. 표시 시점의 권한을 별도로 요구하지 않는다.

## 강제 매핑
각 조건은 서로 다른 메커니즘이 강제한다. 허브 게이트(`rule-protected-paths`·`ADR-0007`, Phase 3)는 "SoT는 사용자 게이트 PR로만"(경로)까지만 강제하며 완료 조건 자체는 평가하지 못한다. ①②③의 결속은 호스트 브랜치 보호(Phase 6)에 의존한다.

**머지 게이트의 두 실현.** 아래 표의 "머지 게이트"는 둘 중 하나로 실현한다 — **(가) 호스트 필수 검사**: 게이트가 판정을 검사 상태로 게시하고 브랜치 보호가 그 검사를 요구한다. **(나) 머지 컨트롤러**: `main`을 갱신할 권한을 지정 신원 하나로 제한하고, 그 신원이 머지 직전에 판정을 계산해 통과할 때만 머지한다. 어느 쪽이든 **판정은 실제로 착지하는 판정 키에 대해 신선해야 한다** — 이전에 계산된 통과를 그대로 재사용해 머지하는 경로는 없다. (나)는 필수 검사 표현력·위조 내성이 호스트마다 다른 문제를 피하고 "누가 갱신할 수 있는가"라는 공통 원시 기능만 쓴다.

| 조건 | 강제 메커니즘 | 강제 시점 |
|---|---|---|
| SoT PR로만 — task 경로 축 | 허브 게이트 pre-receive 경로 규칙: task 브랜치의 `docs/sot/**` 직접 수정 거부 (`rule-protected-paths`) | Phase 3 |
| SoT PR로만 — `main` 갱신 축 | `main` 갱신 권한을 지정 신원으로 제한 — 사람·CI·다른 봇은 `main`을 직접 갱신하지 못하고, SoT는 승인된 PR 머지로만 `main`에 도달 | Phase 6 |
| ① 형식 | 결정적 검사(CI) 산출물을 **필수 검사 또는 머지 컨트롤러의 머지 전 판정**이 소비 | Phase 6 |
| ② 검토 | 호스트 CI가 콘텐츠(트리)당 1회 자동 실행한 산출물을 **필수 검사 또는 머지 컨트롤러의 머지 전 판정**이 소비 | Phase 6 |
| ③ 승인 | 호스트 네이티브 필수 승인 + dismiss-stale + up-to-date-before-merge를 **독립 관문으로 유지**하고, 그 위에 **필수 검사 또는 머지 컨트롤러의 머지 전 판정**이 승인 신원·명단·판정 키를 재확인 | Phase 6 |
| SoT PR 소스 브랜치 = `sot/<slug>` | 소스 브랜치가 `sot/<slug>` 형식(접두 `sot/` + 단일 kebab 세그먼트, `^sot/[a-z0-9]+(?:-[a-z0-9]+)*$`)이 아닌 SoT 변경 PR은 머지 게이트가 거부 (감사 이력 보존 패턴이 실제 적용되도록) | Phase 6 |
| SoT PR 출처 = 대상 저장소 | PR의 head 저장소가 대상 저장소와 다르면(포크) 머지 게이트가 거부 — 브랜치 이름 규약만으로는 포크가 흉내 낼 수 있다 | Phase 6 |
| 감사 이력 보존 | `main` 브랜치 규칙이 허용 머지 방식을 merge commit으로 제한 + force-push·브랜치 삭제 차단 (머지 실행자의 선택에 기대지 않는다) | Phase 6 |

`main` 머지는 **① 통과 ∧ ② 성립 ∧ ③ 승인**이 **동일 SoT 콘텐츠(트리)**에서 함께 참일 때만 열린다. 여기서 **② 성립 = `review_clear`(CI 산출에 open blocking 없음) 또는 CI가 낸 open blocking finding 각각이 호스트 채널에서 `accepted`·`rejected` 표시를 받음**이다(트리 변경으로 실제 해소된 것은 CI 재실행이 `review_clear`로 반영하므로 `resolved`는 이 식에 따로 두지 않는다). 게이트가 대조하는 blocking 목록은 감사 로그가 아니라 CI 검사 산출물이다. `accepted`·`rejected`는 ② 검토를 재실행·상태변경하지 않으며, 게이트가 검사 산출물 위에 사용자 결정을 얹어 계산한다. 로컬 훅은 권고 수준이다.

**적용 대상.** 완료 강제는 **SoT를 바꾸는 PR**에만 적용한다. 제안된 머지 결과가 `docs/sot/requirements/`·`docs/sot/specification/`·`docs/sot/test-design/`를 바꾸지 않는 PR은 ①②③의 대상이 아니며 게이트가 통과시킨다. SoT를 바꾸는지는 **제안된 머지 결과의 변경분**으로 판단하고 브랜치 이름·커밋 메시지로 판단하지 않는다. 머지 게이트를 (나) 머지 컨트롤러로 실현하면 컨트롤러가 저장소의 모든 머지를 수행하므로, 이 구분이 없으면 SoT와 무관한 PR이 브랜치 이름 규약에 걸려 영구히 막힌다.

**신선성.** 게이트 판정은 **실제로 착지하는 판정 키**에 대해 계산돼야 한다. 판정 시점과 머지 시점 사이에 base가 전진하거나 결정·승인·권한이 바뀌면 이전 판정은 무효다. (나) 머지 컨트롤러는 머지를 전역으로 직렬화하고 머지 직전에 판정을 다시 계산해 이를 보장한다 — 결정 취소·권한 변경 같은 이벤트를 빠짐없이 좇아 이전 판정을 무효화하는 방식에 기대지 않는다.

## 근거
- 완료는 곧 자동 개발 시작의 방아쇠(D6)다. 방아쇠 조건이 느슨하면 미완성 SoT 위에서 개발이 출발한다.
- 기계 검증(①)만으로는 "형식은 갖췄으나 내용이 비었거나 모순된" 문서를 거르지 못한다 — 의미 판단(②)이 필요하다. 다만 LLM 검토는 비결정적이므로, 콘텐츠(트리)당 1회 실행·호스트 검사 결속·소멸 규칙·분리 실행·fail-closed로 재현성과 우회를 막는다.
- ③을 단독 충분조건으로 두면 ①②가 실패해도 승인만으로 완료가 성립하는 모순이 생긴다. 완료는 세 조건의 논리곱이다.
- 판정을 콘텐츠에 결속하지 않으면 "승인한 문서"와 "개발이 출발하는 문서"가 달라질 수 있다(승인 후 내용 바꿔치기). 머지 커밋은 승인 커밋과 SHA가 다를 수 있으므로 SHA가 아니라 콘텐츠(트리) 동일성으로 본다.
- 감사·콘텐츠 결속을 위해 SoT 브랜치는 squash 머지를 막는다 — squash는 사용자가 승인한 커밋 자체를 없애 새 단일 커밋으로 바꿔 감사·콘텐츠 추적을 깨뜨린다. merge commit은 승인 head를 부모로 이력에 보존한다.

## 적용범위
- **대상**: `docs/sot/requirements/`·`docs/sot/specification/`·`docs/sot/test-design/`의 완료 판정과, 그 완료를 전제로 하는 자동 개발 시작 트리거(D6).
- **개발 시작 트리거**: **requirements·specification·test-design이 변경된 PR**에만 적용한다. `rule/`·`ADR`만 바꾸는 PR은 개발을 시작시키지 않는다. 한 PR이 req/spec/test-design과 rule/ADR을 함께 바꾸면 req/spec/test-design 변경을 기준으로 판정한다.
- **테스트 설계의 완료 범위**: test-design의 완료 판정은 테스트 조건·블랙박스 커버리지 목표·스위트 구조·추적성·절차 원칙까지만 대상으로 한다. 구체 케이스·데이터·픽스처·실행 스크립트·환경과 화이트박스 커버리지(결정/분기/MC-DC)는 `test/` 코드의 몫이라 요구하지 않는다 — 요구하면 코드 없이 충족 불가라 완료가 영구 미충족이 된다(설계/구현 절단선, `test-design/README`·`ADR-0008`).
- **재검토 트리거 (rule 변경)**: 완료는 req·spec·test-design이 의존하는 rule에도 결속되므로, rule이 바뀌면 그 rule을 따르는 기존 완료의 ② 검토가 무효가 되어 재검토가 필요하다. 재검토 범위는 rule frontmatter의 `scope`로 정한다 — **`scope: local`**은 그 rule을 `rules`에 선언한 req·spec·test-design, **`scope: global`**(용어·네이밍 등 횡단)은 완료된 req·spec·test-design 전량. `scope` 미기재 rule은 보수적으로 global(전량)로 취급한다. 이 재검토는 축3(교차 정합성)에 한정한다. **신규 rule·`scope` 전환**: rule이 새로 추가되거나 기존 rule의 `scope`가 `local`로 바뀌면, 기존 완료 문서는 그 rule을 `rules`에 선언했을 수 없으므로(존재하지 않았거나 전역이었다), 이 이벤트는 declared `scope`를 무시하고 완료된 req·spec·test-design **전량에 대해 축3 재검토를 1회** 발화한다. 이 스윕에서 ②가 미선언 rule 의존을 finding으로 걷어낸 뒤에는 `rules` 선언이 신뢰 가능해져, 이후 그 rule의 변경은 `scope: local` 타겟팅을 쓴다. **초기 마이그레이션**: 완전성 검사(층1: 축3의 미선언 의존 발견)와 이 재검토 결속이 활성화되는 시점(①②③ 강제 도입 = Phase 6, 또는 프로젝트가 이 규칙을 뒤늦게 채택)에는, 그 이전에 완료된 req·spec·test-design 전량을 축3으로 **최초 1회 스윕**해 `rules` 선언 완전성을 확립한다. 이 스윕 뒤에는 층1(완료 시점 검사)과 전량 스윕(신규·전환)이 불변식을 유지한다. 다만 층1 완전성은 LLM 판정이므로 미선언 의존을 놓칠 수 있다. **핵심 잔여**: 문서 D가 local rule R을 의존하면서 `rules`에 누락하면, **R이 바뀌어도 D는 R의 재검토 대상(= R을 `rules`에 선언한 문서)에서 빠져 재검토되지 않는다** — R이 활발히 변경돼도 그렇다. D의 이 누락은 D가 다른 이유로(자기 변경·선언한 다른 rule 변경·global rule 변경·신규/전환 전량 스윕) 재검토될 때에만 층1이 교정한다. 따라서 그 사이 R이 D와 충돌을 만들면 D의 다음 재검토 전까지 놓친다. 이는 비결정 검토기(층1)를 필요조건으로 둔 대가로 수용하며, 완화가 필요하면 R 변경 시 재검토 대상에 '최근 전량 스윕 이후 완료분'을 얇게 더한다. **운영 노트**: 활성화 전에 기존 rule 파일에 `scope`를 명시적으로 채워 둔다 — 미기재는 보수적으로 global로 취급돼 그 rule 편집이 완료 전량 재검토를 부른다. rule 변경 자체는 개발 시작 트리거가 아니다. 재검토 `blocking`이 이미 착수된 작업에 미치는 효과(중단·경고·다음 게이트 반영)는 트리거 상태 머신(Phase 8)이 다룬다.
- **예외**: 개발 착수 이후의 SoT 재변경에 대한 트리거 재발화·멱등성(중복 실행 방지)은 본 규칙이 아니라 트리거 상태 머신(Phase 8)이 다룬다.

## 예시
**준수 (✓)**
- requirements의 `FR-1`을 specification의 `SP-3`이 구체화하고 test-design의 `TD-2`가 검증하며 형식 검사 통과 → 호스트 CI가 `sot-readiness-review`를 1회 실행, `review_clear`가 그 트리의 검사 산출물로 고정 → 사용자가 같은 콘텐츠의 PR을 승인, 머지 게이트가 머지 직전 판정 후 머지 → 개발 시작.
- 검토에서 `blocking`(사양 누락)이 나옴 → 작성자가 SoT를 고쳐 커밋(트리 변경, `resolved`) → 바뀐 콘텐츠에 대해 CI가 재실행 `review_clear` 후 승인.
- ② 검사가 `review_blocked`(사소한 공백 지적 하나)인데 사용자가 그 finding을 호스트 채널에서 `accepted`(감수) 또는 `rejected`(오판)로 표시 → 게이트가 남은 `open` blocking이 모두 처리됐음을 확인, ③ 승인과 함께 머지 허용.
- 네이밍 rule(`scope: global`)을 바꿈 → 완료된 req·spec·test-design 전량의 축3 재검토 발화 → 충돌 없으면 완료 유지, 있으면 `blocking`으로 재작성 요구.

**위반 (✗)**
- 형식은 통과했으나 ② 검사가 `review_blocked`(미처리 blocking 존재)인데 사용자가 머지 → ②를 건너뛴 완료. 게이트가 막아야 한다.
- 사용자가 승인한 뒤 specification에 커밋을 더 얹고(트리 변경) 그대로 개발 시작 → 승인 트리 ≠ 개발 트리, 승인·수용 모두 무효(재확인 필요).
- 감사 로그 파일에 `accepted`를 직접 써넣어 blocking을 닫으려 함 → accepted는 호스트 채널에 결속되므로 게이트는 파일 표기를 신뢰하지 않는다.
