# SoT 완료(개발 착수 준비) 판정 기준

## 규칙문
> `docs/sot/`의 requirements·specification·test-design이 개발을 자동으로 시작해도 되는 상태("완료")인지는 **① 형식 검증 ∧ ② 정합성·공백 및 선언 완전성 검토 ∧ ③ 사용자 승인**이 **모두** 성립할 때만 참이며, 셋은 **동일한 SoT 콘텐츠**(같은 requirements·specification·test-design 트리)에 대해 성립해야 한다. ③은 마지막 관문일 뿐 ①②를 대신하지 못한다.

완료는 특정 SoT 콘텐츠 상태의 속성이다. requirements·specification·test-design에 새 변경이 들어오면 이전 ①②③ 결과는 무효가 되며, 바뀐 콘텐츠에 대해 다시 성립해야 한다. 완료는 그 req·spec·test-design이 의존하는 rule에도 결속된다 — 관련 rule이 바뀌면 기존 완료의 ② 검토가 무효가 되어 재검토가 필요하다(아래 적용범위의 재검토 트리거). 완료를 실제로 강제하는 것은 규칙문이 아니라 **머지 게이트**이며(아래 강제 매핑), 그 결속은 Phase 6 강제 증분에 의존한다(머지 게이트는 두 방식으로 실현한다 — (가) 집계 게이트, (나) 머지 컨트롤러; Phase 6은 (나)를 채택한다, 아래 강제 매핑). 본 규칙은 "무엇이 완료인가"와 "무엇이 무엇을 강제하는가"를 정의한다.

### ① 형식 (기계 검증 · 필요조건)
- requirements·specification·test-design 각각, frontmatter `items`에 항목을 선언한 문서가 최소 1개(README·`_TEMPLATE` 제외).
- 선언 정본은 frontmatter `items`다. 각 문서가 `items`에 선언한 항목 ID가 문서 종류 규약을 따르고(요구 `FR-n`·`NFR-n`, 사양 `SP-n`, 테스트 설계 `TD-n`), 같은 `(topic, ID)` 조합이 둘 이상 선언되지 않는다(중복 선언은 참조 대상을 모호하게 만든다). 본문에 등장하는 그 문서 고유 종류의 항목 ID는 `items` 선언과 일치해야 한다(본문에만 있고 선언에 없거나, 선언에만 있고 본문에 없으면 형식 위반; 타 종류 ID는 참조라 제외). test-design의 `covers`는 자신이 검증하는 요구·사양 항목을 가리킨다(참조 무결성은 아래 항목이 검사).
- 문서 `id`는 `<종류접두>-<topic>`(요구 `req-`·사양 `spec-`·테스트 설계 `td-`)이며 `topic`은 파일 stem이다. `id`↔topic 불일치는 형식 위반이고, 위 `(topic, ID)`와 교차 접두 참조가 이 topic 규약에 근거한다.
- **참조 무결성**: `covers`가 가리키는 요구·사양 항목이 대상 문서의 `items`에 선언돼 있고, `rules`가 가리키는 rule id가 실재하며 **`status: active`여야 한다**(dangling·사문(死文) 참조 없음). 비활성 rule을 `rules`에 선언하면 형식만 통과하고 적용 집합·지문에서 조용히 빠져 "결속됐다"는 착각을 낳으므로 ①이 막는다. `items`↔`covers`·rule 실재/활성 대조라 결정적이어서 ①에 둔다. (`related`는 힌트이므로 무결성 검사 대상이 아니다.)
- 미치환 플레이스홀더(`{{...}}`) 없음 — 코드 블록·인라인 코드는 검사에서 제외. 플레이스홀더 표기를 `{{...}}`로 두는 것은 `<...>`가 Markdown 자동링크(`<https://…>`)·제네릭 타입(`Map<K,V>`) 같은 정당한 본문과 충돌해 오탐하기 때문이다.
- 금지어 없음 — 닫힌 목록 `TBD`·`TODO`·`FIXME`·`미정`·`추후`·`나중에`만 검사한다(ASCII는 대소문자무시·단어경계, 한글은 정확 매칭). 목록은 검사기가 코드 상수로 단일 관리하며 오검 시 조정한다.
- 각 요구·사양·테스트 설계 항목의 수용 기준이 **`## 수용 기준` 섹션(레벨2 제목 정확 일치) 하위의 체크박스**로 채워져 있고, 그 내용이 플레이스홀더·금지어가 아닌 문장이다. 수용 기준 인정은 이 섹션 안으로 한정한다 — 다른 섹션에 있는 체크박스는 수용 기준으로 세지 않는다(엉뚱한 체크박스가 수용 기준 자리를 가로채는 것을 막는다). `items`를 선언한 문서에 `## 수용 기준` 섹션이 없으면 형식 위반(`수용 기준 섹션 없음`)이다. 기재된 내용이 실제로 검증 가능한지, 무의미한 filler인지는 기계가 판정하지 않는다 — ②의 몫이다.
- 형식 검증은 결정적이다. 같은 콘텐츠에는 항상 같은 결과를 낸다.

### ② 정합성·공백 + 선언 완전성 (LLM 검토 · 필요조건)
- 스킬 `sot-readiness-review`가 requirements·specification·test-design을 항목 단위로 검토한다(검토 축·기록 포맷은 스킬이 규정).
- **호스트 CI가 키당 1회 자동 실행 (증거 산출)**: 각 키(정합성=판정 키, 선언 완전성=완전성 스윕 키)가 바뀔 때마다 호스트 CI가 해당 검토를 자동으로 한 번 실행해 **증거 산출**(그 키 + verdict + open blocking `(F-n + 내용 digest)` 목록)을 낸다. CI는 `open` blocking이 하나라도 있으면 `review_blocked`/`completeness_blocked`, 없으면 `review_clear`/`completeness_clear`만 낸다 — 사용자 결정(`accepted`/`rejected`)은 verdict에 넣지 않는다. **이 검토 산출 자체는 머지를 직접 막는 판정이 아니라 증거다.** 머지를 막는 완료 술어(① 통과·두 검토 증거·사용자 override·③ 승인·키 일치)는 **머지 게이트**가 계산한다(아래 강제 매핑: (가) 집계 게이트 `sot-readiness-gate`가 서버사이드 필수 검사로 계산 / (나) 머지 컨트롤러가 머지 직전에 계산; Phase 6은 (나) 채택). (가)에서 증거 검사와 집계 게이트를 별도 필수 검사로 분리하는 이유: 호스트 branch protection에서 "실패 상태의 필수 검사"는 별도 게이트가 성공해도 머지를 막으므로, 검토가 `blocked`여도 사용자 override로 완료가 열리려면 검토 산출이 필수 검사가 아니라 집계 게이트의 입력이어야 한다 — (나) 머지 컨트롤러는 이를 **별도 호스트 필수 검사로 둘 필요는 없으나**, 검토 산출을 즉시 머지 실패로 보지 않고 컨트롤러가 같은 증거 산출을 직접 읽어 override와 합성해 술어를 계산하는 **증거 산출과 완료 판정의 논리적 분리는 유지한다**.
- **결속 키와 무효화**: ②는 입력이 다른 두 갈래 검토를 낸다 — **정합성·공백**(4축; 문서가 *선언한 + global* 적용 rule을 따르는가·문서 간 정합·공백)과 **선언 완전성**(문서가 실제 의존하는 rule을 `rules`에 빠짐없이 선언했는가; 이를 판정하려면 활성 rule **본문 전량**을 봐야 한다). 입력이 다르므로 **각기 다른 키**에 결속한다.
  - **판정 키** — 정합성·공백 판정·각 finding·사용자 수용/기각을 결속. **projection 전체에 대한 단일 값**이며 **네 성분**의 조합이다.
    - **(1) SoT 트리 해시** — **target-content projection**(= requirements·specification·test-design 트리에서 각 디렉터리의 `README.md`·`_TEMPLATE.md`를 제외한 것; 이들은 대상 콘텐츠가 아니라 안내·빈 양식이다)에 대한 해시 = 그 projection 파일들의 `{path, 내용}` **record-set을 아래 정규화·digest 규약으로 직렬화한 digest**(git tree SHA가 아니다 — 그래서 개행(LF) 정규화가 적용되고, 아래처럼 트리 내용이 같고 커밋 SHA만 다른 rebase·merge에도 안정적이다; 레코드가 경로를 포함하므로 실제 파일 rename은 정상적으로 이 키를 바꾼다). ① 형식이 이들을 완료 대상에서 제외하는 것과 정렬한다. **이 투영은 트리 해시뿐 아니라 ② 검토 입력·개발 시작 트리거에도 같은 정의로 재사용**한다(같은 판정 키가 항상 같은 실제 입력을 대표하도록).
    - **(2) 적용 rule 지문** — projection에 **적용되는 rule 집합** 전체를 이은 단일 지문(적용 rule 각각의 `{path, frontmatter 포함 파일 전체 내용}` 레코드를 아래 **정규화·digest 규약**(경로 정렬 + 길이-prefix 프레이밍)으로 이어 해싱 — 구분자 없는 연접이 아니라 완전성 스윕 키와 같은 canonical serialization을 쓴다). 적용 집합은 projection 전체 기준의 **단일 합집합** = `status: active`인 rule 중 **모든 `scope: global` rule + 어느 완료 문서든 `rules`에 선언한 `scope: local` rule**(문서별 지문이 아니다 — 준수 판정 시에만 문서↔선언 매핑을 쓴다). 지문이 rule 전체 내용을 해싱하므로 **적용 rule의 본문 변경은 판정 키에 결속된다.** 아무 문서도 선언하지 않은 local rule은 적용 집합 밖이라 그 본문 변경이 판정 키를 안 바꾼다 — 대신 그 본문은 아래 완전성 스윕 키에 결속돼 선언 완전성 검사가 재실행된다.
    - **(3) `review_policy_epoch`** — 검토 정책 버전(아래 정의). 완료 판정 규칙·검토 절차·검토 주체·실행기가 바뀌면 이 성분이 바뀐다.
    - **(4) rule catalog manifest digest** — 규칙 카탈로그 구조(`{path, id, scope, status}`)의 다이제스트(아래 정의). 규칙 목록 구조가 바뀌면(신규·삭제·이동·scope·status) 정합성 재검토를 발화해 문서가 새 구조를 반영·재선언하게 한다.
  - **완전성 스윕 키**(별도) — **선언 완전성 검사**를 결속. = (projection 트리 해시, **활성 rule 카탈로그 입력 digest**, `review_policy_epoch`)의 조합. **활성 rule 카탈로그 입력 digest = 완전성 검사에 실제로 직렬화돼 들어가는 바이트의 digest**로 정의한다 — 정규화한 repo-relative `path`로 정렬한 `{path, frontmatter 포함 파일 전체 내용}` 레코드의 canonical serialization(활성 rule만; README·`_TEMPLATE` 제외). 즉 `path`도 digest에 포함하므로 파일 이동(rename)도 이 키를 바꾼다. 정합성 축이 보는 적용 rule(선언 local + 활성 global)과 달리 완전성은 **활성 rule 전부의 본문**을 봐야 "선언 누락한 의존"을 탐지하므로, 그 입력 전량을 이 키에 결속한다. 그래서 **미선언 local rule의 본문·경로만 바뀌어도** 이 키가 바뀌어 완전성 검사가 재실행된다 — "같은 키가 서로 다른 실제 입력을 대표하는" 캐시 우회를 닫는다. 완전성 검사의 LLM 컨텍스트에는 이 결속 입력만 넣고, 결과에 영향을 주면 안 되는 메타데이터(PR 번호·ref·시각)는 넣지 않는다(호스트가 verdict 산출 뒤 감사 로그에만 덧붙인다) — 안 그러면 키에 안 들어간 입력이 판정을 흔든다. 성분(2) 적용 rule 지문·성분(4) catalog manifest는 정합성용이라 이 키에 들지 않고, 반대로 카탈로그 입력 digest는 판정 키에 들지 않는다(넣으면 미선언 rule 편집마다 정합성 전량이 헛발화 — `ADR-0015` 효율).

  **무효화·재검토는 각 키 단위**로 판정한다. **판정 키** 네 성분이 모두 같으면(트리 동일, SHA만 다른 rebase·merge 포함) 정합성·공백 판정·finding·수용/기각을 재사용하고, **하나라도 다르면 projection 전체를 현재 입력만으로 4축 전량 재검토**한다. **완전성 스윕 키**가 같으면 선언 완전성 판정을 재사용하고, 다르면 projection 전량에 대해 완전성 검사를 재실행한다. 두 키가 대개 함께 바뀌지만(트리·epoch는 공통 성분), 미선언 local rule 본문 편집은 완전성 스윕 키만 바꾼다. 어느 검토든 이전 회차 finding을 이월하거나 일부 축·문서를 메모이즈하지 않는다(D27; 세탁·유실 방지). ②검토가 문서 간 홀리스틱(커버리지 매트릭스·교차 정합성)이라 부분 재검토가 불건전하므로, **대상을 문서로 쪼개지 않고 키가 바뀌면 전량**을 본다. `scope`는 "어느 문서를 다시 볼지"가 아니라 **"어떤 rule 변경이 성분(2)에 들어가 정합성 무효화를 유발하는가"**를 정한다(global은 항상, local은 선언된 경우만 — C1). 변경의 기준선은 직전 실행이 아니라 **base(`main`)의 현재 완료 상태 대비 제안된 머지 결과**이므로(재검토는 늘 projection 전량이라 대상 선정 자체가 없고, 이 기준선은 키 값이 직전 실행 기준으로 좁혀지지 않게 한다), 같은 PR의 후속 push에서도 그 PR이 base 대비 바꾼 것 전부가 키에 반영된다.

  여러 성분이 동시에 바뀌어도 각 키는 머지 결과에서 **한 번** 계산되고 그 키의 재검토도 **한 번(projection 전량)** 실행돼 중복 발화·분기 순서 의존이 없다. 어느 키도 안 바꾸는 변경(예: 감사 로그 커밋)은 ②를 재발화하지 않으므로 "키당 1회"와 "SHA에 붙는 검사 상태"가 정합한다. (재사용·무효화·fail-closed·③ 승인 stale이 이 두 키를 쓴다. finding 단위 사용자 표시·대조에는 해당 키에 `F-n + 내용 digest`를 더한 **완전 결속 키**를 쓴다.) **완료 = ① 형식 통과 ∧ 정합성 성립 ∧ 완전성 성립 ∧ ③ 승인**. 여기서 각 **성립** = 그 검토가 **현재 키에서 `clear`**(open blocking 없음)이거나 그 검토의 **open blocking이 모두 `accepted`/`rejected`**됨이다 — `clear` 전용이 아니라 사용자 수용도 blocking을 닫는다(강제 매핑 게이트식과 동일). ②를 `clear`-only로 읽으면 finding을 다 수용해도 안 열리는 override 교착이 생기므로, 완료식은 두 검토의 `성립`의 논리곱으로 쓴다.

  성분은 두 출처에서 계산한다(에이전트 산출은 신뢰하지 않는다). **저장소 파생**(트리 해시·적용 rule 지문·catalog manifest digest·완전성 스윕 키의 활성 카탈로그 입력 digest·epoch의 규칙·스킬 내용 부분)은 **호스트가 그 PR의 제안된 머지 결과 상태에서** 계산하고, **실행 환경 파생**(epoch의 모델 revision·신뢰 실행기 버전·스캐폴드 프롬프트·추론 설정 digest)은 **신뢰 CI 실행 환경**이 제공한다.
  - **정규화·digest 규약(모든 저장소 파생 성분 공통)**: 독립 구현·플랫폼 간 동일 값을 재현하도록 아래 정규형에 **SHA-256**을 적용한다. **경로** = repo-relative POSIX `/` 구분자, Unicode NFC. NFC 정규화 후 서로 같아지는 두 경로가 공존하면 **fail-closed(오류)**로 처리한다(정렬 동률의 열거순 의존을 차단). **파일 내용** = 선두 UTF-8 BOM(`EF BB BF`)이 있으면 **1회만** 제거하고 UTF-8 디코드한 뒤, 개행을 **`CRLF→LF`, 이어 남은 단독 `CR→LF`** 순으로 정규화하고 UTF-8 재인코딩한 바이트(Git blob의 CRLF·autocrlf 차이를 흡수). 그 외 유니코드 줄 구분자(U+2028·U+2029 등)와 **파일 종단 개행 유무는 보존**한다(개행 정규화 대상은 CR·LF뿐); 내용 자체는 NFC 정규화하지 않는다 — 유니코드 이형은 기껏해야 과잉 재검토라는 안전 방향으로만 흐른다; **UTF-8 디코드가 실패하는(바이너리) 파일은 fail-closed(오류)**. **길이-prefix 프레이밍**: `len(x)` = 뒤따르는 바이트열 x의 **옥텟(바이트) 수**(텍스트 성분은 UTF-8 인코딩 후의 바이트 수, digest 성분은 32)를 담은 **uint32 big-endian** 4바이트이며 값을 `len(x)‖x`로 감싼다(길이가 `2³²−1`을 넘으면 오류). **레코드** = 스키마로 **필드 수·순서를 고정**한 뒤 각 필드를 `len(필드)‖필드`로 차례로 연쇄한다 — `{path, 내용}` 2필드 파일 레코드는 `len(경로)‖경로‖len(내용)‖내용`, `{path,id,scope,status}` 4필드 catalog manifest 레코드는 `len(path)‖path‖len(id)‖id‖len(scope)‖scope‖len(status)‖status`. 어느 필드가 부재·null·기형(기대 문자열이 아닌 리스트·맵 등)이면 **fail-closed(오류)** — 단 catalog manifest의 **`scope`는 키 부재·null 모두 효과값 `"global"`로 정규화한 뒤 직렬화**한다(§적용범위의 "미기재는 보수적으로 global 취급"과 일치; `path`·`id`·`status`의 부재·null·비문자열은 오류). 나아가 **값 도메인도 fail-closed**로 검증한다 — 정규화 후 `scope ∈ {"local","global"}`·`status ∈ {"active","deprecated","superseded"}`가 아닌 문자열(빈 문자열·오타 `activ`·`globla` 등)이면 오류이고, `id`는 비어 있지 않고 rule ID 규약 — `rule-` 접두 + 소문자 kebab(세그먼트 `[a-z0-9]+`를 `-`로 이음; 정규식 `^rule-[a-z0-9]+(-[a-z0-9]+)*$`) — 을 만족해야 한다. 이는 오타 값이 적용 집합(정확히 `local`/`global`만)·활성 집합(정확히 `active`만)에서 **조용히 빠지는 fail-open**을 차단한다(키 계산기가 그 rule을 무시하는 대신 완료를 막는다). **레코드 집합**(record-set) = 경로 바이트 오름차순 정렬 후 위 레코드들을 이어붙인 바이트열이며, 그 **record-set digest = 이 바이트열의 SHA-256(raw 32바이트)** — 구분자 주입·경계 모호성이 없다. **합성 키**(epoch·판정 키·완전성 스윕 키처럼 이질 성분을 묶는 값) = **키 종류별로 다른 고정 ASCII 버전 태그**(도메인 분리, 예 `axdt.judgment-key.v1`)를 첫 성분으로 두고, 규범 순서로 고정한 성분 열의 각 성분(태그 포함)을 `len‖canonical(성분)`으로 감싸 연쇄한 바이트열의 **SHA-256**이다. `canonical(·)`은 타입별로 고정한다 — **digest 성분 = raw 32바이트**(hex 문자열이 아니다), **텍스트 성분**(UTF-8로 넣는 값 — 합성 키의 모델 revision·버전 태그, catalog manifest 레코드의 `id`·`scope`·`status` 등) = UTF-8 바이트, **파일·파일군 성분**(규칙·스킬 내용 등) = 그 파일들의 **record-set digest(raw 32바이트)** — 원문을 직접 넣지 않고 먼저 축약, **JSON 성분**(추론 설정 등) = RFC 8785 JCS 바이트의 **SHA-256(raw 32바이트)**(JSON도 digest로 축약). 이로써 태그·길이·인코딩·사전해시 여부·성분 경계가 유일해져 독립 구현이 같은 preimage·digest를 낸다(각 성분의 소속·순서는 해당 절). 이 규약의 CRLF·Unicode·경로 충돌·필드 프레이밍·정렬·합성 키 조립 **골든 시험 벡터(preimage 바이트 + 기대 hex)**는 **실행기(판정 키 계산기)**가 단일 구현으로 산출·고정한다(Phase 6 conformance 테스트; 규약이 구현 자유도를 남기지 않으므로 골든 벡터는 알고리즘 확인·회귀 방어용이다).
  - **합성 키 규범 상수(태그·성분 순서 — 독립 구현 preimage 고정)**: 각 합성 키의 버전 태그는 **정확한 ASCII 상수**로, 성분은 아래 **순서**로 확정한다(성분은 모두 위 `canonical(·)`로 축약해 순서대로 `len‖값`으로 감싼다; 태그도 첫 성분으로 `len‖UTF-8`). 태그·순서가 바뀌면 `.v2`로 올린다.
    - **판정 키** = `axdt.judgment-key.v1` · 순서: SoT 트리 해시 → 적용 rule 지문 → `review_policy_epoch` → rule catalog manifest digest (모두 digest 32바이트).
    - **완전성 스윕 키** = `axdt.completeness-sweep-key.v1` · 순서: projection 트리 해시 → 활성 카탈로그 입력 digest → `review_policy_epoch` (모두 digest 32바이트).
    - **`review_policy_epoch`** = `axdt.review-policy-epoch.v1` · 순서: 완료 판정 규칙 두 파일 record-set digest → 검토 스킬 디렉터리 record-set digest → 모델 revision(텍스트) → 신뢰 실행기 버전 digest → 프롬프트 스캐폴드 정규화 digest → 추론 설정 digest.
    - **완전 결속 키**(finding 표시·대조) = `axdt.finding-binding-key.v1` · 순서: 해당 검토의 키(판정 키 또는 완전성 스윕 키, digest 32바이트) → `F-n`(텍스트) → finding 내용 digest.
  - **적용 rule 지문 = 적용되는(선언된 + global) 각 rule의 `{path, frontmatter 포함 파일 전체 내용}` 레코드를 위 정규화·digest 규약(경로 정렬 + 길이-prefix 프레이밍)으로 이어 해싱.** 이로써 한 PR이 규칙과 req·spec·test-design을 함께 바꿔도 req·spec·test-design을 **새 규칙 기준으로** 판정하고(옛 규칙 기준으로 통과하는 구멍을 막는다), 선언된 rule의 본문 변경이 결속된다.
  - **`review_policy_epoch` = 위 합성 키 규약으로 조립한 성분들의 SHA-256.** 성분(규범 순서) = (저장소 파생) **완료 판정 규칙 두 파일(`rule-sot-readiness`·`rule-sot-change-user-gate`)의 record-set digest** + **검토 스킬(`.claude/skills/sot-readiness-review/**` 디렉터리 전체 파일)의 record-set digest**(보조 파일 추가도 결속; 디렉터리에 UTF-8 아닌 파일이 있으면 위 규약대로 fail-closed) + (실행 환경 파생) 검토 실행 모델의 불변 revision(텍스트) + 신뢰 실행기 버전 digest + 정책 불변 프롬프트 스캐폴드·시스템 지시의 정규화 digest + 결과에 영향을 주는 추론 설정(예: 온도·top-p) digest. 각 성분은 위 규약의 타입별 `canonical(·)`로 축약해 넣는다(파일군 = record-set digest, revision = UTF-8 텍스트, 추론 설정 = JCS JSON의 digest).
     - **신뢰 실행기 버전** = CI 하니스·입력 직렬화기·응답 파서/verdict 조립·판정 키 계산기·프롬프트 스캐폴드의 revision(`rule-protected-paths`의 **실행기 manifest가 열거하는 대상**을 단일 출처로 삼는다 — 보호경로 glob은 manifest의 (a)+(b) 전체를, 이 epoch revision digest는 **(a) 실행기 코드 부류만** 본다). base 실행기 코드만 바뀌어도 네 성분이 같은 채 실행 결과가 달라질 수 있어(같은 키 캐시 오재사용), 이 revision을 epoch에 넣어 실행기 promotion이 기존 판정을 무효화하게 한다.
     - **프롬프트 digest 경계**: 여기 들어가는 것은 **정책 불변부**(고정 시스템 지시·검토 지침 스캐폴드)뿐이고, **실행별 가변 데이터**(대상 SoT·rule 입력 (a)(b) 본문, 판정 키·완전성 키 값, PR 번호·ref·시각)는 **제외**한다 — 넣으면 (i) SoT·카탈로그 본문이 epoch를 통해 판정 키로 새어들어 §완전성 입력의 "카탈로그 본문은 판정 키에 안 넣는다"와 모순되고, (ii) 프롬프트가 실행마다 달라져 "같은 키면 재사용"이 죽는다(라이브니스). 정규화 = UTF-8/LF·canonical JSON·필드 순서·포함 메시지/도구 스키마 고정 + 위 가변부 제외. 이 digest는 검토 실행 **전에** 결정적으로 계산 가능해야 CI가 재실행/스킵을 판정한다.
     - 규칙·스킬 부분은 머지 결과에서, 모델 revision·실행기 버전·스캐폴드 프롬프트·추론 설정은 신뢰 CI 실행 환경에서 취한다. 스킬 해시만으로는 호스트가 조립·주입하는 스캐폴드와 base 실행기 코드를 결속하지 못하므로 이들을 함께 넣어 **"판정 키가 가리키는 정책·실행기 = 실제 실행된 것"**을 보장한다. 값이 바뀌면 epoch가 바뀌어 기존 검사 상태(증거 산출)를 무효화하며, 모델·실행기 업그레이드처럼 push 없는 변경의 재발화·재검증은 **CI 환경 설정 변경을 신뢰 트리거로 삼는 호스트 측 책임**이다. 완료 판정 규칙은 대상 SoT 콘텐츠를 제약하는 콘텐츠 규범이 아니라 거버넌스라 `scope`로는 완료 무효화를 걸 수 없어, 그 결속을 이 성분이 담당한다(`ADR-0014`).
  - **rule catalog manifest digest = 규칙 카탈로그의 `{path, id, scope, status}` 집합에 대한 해시.** 카탈로그 대상은 `docs/sot/rule/`의 규칙 파일이며 **`README.md`·`_TEMPLATE.md`는 규칙이 아니라 제외**한다(이들이 예시용 `id`·`scope`를 갖더라도 카탈로그·적용 집합에 넣지 않는다). 규칙 파일의 **추가·삭제·이동(`path`)·`id` 변경·`scope` 전환·`status` 변경**을 감지한다(규칙 **본문** 변경은 감지하지 않는다 — 그건 선언 문서의 적용 rule 지문이 잡는다). `status`를 성분에 넣는 것은 **활성 전이(`active`↔`deprecated`·`superseded`)가 적용 집합을 바꾸기 때문**이다 — 적용 집합(성분(2))은 `status: active`인 rule만 포함하므로, global rule이 `deprecated`가 되면 그 rule은 더는 모든 문서에 적용되지 않고(적용 집합에서 빠짐) 완료 전량이 재검토된다. `scope: local` rule을 새로 추가하면 아직 어떤 문서도 `rules`에 선언하지 않아 성분(2)만으로는 재검토 대상이 0건이 되는 공백을, 이 성분이 완료 전량 재검토로 닫는다(`ADR-0014`).

  **실행기(신뢰 기반, base) 대 검토 정책(후보, merge-result)**: 검토를 구동하는 CI 하니스·결정적 검사 코드(`sot-lint`)·판정 키 계산은 `rule-protected-paths`대로 신뢰 ref(base)에서 읽는다 — 후보 브랜치가 **실행기 자체**를 바꿔 검사를 무력화하지 못하게 하기 위함이다. 반면 **검토 대상 정책**(SoT가 따라야 할 규칙·검토 스킬 내용)은 **머지 결과(후보)**에서 취한다 — 규칙·스킬을 고치는 PR은 그 **새 정책 기준으로** 판정돼야 옛 정책 통과 세탁을 막기 때문이다. 즉 신뢰 실행기(base)가 후보 정책(merge-result)을 후보 콘텐츠(merge-result)에 적용하며, epoch가 후보 정책·실행기 버전을 결속한다.
  **부트스트랩·거버넌스 결속**: 실행기(하니스·직렬화기·응답 파서·키 계산기·프롬프트 스캐폴드) 대상은 `rule-protected-paths`의 **실행기 manifest가 단일 출처로 열거**하며(Phase 6에서 실 경로·artifact ID 확정), 같은 manifest가 **보호경로 glob과 epoch의 실행기 revision digest에 함께** 쓰여 "(실행기 코드에 한해) 보호되는 것 = 버전 결속되는 것"을 일치시킨다. ① 결정적 검사기(`sot-lint`)도 후보 브랜치가 검사를 무력화하지 못하게 manifest **보호경로**에 포함하되, ①은 키 캐시가 없어 매 실행 결정적으로 재계산되므로 **epoch revision 결속은 불요**하다(보호만 필요) — manifest는 이 둘을 **구분해 열거**한다: (a) 보호 + epoch 결속 대상 = 실행기 코드(하니스·직렬화기·응답 파서·키 계산기·프롬프트 스캐폴드), (b) 보호 전용 대상 = ① 검사기. 정본 열거·이 2분류는 `rule-protected-paths`가 소유한다. 실행기 변경은 base에 머지돼야 효력이 생기고(protected-paths), 그 revision이 epoch에 결속돼 promotion 후 기존 완료가 새 epoch로 재검증된다. 검토 스킬(`.claude/skills/sot-readiness-review/**`)·완료 판정 규칙·게이트 규칙 같은 **거버넌스 정책 파일은 `rule-protected-paths` 보호경로에 포함**돼 그 변경이 ③ 사용자 게이트 승인을 요구한다 — 이게 없으면 일반 PR로 스킬을 먼저 "항상 통과"로 약화한 뒤 다음 SoT PR에서 쓰는 2단계 우회가 가능하다. 무력화를 **막는** 것은 epoch가 아니라 **PR diff에 대한 ③ 승인자 검토**다(epoch는 기존 판정을 무효화해 새 게이트 사이클을 강제할 뿐, 무력화된 스킬로 재실행하면 그대로 통과하므로 탐지는 못 한다). 즉 무력화 탐지·차단은 ③가, 기존 판정 무효화는 epoch가, 실행기 무결성은 base 신뢰가 앵커한다.
  비규범 부분(근거·예시) 편집도 지문·epoch를 바꿔 재검토를 유발할 수 있으나, **정합성 발화 조건이 좁거나**(지문 — 그 rule을 `rules`에 선언한 문서가 있을 때만 지문에 반영; 발화하면 재검토는 늘 projection 전량) 저빈도라(epoch — 완료 판정 규칙의 실제 변경) 보수적으로 수용한다. 저장소 파생 성분이 머지 결과에 의존하므로, PR이 그대로여도 base(`main`)가 다른 rule·정책 변경으로 전진하면 판정 키가 달라질 수 있다(정확성상 의도된 동작 — 새 rule·정책 반영).
- **머지 게이트가 신뢰하는 산출물**: 머지 게이트((가) 집계 게이트 `sot-readiness-gate` / (나) 머지 컨트롤러)가 "남은 blocking이 모두 처리됐는가"를 계산하려면 **현재 두 키의 open blocking 목록**과 그에 대한 사용자 표시가 필요하다. 그 신뢰 출처는 **감사 로그 파일이 아니라 CI 검토 증거 산출**이다 — CI는 두 검토(정합성=판정 키, 선언 완전성=완전성 스윕 키) **각각**에 대해 verdict와 함께 **해당 키와 각 open blocking finding의 `(F-n + 내용 digest)` 목록**을 증거 산출로 낸다. 게이트는 이 신뢰 목록을 호스트 채널의 사용자 표시(아래 스키마)와 **같은 완전 결속 키로** 대조하고, CI는 각 산출에 실린 키로 그 검토의 재실행/스킵을 판정한다(감사 로그는 비신뢰라 읽지 않는다). 감사 로그는 사람이 읽는 사본일 뿐 게이트의 입력이 아니다. 여기서 **finding 내용 digest = 그 finding의 (축 + 참조 문서·항목 ID + 심각도 + 설명 본문)을 정규화해 취한 해시**로 결정적으로 정의한다 — CI 산출과 사용자 표시가 같은 값을 재현해야 대조가 안정적이다. 정규화 규칙(공백·항목 순서·마크다운 표준화)은 **실행기(응답 파서·키 계산기)**가 단일 구현으로 고정한다(① 형식 금지어 목록을 검사기가 단일 관리하는 것과 같은 원칙).
- 검토는 CI 실행이라 문서를 작성한 세션과 자연히 분리된다(자기검토 편향 방지). 실행 모델·프롬프트는 검사 산출과 감사 로그에 기록한다.
- **재검토는 이전 회차의 판정·finding을 입력으로 받지 않는다.** 어느 키(판정 키·완전성 스윕 키)가 바뀌어 그 검토가 발화하면, 매 실행은 **현재 콘텐츠 + 그 검토의 입력**만 보고 처음부터 판정한다 — 이전 판정을 입력으로 이월하면 바뀐 내용이 옛 판정으로 세탁(laundering)되는 우회가 생기기 때문이다(감사 로그는 위에서 밝힌 대로 비신뢰 사본이라 여기서도 입력이 아니다). 검토 입력 두 갈래 = **적용 rule 원문**(선언된 local + 활성 global — 정합성 판정용, 판정 키 성분(2)에 결속)과 **활성 rule 카탈로그 본문 전량**(선언 완전성 검사용; `README`·`_TEMPLATE` 제외 — 완전성 스윕 키에 결속). 후자는 "선언 누락한 의존"을 탐지하는 **선언 완전성 검사**가 rule 본문을 실제로 보게 하려는 것이다(적용범위 §완전성 입력).
- `blocking` finding은 두 갈래로만 닫힌다(정합성·완전성 어느 검토의 blocking이든 동일) — (a) SoT/선언 변경으로 실제 해소(그 finding이 **그 검토의 새 키** CI 산출에 더는 나타나지 않음), (b) **사용자 채널 결정**으로 `accepted`(위험 인지 후 수용) 또는 `rejected`(오판). 게이트는 어느 경우든 **두 검토의 현재 키(판정 키·완전성 스윕 키) CI 산출의 `open` blocking 목록 전부**를 대조한다(어느 한 검토의 blocking도 빠뜨리지 않는다). (a)의 `resolved` 표기는 **호스트가 같은 검토의 키 간 신뢰 CI 산출을 비교해** 감사용으로 붙이는 것이지 LLM 검토가 내는 산출이 아니다 — 매 검토 실행은 이전 finding을 입력받지 않으므로(위) "이전 키에 있던 finding이 사라졌다"는 판단은 검토가 아니라 호스트 조립 단계가 한다.
- **사용자 채널 스키마**: `accepted`·`rejected`는 감사 로그 표기가 아니라 **호스트 채널**(그 커밋 PR)에서 **완전 결속 키(그 finding을 낸 검토의 키 — 정합성이면 판정 키 네 성분, 완전성이면 완전성 스윕 키 — + finding ID `F-n` + finding 내용 digest)**를 참조해 표시한다 — 위 CI 신뢰 산출과 같은 키다. 위조 가능한 파일에 완료를 여는 권한을 두지 않고, 콘텐츠·rule 경계를 넘어 이월되지 않게 하기 위함이다. **그 표시가 참조한 완전 결속 키(그 finding을 낸 검토의 키 — 정합성이면 판정 키, 완전성이면 완전성 스윕 키)의 성분이** 바뀌면 그 표시는 무효가 되어 재확인이 필요하다(다른 검토의 키 변화가 이 표시를 교차 무효화하지는 않는다 — 표시는 자기 검토의 키에만 결속된다).
- **표시 행위자 인증**: `accepted`·`rejected`를 얹을 수 있는 주체는 **③ 사용자 게이트의 승인자와 동일한 결정권자**(저장소 permission `admin` ∧ 지정 명단 등재 ∧ 사람 계정, PR 작성자 아님 — 스펙 §2.7)여야 한다. `.github/CODEOWNERS`는 다인 구성 전환 시 켜지는 경로별 2차 관문일 뿐 이 명단을 대체하지 않는다. 완료를 여닫는 "사용자"를 여기서 새로 정의하지 않고 ③의 승인자를 재사용한다 — 완료 판정 전체가 한 승인 주체에 결속되게 하기 위함이다. 게이트는 **호스트 이벤트의 actor를 그 승인자와 대조**하고, 호스트별(GitHub/GitLab/Forgejo)로 actor를 신뢰 확인하지 못하면 표시를 **거부한다(fail-closed)**. 이 인증은 머지 게이트가 강제하며((가) 서버사이드 집계 게이트 / (나) 머지 컨트롤러가 호스트 이벤트 actor를 검증), 클라이언트가 주장하는 actor 문자열은 신뢰하지 않는다.
- 두 검토 중 하나라도 검사 상태가 없거나 현재 키(정합성=판정 키, 완전성=완전성 스윕 키)에 대한 것이 아니면 **미완료로 처리한다(fail-closed)** — 판정 키 성분(트리·적용 rule 지문·`review_policy_epoch`·규칙 카탈로그 digest)이나 완전성 스윕 키 성분(트리·활성 카탈로그 입력 digest·epoch) 어느 것이든 어긋나면 그 검사 상태는 무효다.

### ③ 사용자 승인 (최종 게이트)
- 사용자가 `rule-sot-change-user-gate`의 게이트 PR을 승인해야 완료가 확정된다.
- 승인은 승인 시점의 **두 키(판정 키·완전성 스윕 키)**에 대해서만 유효하다. requirements·specification·test-design push(트리)든 적용 rule 변경(지문)이든 완료 판정 규칙·검토 주체·실행기 변경(`review_policy_epoch`)이든 규칙 목록 구조 변경(카탈로그 manifest)이든 미선언 rule 본문·경로 변경(카탈로그 입력 digest)이든 어느 키가 바뀌면 승인은 무효가 된다(호스트의 "최신 push 재승인" 보호 = dismiss-stale; ②의 fail-closed와 같은 키 기준). push 없이 base(`main`) 전진만으로 키가 바뀌는 경우(트리·적용 rule 지문·카탈로그 manifest·카탈로그 입력 digest·epoch — 두 키의 모든 성분)는 dismiss-stale이 아니라 머지 게이트의 신선성 보장이 커버한다 — **(가)** up-to-date-before-merge + 집계 게이트 fail-closed(재싱크가 증거 검사를 재실행시키고, 집계 게이트가 현재 키와 어긋난 검사 상태를 미완료로 계산해 머지를 막는다) / **(나)** 머지 컨트롤러가 머지를 전역 직렬화하고 머지 직전 착지 두 키에서 술어를 재계산해 **착지 두 키 = CI 증거 두 키 = 승인 시 고정된 두 키**의 삼자 일치를 확인한다(승인 시 두 키는 재계산하지 않고 고정 — 재계산하면 낡은 승인이 새 상태의 승인으로 되살아난다; 스펙 §2.3). 어긋나면 미완료로 처리한다.

## 강제 매핑
각 조건은 서로 다른 메커니즘이 강제한다. 허브 게이트(`rule-protected-paths`·`ADR-0007`, Phase 3)는 "SoT는 사용자 게이트 PR로만"(경로)까지만 강제하며 완료 조건 자체는 평가하지 못한다. ①②③의 결속은 Phase 6 강제 증분에 의존한다.

**머지 게이트의 두 실현.** 완료 술어(**① 통과 ∧ ② 성립 ∧ ③ 승인**, 현재 두 키에 결속)를 머지 시점에 강제하는 것을 **머지 게이트**라 하며, 둘 중 하나로 실현한다. 완료 술어·두 키·CI 증거 산출은 두 실현이 공유하고, **술어를 어디서 계산해 어떻게 머지를 막는가**만 다르다.
- **(가) 집계 게이트** — 서버사이드 필수 검사 `sot-readiness-gate`가 완료 술어(① 통과·두 증거·사용자 override·③ 승인·현재 두 키 일치)를 계산해 성공/실패를 내고, 호스트 브랜치 보호가 그 검사를 요구한다. ①형식·정합성·완전성 검토는 **비필수 증거 검사**로 두고 이 집계 게이트만 유일 필수 검사로 삼는다 — 호스트 브랜치 보호에서 "실패 상태의 필수 검사"는 다른 검사가 성공해도 머지를 막으므로, 검토가 `blocked`여도 사용자 override로 완료가 열리려면 검토 산출이 필수 검사가 아니라 집계 게이트의 **입력**이어야 하기 때문이다(증거/집계 분리는 (가) 특유의 요구다).
- **(나) 머지 컨트롤러** — `main` 갱신 권한을 지정 신원 하나로 제한하고, 그 신원이 머지 직전에 완료 술어를 **착지하는 두 키에 대해 신선하게 계산**해 통과할 때만 머지한다. 호스트 필수 검사를 쓰지 않으므로 이를 별도 필수 검사로 둘 필요는 없으나 증거 산출과 완료 판정의 논리적 분리는 유지하며(컨트롤러가 같은 CI 증거 산출을 직접 읽어 술어를 계산), 필수 검사 표현력·위조 내성이 호스트마다 다른 문제를 피해 "누가 `main`을 갱신할 수 있는가"라는 공통 원시 기능만 쓴다. 포크 거부·강제-필수 경로 관문도 이 컨트롤러가 강제한다.

**Phase 6은 (나) 머지 컨트롤러를 채택한다**(`ADR-0009`·스펙 `2026-07-08-phase6-enforcement-host-branch-protection-design`). 아래 표는 (나) 기준으로 읽되 실현이 갈리는 행에는 (가) 대응을 병기한다(나머지 행은 두 실현의 메커니즘이 같다) — 어느 실현이든 같은 완료 술어를 강제하며, **판정은 실제로 착지하는 두 키에 대해 신선해야 한다**(이전에 계산된 통과를 그대로 재사용해 머지하는 경로는 없다).

**`touches_sot`가 참인 PR**(target-content projection을 바꾸는 PR)의 `main` 머지는 **① 통과 ∧ ② 성립 ∧ ③ 승인**이 **현재 동일한 두 키(판정 키 네 성분: 트리·적용 rule 지문·`review_policy_epoch`·rule catalog manifest; 완전성 스윕 키: 트리·활성 카탈로그 입력 digest·epoch)**에 함께 결속돼 참일 때만 열린다(강제-필수 경로 PR·그 밖 PR의 분기는 아래 "강제-필수 경로"가 정한다). 여기서 **② 성립 = 정합성 검사 `review_clear` ∧ 완전성 검사 `completeness_clear`**(각 CI 산출에 open blocking 없음), **또는** 그 두 검사가 낸 open blocking finding 각각이 호스트 채널에서 `accepted`·`rejected` 표시를 받음이다(트리 변경으로 실제 해소된 것은 CI 재실행이 해당 키의 `clear`로 반영하므로 `resolved`는 이 식에 따로 두지 않는다). 머지 게이트가 대조하는 blocking 목록은 감사 로그가 아니라 CI 검사 산출이다. `accepted`·`rejected`는 ② 검사를 재실행·상태변경하지 않으며, 머지 게이트가 검사 상태 위에 사용자 결정을 얹어 계산한다. 로컬 훅은 권고 수준이다.

**강제-필수 경로.** 완료 강제(①②③)는 **SoT를 바꾸는 PR**(제안된 머지 결과가 target-content projection을 바꾸는 PR)에만 적용한다. 다만 **강제 장치 자체를 바꾸는 PR**(완료 술어를 계산하는 규칙 지문·검토 CI·실행기 코드·승인자 정의·게이트 코드 등 "강제-필수 경로")은 ①②③ 대상은 아니되 **머지 게이트가 그대로 통과시키지 않는다** — 최소한 대상 저장소 출처(포크 거부)와 **결정권자 승인**을 요구한다(위 거버넌스 정책 무결성 행과 같은 정책을 머지 시점에 강제하되, **이 분기는 ③ 완료 승인과 달리 ①②·두 키 결속은 요구하지 않는다** — 오직 `pr_state == OPEN` + 포크 거부 + 결정권자 승인, 스펙 §2.6 fail-closed 항목 1·3). 여기서 **결정권자 = 저장소 permission `admin` ∧ 지정 명단 등재 ∧ 사람 계정 ∧ PR 작성자 아님 ∧ 승인 철회 안 됨**이며(스펙 §2.6·§2.7), `.github/CODEOWNERS`는 다인 구성 전환 시 켜지는 2차 관문일 뿐 이 명단을 대체하지 못한다. **분기 우선순위**: `touches_sot`가 참이면(대상 콘텐츠를 바꾸면) 강제-필수 경로 동시 변경 여부와 무관하게 ①②③을 전부 적용하고, `touches_sot`이 거짓이고 강제-필수 경로만 참일 때 이 관문(`pr_state == OPEN` + 포크 거부 + 결정권자 승인)을 적용한다(스펙 §2.6·`evaluate_gate` — 결합 PR에서 ①②가 생략되지 않게 한다). 그 경로 집합의 권위 정의는 `rule-protected-paths`의 **강제-필수 경로 블록(`axdt-critical-paths`)**이 **유일 입력**으로 소유하며(게이트는 그 블록의 `critical <glob>` 줄을 읽고 목록을 여기 복제하지 않는다 — 스펙 §2.6; 블록이 `main`에 없거나 기형이면 판독·파싱을 `evaluate_gate`에 앞선 **사전 관문**에서 fail-closed RED로 처리한다 — 스펙 §7 (ㅁ)), 이 관문은 Phase 6 강제 층이 실현한다. 두 경로 어디에도 해당하지 않는 PR은 `pr_state == OPEN`만 확인하고 통과시킨다(pass-through; 스펙 §2.6 그 밖 분기). SoT·강제-필수 경로 여부는 **제안된 머지 결과의 변경분**으로 판단하고 브랜치 이름·커밋 메시지로 판단하지 않는다 — (나) 머지 컨트롤러는 저장소의 모든 머지를 수행하므로, 이 구분이 없으면 무관한 PR이 브랜치 이름 규약에 걸려 영구히 막히거나 강제 장치가 무관문으로 교체된다.

**신선성.** 머지 게이트 판정은 **실제로 착지하는 두 키**에 대해 계산돼야 한다. 판정과 머지 사이에 base가 전진하거나 결정·승인·권한이 바뀌면 이전 판정은 무효다. **(나)** 머지 컨트롤러는 머지를 전역으로 직렬화하고 머지 직전에 판정을 다시 계산해 이를 보장한다 — base 부동은 전역 직렬화(`main` 갱신 신원이 컨트롤러 하나뿐이라 잠금 안에서 base를 움직일 주체가 없음)가, head 이동은 **평가에 쓴 head SHA를 머지 호출에 고정**(불일치 시 거부·재평가)하는 계약이 막는다(스펙 §2.5). 그래서 결정 취소·권한 변경·base 전진 같은 이벤트를 빠짐없이 좇아 이전 판정을 무효화하는 방식에 기대지 않는다. **(가)** 집계 게이트는 낡은 base·키 불일치·낡은 승인을 up-to-date-before-merge + 키 대조(fail-closed)로 막지만, **키에 안 잡히는 변화**(결정 번복·승인자 명단 강등)는 그 이벤트가 게이트 검사를 재발화시켜야 잡히고 마지막 검사~머지 창이 남는다 — 이 잔여 창은 (나)의 머지 직전 재계산만 구조적으로 닫는다.

## 근거
- 완료는 곧 자동 개발 시작의 방아쇠(D6)다. 방아쇠 조건이 느슨하면 미완성 SoT 위에서 개발이 출발한다.
- 기계 검증(①)만으로는 "형식은 갖췄으나 내용이 비었거나 모순된" 문서를 거르지 못한다 — 의미 판단(②)이 필요하다. 다만 LLM 검토는 비결정적이므로, 키당 1회 실행·호스트 검사 결속·소멸 규칙·분리 실행·fail-closed로 재현성과 우회를 막는다.
- ③을 단독 충분조건으로 두면 ①②가 실패해도 승인만으로 완료가 성립하는 모순이 생긴다. 완료는 세 조건의 논리곱이다.
- 판정을 콘텐츠에 결속하지 않으면 "승인한 문서"와 "개발이 출발하는 문서"가 달라질 수 있다(승인 후 내용 바꿔치기). 머지 커밋은 승인 커밋과 SHA가 다를 수 있으므로 SHA가 아니라 콘텐츠(트리) 동일성으로 본다.
- 감사·콘텐츠 결속을 위해 `main` 룰셋이 SoT PR의 squash 머지를 막는다(`allowed_merge_methods=["merge"]`) — squash는 사용자가 승인한 커밋 자체를 없애 새 단일 커밋으로 바꿔 감사·콘텐츠 추적을 깨뜨린다. merge commit은 승인 head를 부모로 이력에 보존한다. `sot/*` 소스 브랜치 자체 보호는 요구하지 않는다(`ADR-0009` 결정 8·스펙 §2.8).
- 판정 키가 하나라도 바뀌면 projection **전량**을 네 축 전량 재검토한다 — rule 변경이 주로 교차 정합성(축3)에 걸려도 축1·2·4까지 전 문서에 다시 도는 비용을 **의식적으로 감수**한다. ②검토가 문서 간 홀리스틱(커버리지 매트릭스·교차 정합성)이라 부분 재검토가 불건전하고, 이월·메모이즈를 허용하면 세탁·유실 우회가 생기기 때문이다. 문서별 증분 최적화(바뀐 부분만 좁혀 재검토)는 정확성을 확보한 뒤의 Phase 6 최적화 과제로 미룬다.

## 적용범위
- **대상**: `docs/sot/requirements/`·`docs/sot/specification/`·`docs/sot/test-design/`의 완료 판정과, 그 완료를 전제로 하는 자동 개발 시작 트리거(D6).
- **개발 시작 트리거**: **target-content projection(§②)이 변경된 PR**에만 적용한다 — req·spec·test-design의 실제 콘텐츠가 바뀔 때다. `README`·`_TEMPLATE`만 바꾸는 PR이나 `rule/`·`ADR`만 바꾸는 PR은 개발을 시작시키지 않는다. 한 PR이 target-content와 rule/ADR을 함께 바꾸면 target-content 변경을 기준으로 판정한다.
- **테스트 설계의 완료 범위**: test-design의 완료 판정은 테스트 조건·블랙박스 커버리지 목표·스위트 구조·추적성·절차 원칙까지만 대상으로 한다. 구체 케이스·데이터·픽스처·실행 스크립트·환경과 화이트박스 커버리지(결정/분기/MC-DC)는 `test/` 코드의 몫이라 요구하지 않는다 — 요구하면 코드 없이 충족 불가라 완료가 영구 미충족이 된다(설계/구현 절단선, `test-design/README`·`ADR-0008`).
- **재검토 트리거 (rule 변경)**: 완료는 req·spec·test-design이 의존하는 rule에도 결속되므로, rule이 바뀌면 기존 완료의 ② 검토가 무효가 될 수 있다. 무효화·재검토는 §②의 **두 키**로 판정한다 — 정합성은 판정 키(성분(2) 적용 rule 지문·성분(4) catalog manifest), 선언 완전성은 완전성 스윕 키(활성 카탈로그 입력 digest). 어느 키가 바뀌든 그 키가 결속한 검토를 **projection 전량 홀리스틱**으로 재실행한다(대상을 문서로 쪼개지 않는다 — §②대로 부분 재검토가 불건전). `scope`는 "어느 문서를 다시 볼지"가 아니라 **"어느 rule의 변경이 성분(2)에 들어가 정합성 무효화를 유발하는가"**를 정한다.
  - **scope 게이팅**: **`scope: global`** rule의 본문 변경은 항상 성분(2)에 들어가고(모든 완료 문서가 그 rule에 종속), **`scope: local`** rule의 본문 변경은 **어느 완료 문서든 `rules`에 선언했을 때만** 성분(2)에 들어간다. 어느 쪽이든 성분(2)가 바뀌면 정합성 재검토가 projection 전량 돈다 — local이라 대상이 좁아지는 게 아니라, 아무도 선언하지 않은 local rule의 변경은 성분(2)를 아예 안 건드려 **정합성** 재검토가 안 도는 것이다(대신 그 본문은 완전성 스윕 키를 바꿔 선언 완전성 검사를 재실행한다 — 아래). `global`은 **도메인 공용 용어·공용 식별자 명명처럼 대상 SoT 콘텐츠를 실제로 횡단 제약하는 도메인 콘텐츠 규칙**을 위한 것이다 — 현행 rule 전부(11종)는 오케스트레이션·거버넌스라 대상 콘텐츠를 제약하지 않으므로 모두 `local`이다(`ADR-0015`). **분류 기준**: 콘텐츠 규범(대상 SoT가 '무엇을' 말하는지 제약)은 횡단성에 따라 global/local, 거버넌스·완료정책(누가·어떻게 절차를 밟는지)은 local. `scope` 미기재 rule은 보수적으로 global로 취급하므로, 모든 rule은 `scope`를 명시한다.
  - **catalog 구조 변경 (추가·삭제·이동(`path`)·`id`·`scope`·`status`)**: rule 카탈로그의 `{path, id, scope, status}` 집합이 바뀌면 성분(4) catalog manifest digest가 달라져 판정 키가 바뀌고, 정합성 재검토가 projection **전량**을 돈다. 이는 신규·전환된 rule을 기존 완료 문서가 아직 `rules`에 선언했을 수 없어 성분(2)만으로는 안 걸리는 공백을, 문서가 새 구조를 반영·재선언하게 해 닫는다. 카탈로그 본문이 바뀌면 **완전성 스윕 키**도 함께 바뀌어 선언 완전성 검사가 미선언 의존을 실제로 탐지하며(아래 완전성 입력), 선언이 채워지면 이후 그 rule의 본문 변경은 성분(2)로도 정합성에 결속된다.
  - **완전성 입력 (미선언 의존 탐지)**: **선언 완전성 검사**가 "문서 D가 실제로는 **`scope: local` rule R**의 규율 대상인데 `rules` 선언을 누락했다"를 탐지하려면 R의 본문을 봐야 한다(대상은 local rule에 한정 — global은 선언 없이 항상 적용되므로 "선언 누락"이 성립하지 않는다). 그래서 검토 입력은 **적용 rule 원문**(선언된 local + 활성 global — 정합성 판정용, 판정 키 성분(2))과 **활성 rule 카탈로그 본문 전량**(선언 완전성 탐지용; README·`_TEMPLATE` 제외 — **완전성 스윕 키에 결속**)을 구분해 싣는다. 이로써 완전성 검사가 "본문을 못 봐 원리적으로 비교 불가"인 상태를 벗어난다. 카탈로그 본문 전량은 **판정 키(정합성)에는 넣지 않는다** — 넣으면 미선언 local rule(대개 오케스트레이션·거버넌스) 본문 편집마다 정합성 완료 전량이 헛발화해 `ADR-0015` 결정과 어긋나기 때문이다. 대신 **완전성 스윕 키에 결속**하므로, 미선언 rule 본문 편집은 정합성은 안 건드리면서 선언 완전성 검사만 재실행시킨다 — "같은 키가 다른 실제 입력을 대표하는" 캐시 우회가 없다(완전성 판정은 자기 입력 전량이 자기 키에 결속). 정합성 헛발화 방지와 완전성 소여 결속을 **두 키로 분리해** 둘 다 얻는다. 미선언 rule R을 문서 D가 **현재 이미 위반**하는 경우, 정합성 축은 R을 입력(a)(선언 local + 활성 global)에서 보지 못해 그 위반을 잡을 수 없다 — 그래서 R 본문을 보는 **완전성 검사가 선언 누락 blocking과 함께 그 현재 위반도 같은 완전성 스윕 키에 별도 blocking으로 산출**한다(선언이 채워져 R이 성분(2)에 들어온 뒤부터는 정합성 축이 이어받는다).
  - **초기 도입 (마이그레이션)**: 이 재검토 결속과 완전성 검사(미선언 의존 발견)가 활성화되는 시점(①②③ 강제 도입 = Phase 6, 또는 프로젝트가 이 규칙을 뒤늦게 채택)에, 그 이전 완료분 전량을 **최초 1회 스윕**해 `rules` 선언 완전성을 확립한다. 이후엔 완료 시점 검사와 두 키의 변화가 불변식을 유지한다.
  - **알려진 잔여 위험 (선언 완전성)**: 미선언 rule 본문 편집이 완전성 스윕 키를 바꿔 검사를 재실행시키므로 "본문 편집이 안 잡히는" 우회는 없다. 잔여는 오직 **완전성 검사가 비결정(LLM) 판정이라 본문을 보고도 의존을 놓칠 수 있음**뿐이다 — 비결정 검토기를 필요조건으로 둔 대가로 수용하며, ①의 사문(死文) 선언 차단·저술 시 `rules` 스캔·완료 시점 검사가 그 빈도를 줄인다.
  - **운영 노트**: 현행 rule 전부(11종)가 `scope: local`이다(`ADR-0015`, D31). rule 변경 자체는 개발 시작 트리거가 아니다. 재검토 `blocking`이 이미 착수된 작업에 미치는 효과(중단·경고·다음 게이트 반영)는 트리거 상태 머신(Phase 8)이 다룬다.
- **예외**: 개발 착수 이후의 SoT 재변경에 대한 트리거 재발화·멱등성(중복 실행 방지)은 본 규칙이 아니라 트리거 상태 머신(Phase 8)이 다룬다.

## 예시
**준수 (✓)**
- requirements의 `FR-1`을 specification의 `SP-3`이 구체화하고 test-design의 `TD-2`가 검증하며 형식 검사 통과 → 호스트 CI가 `sot-readiness-review`의 정합성·선언 완전성 검사를 각각 1회 실행, `review_clear`·`completeness_clear`가 각각 판정 키·완전성 스윕 키의 검사 상태로 고정 → 사용자가 같은 두 키의 PR을 승인·머지 → 개발 시작.
- 검토에서 `blocking`(사양 누락)이 나옴 → 작성자가 SoT를 고쳐 커밋(트리 변경) → 바뀐 콘텐츠에 대해 CI가 재실행해 새 판정 키의 산출에 그 finding이 더는 안 나타남(`review_clear`) → 호스트가 이전 판정 키 CI 산출과 비교해 그 finding을 감사상 `resolved`로 조립 → 승인.
- ② 검사가 `review_blocked`(사소한 공백 지적 하나)인데 사용자가 그 finding을 호스트 채널에서 `accepted`(감수) 또는 `rejected`(오판)로 표시 → 게이트가 남은 `open` blocking이 모두 처리됐음을 확인, ③ 승인과 함께 머지 허용.
- 브랜치 명명 rule(`branch-workspace-naming`, `scope: local`)을 **어느 완료 문서든 `rules`에 선언한 상태에서** 바꿈 → 성분(2) 적용 rule 지문 변경 → **projection 전량 정합성 재검토**(네 축 전량). 아무도 선언 안 한 상태면 성분(2) 불변 → **정합성** 재검토 없음(단 그 rule 본문이 바뀌면 완전성 스윕 키가 바뀌어 선언 완전성 검사는 재실행). 도메인 공용 용어집 rule(`scope: global`, 향후 도입 시)을 바꾸면 항상 성분(2)에 들어가 전량 재검토 → 충돌 없으면 완료 유지, 있으면 `blocking`으로 재작성 요구.
- 완료 판정 규칙(`rule-sot-readiness`)이나 검토 스킬·모델·실행기를 바꿈 → `review_policy_epoch` 변경(두 키 공통 성분) → 완료된 req·spec·test-design 전량 정합성·완전성 재검토(옛 판정 무효). 새 `scope: local` rule을 추가 → catalog manifest digest(정합성 전량 재검토) + 활성 카탈로그 입력 digest(완전성 스윕 재실행) 변경 → 미선언 의존 탐지.
- **어느 문서도 선언하지 않은** local rule의 본문을 "이제 기존 문서 D도 규율하도록" 확장 → `{path,id,scope,status}` 불변이라 판정 키(정합성)는 그대로지만 **활성 카탈로그 입력 digest가 바뀌어 완전성 스윕 키가 변경** → 선언 완전성 검사가 재실행돼 "D가 이 rule을 선언해야 함"을 `blocking`으로 올림(같은 키로 옛 `completeness_clear`를 재사용하는 캐시 우회가 없다).

**위반 (✗)**
- 형식은 통과했으나 ② 검사가 `review_blocked`(미처리 blocking 존재)인데 사용자가 머지 → ②를 건너뛴 완료. 게이트가 막아야 한다.
- 사용자가 승인한 뒤 specification에 커밋을 더 얹고(트리 변경) 그대로 개발 시작 → 승인 트리 ≠ 개발 트리, 승인·수용 모두 무효(재확인 필요).
- 감사 로그 파일에 `accepted`를 직접 써넣어 blocking을 닫으려 함 → accepted는 호스트 채널에 결속되므로 게이트는 파일 표기를 신뢰하지 않는다.
- ③ 게이트 승인자가 아닌 주체가 호스트 채널에서 blocking을 `accepted`로 표시 → 게이트가 이벤트 actor를 승인자와 대조해 거부(fail-closed). 완료를 여닫는 권한은 ③ 승인자에만 있다.
