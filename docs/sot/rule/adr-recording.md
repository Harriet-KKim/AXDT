# 결정 근거 기록 — PR 요약과 ADR 승격

## 규칙문
> 모든 PR은 핵심 결정과 기각한 대안을 본문에 요약한다. 아래 **촉발 조건**에 해당하는 지속적 설계 결정은 ADR을 작성하거나, 작성하지 않은 사유를 PR 본문에 남긴다.

PR 요약은 항상 필수인 **바닥선**이고, ADR은 촉발 조건이 걸릴 때의 **승격 층**이다. ADR *파일의 존재 자체*는 기계로 강제하지 않는다 — 결정의 ADR 가치는 판단이라, 존재를 강제하면 통과용 빈 ADR·"해당 없음"을 부른다. 강제는 "촉발을 PR에서 인정하게" 하는 데까지만 둔다.

### 촉발 조건 (판단 앵커 — 완결 목록 아님)
결정이 아래 중 하나라도 해당하면 ADR 대상이다.
- **대안 기각**: 실질적 대안이 둘 이상 있었고 하나를 이유를 들어 버렸다 — 나중에 "왜 그걸 버렸나"를 다시 물을 가능성이 높다.
- **높은 되돌림 비용·넓은 파급**: 되돌리기·마이그레이션·운영·보안 비용이 크거나, 여러 req·사양·테스트 설계 항목·이후 Phase·후속 task에 걸친다.
- **구조 선택**: 공개 인터페이스·데이터 모델·저장소·인증/권한·배포·런타임·동시성/일관성 모델을 정하거나 바꾼다.
- **워크플로 규범 변경**: 역할 권한·보호 경로·게이트·브랜치/워크스페이스 규칙·SoT 문서 구조 같은 규범을 바꾼다.
- **기존 결정과의 충돌·이탈**: 기존 rule·ADR·확정 결정과 충돌하거나 이를 supersede한다. 관례·템플릿·기본값에서 벗어난다.
- **장기 외부 의존**: 외부 서비스·라이브러리·라이선스·호스트 기능에 오래 의존한다.

### 비촉발 (명시적 제외 — D13 계승)
- 1차 선택·취향, 국소적이고 가역적인 선택.
- 기존 rule·ADR이 이미 근거를 담은 결정.
- 근거가 PR 요약 한두 줄로 충분하고 PR 수명을 넘겨 참조될 일이 없는 결정(두 조건을 다 만족할 때. 하나만 참이면 이 항목에 걸리지 않고 촉발 6종으로 판정한다).

### 강제 수준 (2층)
- **강제 (구조·확인 가능)**: PR 본문에 결정 요약이 있다. 촉발 조건에 걸린 결정은 **ADR 링크 또는 "ADR 생략 사유"**를 PR 본문에 담는다.
- **권고 (판단)**: ADR 파일 생성 자체. 자동 CI blocking 대상이 아니다. ADR 파일은 결정이 내려진 시점(작업 중 언제든)에 쓰며, PR은 그것을 인정·연결하는 강제 지점일 뿐이다 — ADR 작성을 PR로 미루라는 뜻이 아니다.
- **판단 채널**: SoT PR은 `sot-readiness-review`(②)가 "촉발 조건에 걸렸는데 ADR·생략 사유가 없다/약하다"를 **non-blocking finding**으로 지적한다. 구현 PR은 Reviewer·Maintainer 수동 게이트가 같은 판정을 한다. 어느 쪽도 ADR 개수·파일 존재로 머지를 막지 않는다.

## 근거
- "비자명하면 ADR"이라는 소프트 관행은 Author·Leader·Maintainer마다 해석이 갈려, PR 요약은 남지만 ADR은 빠지는 실패를 낳는다. 결정을 방금 내린 에이전트에겐 그 결정이 자명해 보이는 자기평가 편향이 있어 "비자명"은 특히 LLM에게 약한 기준이다. 촉발 조건은 판단을 패턴 대조로 바꿔 분산을 줄인다.
- ADR 가치는 판단이라 기계 강제에 맞지 않는다 — `rule-sot-readiness`가 의미 판정을 ②/③에 두고 `rule-protected-paths`가 로컬 훅을 권고 수준으로 두는 것과 같은 성질이다. 존재를 blocking으로 강제하면 통과용 빈 ADR로 신호가 소음에 묻혀, 규칙이 없느니만 못해진다. 그래서 판단은 이미 있는 채널(② finding·수동 게이트)에 태우고 새 결정적 게이트를 만들지 않는다.
- 규칙은 "무엇을", ADR은 "왜 이렇게"를 담는다(`rule/_TEMPLATE`). 이 규칙은 그 경계에서 "ADR을 언제 남기는가"를 규정한다.
- **핵심 잔여**: 촉발 목록이 못 잡는 ADR감 결정은 남는다. 백스톱은 ③ 사용자 게이트에서 "왜?"라고 묻는 것이며, 규칙은 그 빈도를 줄일 뿐 없애지 못한다(수용).

## 적용범위
- **대상**: 모든 역할의 PR(SoT authoring·구현 공통). 촉발 조건은 *결정의 성격*에 대한 것이라 누가 내렸는지와 무관하다.
  - **강제 기구의 현 범위**: 판단 채널(②)은 현재 SoT PR에만 존재한다. 그래서 규범은 전 역할에 적용하되, 구현 PR의 강제는 지금은 **수동 Reviewer·Maintainer 게이트**에 의존한다. 구현 검토 흐름(Phase 후반)이 설계되면 그 채널로 확장한다. 규범을 좁게 쓰지 않는 이유는 "큰 결정은 기록한다"가 역할과 무관하게 보편적이고, 좁게 뒀다가 나중에 rule을 다시 고치는 비용을 피하기 위해서다.
- **ADR 위치·형식**: AXDT 자체 결정은 `WIP/adr/`(형식 `WIP/adr/_TEMPLATE.md`), 대상 프로젝트 결정은 `docs/interim/ADR/`(형식 `docs/interim/ADR/_TEMPLATE.md`).
- **예외**: rule 변경 자체는 개발 시작 트리거가 아니다(`rule-sot-readiness`). 이 rule은 `scope: local`이고 이를 `rules`에 선언하는 req·spec·test-design이 없어, 이후 *일반 편집*은 ② 재검토 대상이 없다. 단 **신규 rule 도입**은 `rule-sot-readiness`에 따라 완료된 req·spec·test-design 전량의 축3(교차 정합성) 재검토를 1회 발화한다(신규 rule은 선언된 `scope`와 무관).

## 예시
**준수 (✓)**
- Author가 테스트 설계를 4번째 SoT 타입으로 신설(워크플로 규범 변경·기존 구조 이탈 → 촉발) → `WIP/adr/0008` 작성 + PR 요약에 링크.
- Developer가 사소한 헬퍼 함수명을 고름(1차 선택·가역 → 비촉발) → PR 요약 한 줄, ADR 없음.
- Leader가 큐 대신 폴링을 택함(대안 기각·운영 비용 → 촉발) → ADR을 쓰거나, "생략 사유: 임시 스파이크라 후속 task에서 재결정"을 PR 본문에 명시.

**위반 (✗)**
- 인증 방식을 바꾸면서(구조 선택 → 촉발) PR 본문에 ADR도 생략 사유도 없음 → 판단 채널(②·Reviewer)이 non-blocking으로 지적.
- 통과 목적의 빈 ADR("해당 없음")을 찍어 촉발을 형식만 충족 → 규칙 취지 위반, Reviewer가 사유 불충분으로 되돌림.
